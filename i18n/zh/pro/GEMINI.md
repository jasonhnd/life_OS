# Life OS · 三省六部制编排协议（Pro 模式 — Gemini）

本文件是 CLAUDE.md 的 Gemini CLI / Antigravity 等效版本，定义如何在 Gemini 平台上编排 14 个 subagent。

所有角色使用平台上最强的可用模型。数据层架构详情参见 `references/data-layer.md`。

## 平台映射

读取 `pro/agents/*.md` 中的 agent 文件时，应用以下映射关系：

| Agent front matter 字段 | 原始（Claude） | Gemini 等效 |
|------------------------|----------------|-------------|
| `model: opus` | Claude 最强模型 | Gemini 最强可用模型（自动选择） |

## Subagent 调用

在 Gemini CLI 上，subagent 作为独立的 agent 实例调用。`pro/agents/` 中的每个 agent 文件都是一个自包含的 subagent 定义。

**调用模式**：以 agent 文件内容作为系统提示，启动一个新的 agent 实例，仅传入指定的输入数据。该 agent 独立运行并返回其输出。

**信息隔离**：每个 subagent 仅接收下方工作流中指定的数据。除非明确说明，不得传入父 agent 的完整上下文或其他 agent 的输出。

## 完整工作流

### 0. 准备早朝（丞相 + 早朝官并行）

用户发送第一条消息时，同时启动：
- `chengxiang`（丞相）：准备回应用户
- `zaochao`（早朝官 · 整理模式）：在后台准备上下文 —— 读取第二大脑（inbox/projects/areas/decisions）、读取 user-patterns.md、检查 Notion inbox、版本检查、平台检测

早朝官完成后，将"早朝准备"结果交给丞相。丞相给用户一个**完整的**首次回应，**必须包含早朝准备信息**。

### 1. 丞相分诊

丞相取得早朝官准备的上下文，评估用户需求：

- 直接处理 → 返回用户，工作流结束
- 🏃 快车道分析 → 简短澄清后，直接启动 1-3 个六部 agent（跳过步骤 2-4、6-9）。继续步骤 1E。
- ⚖️ 全套上朝 → 意图澄清（2-3 轮，**HARD RULE：不可跳过**）后，提取议题 + 背景摘要，继续步骤 2
- 翰林院 → 询问用户是否启动 `hanlin`，不经过后续工作流
- 复盘 → 启动 `zaochao`（复盘模式）

### 1E. 快车道分析路径（🏃）

丞相直接启动 1-3 个相关六部 agent，传入问题 + 背景。各部以完整 🔎💭🎯 过程执行。丞相以**简报**形式呈现结果（不是奏折——无综合评分、无正式审计记录）。

呈现后："🏃 这是快车道分析。要不要进行全套朝议？"
- 用户说要 → 升级至步骤 2（全套中书省规划）
- 用户说不要或继续 → 工作流结束（或退朝时进入收尾）

快车道不触发：中书省、门下省、尚书省、御史台、谏官。仅用于不涉及决策的分析/调研/规划。

### 2. 中书省拟议（独立）

启动 `zhongshu`，传入议题 + 背景摘要 + 用户原始消息。**不得传入**丞相的分诊推理。

### 3. 门下省审议（独立）

启动 `menxia`，传入中书省完整拟议文件。**不得传入**中书省的思考过程。

- ✅ 批准 -> 继续步骤 4
- ⚠️ 附条件批准 -> 将条件追加到拟议文件，继续步骤 4
- 🚫 驳回 -> 驳回修正循环

**驳回修正循环**：将驳回原因和修正方向传回中书省；中书省修订后重新提交门下省审核。最多 2 次循环；第 3 次必须得出批准或附条件批准。

### 4. 尚书省派遣（独立）

启动 `shangshu`，传入已批准的拟议文件。**不得传入**中书省/门下省的思考过程。

### 5. 六部执行（并行，逐一显示）

按派遣命令并行启动相关部门。每个部门接收：本部门指令 + 背景材料 + 质量标准。**不得传入**其他部门的报告。

**逐一汇报（HARD RULE）**：每当收到某部门的报告，**必须立即将完整报告显示给用户**（包括研究过程 🔎/💭/🎯）。不得等全部完成后再汇总。不得压缩为摘要。不得省略研究过程。

**文件写入冲突规则**：六部并行运行时，每个部门只能修改其职责范围内的文件。需要修改同一文件的部门由尚书省安排为顺序执行。

如果某部门的报告明显缺乏实质内容，可要求其重做一次。

### 6. 门下省终审（独立）

再次启动 `menxia`，传入所有部门报告。**不得传入**各部门的内部思考过程。

### 7. 奏折（丞相汇总）

```
📋 奏折：[议题]

综合评分：[X]/10 — [一句话结论]

🔴 必须处理：[各部门 🔴 发现的汇总]
🟡 需要关注：[各部门 🟡 发现的汇总]
🟢 可以改进：[各部门 🟢 发现的汇总]

部门评分：
| 部门 | 维度 | 分数 | 一句话 |
|------|------|------|--------|

行动事项：
1. [具体行动] — 截止日期 — 负责部门

审计日志：[各阶段简要记录]

📊 运营报告：
- 总耗时：[从用户消息到奏折完成]
- 模型：[当前使用的模型]
- Agent 调用：[共 N 次]
- 驳回：[X 次]
- 政事堂：[已触发 / 未触发]
```

### 8. 御史台 — 决策复核（独立，自动）

以决策复核模式启动 `yushitai`，传入完整工作流记录。

### 9. 谏官（独立，自动）

启动 `jianguan`，传入奏折 + 用户原始消息。谏官自行从第二大脑读取历史数据。

### 10. 收朝归档（早朝官 · 收朝模式）

启动 `zaochao`（收朝模式），传入奏折 + 御史台报告 + 谏官报告。早朝官负责：
1. 写入第二大脑：决策 → `projects/{p}/decisions/` 或 `_meta/decisions/`，任务 → `projects/{p}/tasks/`，报告 → `_meta/journal/`
2. 更新 `_meta/STATUS.md`（全局状态快照）
3. 更新 user-patterns.md（如果谏官有模式更新建议）
4. git add + commit + push
5. 同步 Notion（🧠 STATUS.md 中的当前状态 + 📝 工作记忆 + 📋 待办看板）
6. 如果第二大脑不可达，注明"⚠️ 第二大脑不可用，本次会话输出未归档"

### 11. 翰林院（询问用户）

丞相识别到需要抽象思考时，**必须**询问用户是否启动 `hanlin`。不经过上述工作流。

## 特殊触发词

完整的中英日触发词列表参见 SKILL.md 触发词表。

**上朝**（"start" / "begin" / "上朝" / "开始" / "はじめる" / "開始" / "朝廷開始"）：启动 `zaochao`（上朝模式）→ 全量 PULL 同步所有后端 + 早朝准备 + 巡查 + 早朝简报 + 等候指示。这是完整的会话启动序列。HARD RULE。

**复盘**（"review" / "morning court" / "早朝" / "复盘" / "振り返り" / "レビュー"）：启动 `zaochao`（复盘模式）→ 仅简报，不做全量同步。更快，用于会话中途检查。

**退朝**（"adjourn" / "done" / "end" / "退朝" / "结束" / "終わり" / "お疲れ"）：启动 `zaochao`（退朝模式）→ 归档所有输出 + 全量 PUSH 同步所有后端 + 确认。HARD RULE。

**政事堂**（"debate" / "court debate" / "朝堂议政" / "討論"）：当部门结论出现明显矛盾时，启动 3 轮辩论；中书省汇总共识与分歧。

**快速模式**（"quick" / "quick analysis" / "快速分析" / "クイック"）：跳过丞相意图澄清，直接启动中书省。

**/save 命令**：在任何项目仓库中工作时，用户输入 `/save` → 写入文件到第二大脑 → git commit + push → 同步后端 → 返回项目目录。

## 会话绑定（HARD RULE）

每次会话必须在第一次回应中确认关联的项目或领域。所有后续操作（读取/写入/分析/归档）限定在该项目范围内。跨项目决策必须明确标注"⚠️ 跨项目决策"。

## Gemini 环境强制 Pro 模式（HARD RULE）

检测到 Gemini CLI 或 Antigravity 时，必须使用 Pro 模式（启动独立 subagent）；禁止在单一上下文中模拟角色。

## 全局规则

所有 agent 必须遵守 `pro/GLOBAL.md` —— 安全边界、上游输出保护、研究过程显示、进度汇报以及通用反模式。各 agent 文件仅定义角色特定行为。

## 编排行为准则

以下规则管辖编排层（本文件）。与 SKILL.md 的丞相规则和 GLOBAL.md 的通用 agent 规则互补。

1. **封驳是灵魂** — 门下省必须认真审议，包含情绪维度。HARD RULE。
2. **御史台 + 谏官自动触发** — 每次三省六部流程后，两者必须运行。不可跳过。HARD RULE。
3. **所有 subagent 输出必须完整展示 + emoji** — 每个 subagent 展示完整过程（🔎/💭/🎯）。收到即刻展示。不得批量。不得压缩。HARD RULE。
4. **Pro 环境强制真 subagent** — 必须启动真正独立的 subagent。禁止单上下文角色模拟。HARD RULE。
5. **数据层降级** — 不可达时标注"⚠️ second-brain 不可用"；Notion 不可用仅影响移动端同步，不影响核心功能。

## 工作流状态机

合法状态转换。任何违规 = 流程错误，御史台必须标记。

| 当前状态 | 可转换至 | 不可跳转至 |
|---------|---------|-----------|
| 早朝准备 | 丞相分诊 | 其他任何状态 |
| 丞相分诊 | 中书省 / 直接处理 / 快车道分析 / 翰林院 / 早朝官 | 六部（快车道除外） |
| 快车道分析（🏃） | 六部执行 → 简报 →（结束或升级至中书省） | 门下省 / 御史台 / 谏官 |
| 中书省拟议 | 门下省审核 | 尚书省 / 六部 |
| 门下省审核 | 尚书省 / 驳回回中书省 | 六部（必须经过派遣） |
| 尚书省派遣 | 六部执行 | 奏折（必须先执行） |
| 六部执行 | 门下省终审 | 奏折（必须先审核） |
| 门下省终审 | 奏折 / 政事堂 | 收朝（必须先出奏折） |
| 奏折 | 御史台 | 收朝（必须先运行御史台） |
| 御史台 | 谏官 | 收朝（必须先运行谏官） |
| 谏官 | 收朝归档 | — |

## 模型独立性

**本文件（GEMINI.md）是 Gemini 专属编排文件。** 所有其他智能 —— 提取规则、lint 规则、角色定义、知识网络、目录结构 —— 均为任何 LLM 可读的纯 markdown。同一套 agent 文件（`pro/agents/*.md`）在 Claude、Gemini 和 Codex 平台间共享。

## 存储后端

Life OS 支持 GitHub、Google Drive 和 Notion 作为存储后端（1、2 或全部 3 个）。用户在首次会话时选择；配置存储在 `_meta/config.md`。多后端模式：写入所有已选后端，从主后端读取（自动：GitHub > GDrive > Notion）。每次会话开始时跨设备同步。

- 标准数据模型和操作：`references/data-model.md`
- 后端适配器：`references/adapter-github.md`、`references/adapter-gdrive.md`、`references/adapter-notion.md`
- 架构和认知管道：`references/data-layer.md`

所有数据读写由早朝官执行；丞相不直接操作存储后端。
