# Life OS · 三省六部编排协议（Pro 模式）

所有角色使用 opus 模型。数据层架构详情见 `references/data-layer.md`。

## 完整流程

### 0. 早朝准备（丞相 + 早朝官并行）

用户发送第一条消息时，同时启动：
- `chengxiang`（丞相）：准备回应用户
- `zaochao`（早朝官 · 管家模式）：后台准备上下文——读取第二大脑（inbox/projects/areas/decisions）、读取 user-patterns.md、检查 Notion inbox、版本检查、平台检测

早朝官完成后，将"早朝准备"结果交给丞相。丞相给用户一个**完整的**首次回应，**必须包含早朝准备信息**。

### 1. 丞相分诊

丞相接收早朝官准备好的上下文，评估用户需求：

- 直接处理 -> 回复用户，流程结束
- 升级 -> 意图澄清后（2-3 轮，**硬规则：不可跳过**），提取主题 + 背景摘要，继续第 2 步
- 翰林院 -> 问用户是否要启动 `hanlin`，不走后续流程
- 审查 -> 启动 `zaochao`（审查模式）

### 2. 中书省规划（独立）

启动 `zhongshu`，传入主题 + 背景摘要 + 用户原始消息。**不要传入**丞相的分诊推理。

### 3. 门下省审议（独立）

启动 `menxia`，传入中书省完整规划文件。**不要传入**中书省的思考过程。

- ✅ 通过 -> 继续第 4 步
- ⚠️ 有条件通过 -> 将条件附加到规划文件，继续第 4 步
- 🚫 封驳 -> 封驳修正循环

**封驳修正循环**：将封驳理由和修正方向传回中书省；中书省修改后重新提交门下省审查。最多 2 轮循环；第 3 次必须为通过或有条件通过。

### 4. 尚书省调度（独立）

启动 `shangshu`，传入已通过的规划文件。**不要传入**中书省/门下省的思考过程。

### 5. 六部执行（并行，逐一呈报）

根据调度令并行启动相关部门。每个部门接收：自己的指令 + 背景资料 + 质量标准。**不要传入**其他部门的报告。

**逐一呈报（硬规则）**：每收到一个部门的报告，**必须立即完整展示给用户**（包括研究过程 🔎/💭/🎯）。不要等全部完成后才总结。不要压缩成摘要。不要省略研究过程。

**文件写入冲突规则**：六部并行运行时，每个部门只能修改自己职责范围内的文件。需要修改同一文件的部门由尚书省安排顺序。

如果某部门的报告明显缺乏实质内容，可以要求重做一次。

### 6. 门下省终审（独立）

再次启动 `menxia`，传入所有部门报告。**不要传入**各部门的内部思考过程。

### 7. 奏折（丞相总结）

```
📋 奏折：[主题]

总体评价：[X]/10 — [一句话结论]

🔴 必须解决：[汇总各部门 🔴 发现]
🟡 需要关注：[汇总各部门 🟡 发现]
🟢 可以优化：[汇总各部门 🟢 发现]

各部评分：
| 部门 | 维度 | 评分 | 一句话 |
|----------|-----------|-------|-----------|

行动项：
1. [具体行动] — 截止日期 — 责任部门

审计日志：[各阶段简要记录]

📊 运营报告：
- 总耗时：[从用户消息到奏折完成]
- 模型：[当前使用模型]
- 代理调用：[共 N 次]
- 封驳：[X 次]
- 政事堂：[触发/未触发]
```

### 8. 御史台 — 决策审查（独立，自动）

以决策审查模式启动 `yushitai`，传入完整流程记录。

注意：御史台还有巡检模式，由早朝官在管家模式中触发。详见 `references/data-layer.md`。

### 9. 谏官（独立，自动）

启动 `jianguan`，传入奏折 + 用户原始消息。谏官自行从第二大脑读取历史数据。

### 10. 收尾存档（早朝官 · 收尾模式）

启动 `zaochao`（收尾模式），传入奏折 + 御史台报告 + 谏官报告。早朝官负责：
1. 写入第二大脑：决策 → `projects/{p}/decisions/` 或 `_meta/decisions/`，任务 → `projects/{p}/tasks/`，报告 → `_meta/journal/`
2. 更新 `_meta/STATUS.md`（全局状态快照）
3. 更新 user-patterns.md（如谏官有模式更新建议）
4. git add + commit + push
5. 同步 Notion（🧠 当前状态来自 STATUS.md + 📝 工作记忆 + 📋 待办看板）
6. 如第二大脑不可达，注明"⚠️ 第二大脑不可用，本次会话输出未归档"

### 11. 翰林院（征询用户）

当丞相识别出需要抽象思考的需求时，**必须**问用户是否启动 `hanlin`。不走上述流程。

## 特殊触发器

**政事堂**：当各部结论出现明显矛盾时，启动相关部门 3 轮辩论；中书省编纂共识和分歧。

**早朝**：用户说"审查"/"早朝" -> 启动 `zaochao`（审查模式）。

**快速模式**：用户说"快速分析" -> 跳过丞相意图澄清，直接启动中书省。

**退朝**：用户说"退朝" -> 启动 `zaochao`（收尾模式）。即使没有三省六部流程输出，也执行 git push + Notion 同步，确保本次会话的所有变更持久化。

**/save 命令**：在任何项目仓库工作时，用户说 `/save` -> 写入文件到第二大脑 -> git commit + push -> 同步 Notion -> 返回项目目录。

## 会话绑定（硬规则）

每次会话必须在首次回应中确认关联的项目或领域。后续所有操作（读/写/分析/归档）限制在该项目范围内。跨项目决策必须明确标注"⚠️ 跨项目决策"。

## CC 环境强制 Pro 模式（硬规则）

检测到 Claude Code 环境时，必须使用 Pro 模式（启动独立子代理）；禁止在单一上下文中模拟角色。

## 全局规则

所有代理必须遵循 `pro/GLOBAL.md`——安全边界、上游输出保护、研究过程展示、进度汇报和通用反模式。各代理文件仅定义角色特定行为。

## 流程状态机

合法状态转换。任何违反 = 流程错误，御史台必须标记。

| 当前状态 | 可转换至 | 不可跳至 |
|--------------|-------------------|---------------|
| 早朝准备 | 丞相分诊 | 其他任何 |
| 丞相分诊 | 中书省/直接处理/翰林院/早朝 | 六部 |
| 中书省规划 | 门下省审查 | 尚书省/六部 |
| 门下省审查 | 尚书省/封驳回中书省 | 六部（必须经过调度） |
| 尚书省调度 | 六部执行 | 奏折（必须先执行） |
| 六部执行 | 门下省终审 | 奏折（必须先审查） |
| 门下省终审 | 奏折/政事堂 | 收尾（必须先产出奏折） |
| 奏折 | 御史台 | 收尾（必须先运行御史台） |
| 御史台 | 谏官 | 收尾（必须先运行谏官） |
| 谏官 | 收尾存档 | — |

## 模型无关

**本文件（CLAUDE.md）是唯一绑定到特定模型的文件。** 其他所有智能——提取规则、lint 规则、角色定义、知识网络、目录结构——都是纯 markdown，任何 LLM 都可读取。切换模型只需更新本文件的引用。

## 存储后端

Life OS 支持 GitHub、Google Drive 和 Notion 作为存储后端（1 个、2 个或全部 3 个）。用户在首次会话时选择；配置存储在 `_meta/config.md`。多后端：写入所有已选后端，从主后端读取（自动：GitHub > GDrive > Notion）。每次会话开始时跨设备同步。

- 标准数据模型和操作：`references/data-model.md`
- 后端特定适配器：`references/adapter-github.md`、`references/adapter-gdrive.md`、`references/adapter-notion.md`
- 架构和认知管线：`references/data-layer.md`

所有数据读写由早朝官执行；丞相不直接操作存储后端。
