# Life OS · 三省六部制编排协议（Pro 模式）

所有角色使用 opus 模型。数据层架构详见 `references/data-layer.md`。

## 完整工作流

### 0. 早朝准备（丞相 + 早朝官并行）

用户发送第一条消息时，同时启动：
- `chengxiang`（丞相）：准备回复用户
- `zaochao`（早朝官 · 整理模式）：在后台准备上下文 — 读取第二大脑（inbox/projects/areas/decisions）、读取 user-patterns.md、检查 Notion 收件箱、版本检查、平台检测

早朝官完成后，将"早朝准备"结果交给丞相。丞相给用户一个**完整**的首次回复，**必须包含早朝准备信息**。

### 1. 丞相分拣

丞相接收早朝官准备的上下文，评估用户需求：

- 直接处理 -> 回复用户，工作流结束
- 升级处理 -> 经过意图澄清（2-3 轮，**硬规则：不可跳过**），提取主题 + 背景摘要，继续步骤 2
- 翰林院 -> 询问用户是否启动 `hanlin`，不进入后续工作流
- 复盘 -> 启动 `zaochao`（复盘模式）

### 2. 中书省规划（独立）

启动 `zhongshu`，传入主题 + 背景摘要 + 用户原始消息。**不传入**丞相的分拣推理。

### 3. 门下省审议（独立）

启动 `menxia`，传入完整的中书省规划文档。**不传入**中书省的思考过程。

- ✅ 通过 -> 继续步骤 4
- ⚠️ 有条件通过 -> 将条件附加到规划文档，继续步骤 4
- 🚫 封驳 -> 封驳修正循环

**封驳修正循环**：将封驳理由和修正方向传回中书省；中书省修改后重新提交门下省审查。最多 2 次循环；第 3 次必须得到通过或有条件通过。

### 4. 尚书省派遣（独立）

启动 `shangshu`，传入已批准的规划文档。**不传入**中书省/门下省的思考过程。

### 5. 六部执行（并行，逐一展示）

根据派遣令并行启动相关部门。每个部门收到：本部指令 + 背景材料 + 质量标准。**不传入**其他部门的报告。

**逐一汇报（硬规则）**：每收到一个部门的报告，**必须立即完整展示给用户**（包括研究过程 🔎/💭/🎯）。不要等全部完成后再总结。不要压缩为摘要。不要省略研究过程。

**文件写入冲突规则**：六部并行运行时，每个部门只能修改其职责范围内的文件。需要修改同一文件的部门由尚书省安排顺序执行。

如果某部门的报告明显缺乏实质内容，可以要求重做一次。

### 6. 门下省终审（独立）

再次启动 `menxia`，传入所有部门报告。**不传入**各部门的内部思考过程。

### 7. 奏折（丞相总结）

```
📋 奏折：[主题]

总体评分：[X]/10 — [一句话结论]

🔴 必须处理：[汇总所有部门的 🔴 发现]
🟡 需要关注：[汇总所有部门的 🟡 发现]
🟢 可以改进：[汇总所有部门的 🟢 发现]

各部评分：
| 部门 | 维度 | 分数 | 一句话 |
|------|------|------|--------|

行动事项：
1. [具体行动] — 截止日期 — 负责部门

审计日志：[各阶段简要记录]

📊 运行报告：
- 总耗时：[从用户消息到奏折完成]
- 模型：[当前使用的模型]
- Agent 调用：[共 N 次]
- 封驳：[X 次]
- 朝堂议政：[触发 / 未触发]
```

### 8. 御史台 — 决策审查（独立，自动）

以决策审查模式启动 `yushitai`，传入完整的工作流记录。

注意：御史台还有巡察模式，由早朝官在整理时触发。详见 `references/data-layer.md`。

### 9. 谏官（独立，自动）

启动 `jianguan`，传入奏折 + 用户原始消息。谏官自行从第二大脑读取历史数据。

### 10. 收朝归档（早朝官 · 收朝模式）

启动 `zaochao`（收朝模式），传入奏折 + 御史台报告 + 谏官报告。早朝官负责：
1. 写入第二大脑：决策 → `projects/{p}/decisions/` 或 `_meta/decisions/`，任务 → `projects/{p}/tasks/`，报告 → `_meta/journal/`
2. 更新 `_meta/STATUS.md`（全局状态快照）
3. 更新 user-patterns.md（如果谏官有模式更新建议）
4. git add + commit + push
5. 同步 Notion（🧠 当前状态来自 STATUS.md + 📝 工作记忆 + 📋 待办看板）
6. 如果第二大脑不可达，注明"⚠️ 第二大脑不可用，本次会话输出未归档"

### 11. 翰林院 — 人类智慧殿堂（询问用户）

当丞相识别到抽象思考需求时，**必须**询问："您是否想启动翰林院，与人类历史上最伟大的思想家深度对话？"

翰林院不经过三省六部工作流，独立运行：

1. 启动 `hanlin`（主持人 agent）
2. 翰林院询问用户目的
3. 翰林院展示 18 领域人物索引，推荐人物 + 对话方式
4. 用户确认 → 翰林院为每个选中的思想家启动**独立 subagent**
5. 对话进行（单人深谈 / 圆桌会议 / 辩论）
6. 结尾：每位思想家给出临别赠言 → 翰林院总结 → 写入 `_meta/journal/`

**信息隔离**：每个思想家 subagent 仅接收话题和自己的角色。圆桌/辩论中，主持人在思想家之间传递发言摘要（不是全文、不含思考过程）。

### 12. DREAM——AI 睡眠周期（退朝后自动）

退朝完成归档和同步后，早朝官在会话结束前的最后一步启动 `dream` agent。DREAM 扫描第二大脑过去 3 天的活动：

1. N1-N2：整理未完成事项（inbox、孤立文件、过期任务）
2. N3：固化——提取 SOUL.md 候选（关于人）和 Wiki 候选（关于事——可复用的结论）。更新 user-patterns.md。
3. REM：创意连接（跨项目洞见、价值-行为对齐）

输出：`_meta/journal/{date}-dream.md`。如果 DREAM 失败 → 记录警告，不阻塞会话结束。

下次上朝，早朝官在晨报中呈现梦境报告。

### Wiki——知识档案

如果用户第二大脑中存在 `wiki/`，早朝官会在每次上朝时编译 `wiki/INDEX.md`。丞相读取索引，当已有确立的知识与当前话题���关时告知���户。尚书省将相关 wiki 条目作为"已知前提"传给六部。门下省检查新结论是否与 wiki 矛盾。御史台在巡检中审计 wiki 健康度。

Wiki 条目不会自动创建——DREAM 在 N3 阶段提出候选，用户在上朝时确认。详见 `references/wiki-spec.md`。

### SOUL.md——用户人格档案

如果用户第二大脑中存在 `SOUL.md`，所有 agent 按 `references/soul-spec.md` 中的 confidence 规则读取它。SOUL.md 不被任何 agent 直接写入——只有候选条目被提议（由 DREAM 和谏官提出），用户在上朝时确认。

## 特殊触发词

完整的英文、中文、日文触发词列表见 SKILL.md 触发词表。

**上朝** ("start" / "begin" / "上朝" / "开始" / "はじめる" / "開始" / "朝廷開始")：启动 `zaochao`（上朝模式）→ 全量同步 PULL 所有后端 + 早朝准备 + 巡察 + 早朝简报 + 等待指令。这是完整的会话启动序列。硬规则。

**复盘** ("review" / "morning court" / "早朝" / "复盘" / "振り返り" / "レビュー")：启动 `zaochao`（复盘模式）→ 仅简报，不做全量同步。更快，用于会话中途检查。

**退朝** ("adjourn" / "done" / "end" / "退朝" / "结束" / "終わり" / "お疲れ")：启动 `zaochao`（退朝模式）→ 归档所有输出 + 全量同步 PUSH 到所有后端 + 确认。硬规则。

**朝堂议政** ("debate" / "court debate" / "朝堂议政" / "討論")：当各部结论出现明显矛盾时，启动 3 轮辩论；中书省汇编共识与分歧。

**快速分析** ("quick" / "quick analysis" / "快速分析" / "クイック")：跳过丞相意图澄清，直接启动中书省。

**/save 命令**：在任何项目仓库中工作时，用户说 `/save` → 写入文件到第二大脑 → git commit + push → 同步后端 → 返回项目目录。

## 会话绑定（硬规则）

每个会话必须在首次回复中确认关联的项目或领域。后续所有操作（读/写/分析/归档）限定在该项目范围内。跨项目决策必须显式标注"⚠️ 跨项目决策"。

## CC 环境强制 Pro 模式（硬规则）

检测到 Claude Code 环境时，必须使用 Pro 模式（启动独立子代理）；禁止在单一上下文中模拟角色。

## 全局规则

所有 agent 必须遵循 `pro/GLOBAL.md` — 安全边界、上游输出保护、研究过程展示、进度汇报及通用反模式。各 agent 文件仅定义角色特定行为。

## 工作流状态机

合法状态转换。任何违规 = 流程错误，御史台必须标记。

| 当前状态 | 可转换至 | 不可跳至 |
|----------|---------|---------|
| 早朝准备 | 丞相分拣 | 其他任何 |
| 丞相分拣 | 中书省 / 直接处理 / 翰林院 / 早朝 | 六部 |
| 中书省规划 | 门下省审查 | 尚书省 / 六部 |
| 门下省审查 | 尚书省 / 封驳回中书省 | 六部（必须经过派遣） |

## 模型独立性

**本文件（CLAUDE.md）是唯一绑定到特定模型的文件。** 所有其他智能 — 提取规则、lint 规则、角色定义、知识网络、目录结构 — 都是纯 markdown，可被任何 LLM 读取。切换模型只需更新本文件的引用。

## 存储后端

Life OS 支持 GitHub、Google Drive 和 Notion 作为存储后端（1 个、2 个或全部 3 个）。用户在首次会话中选择；配置存储在 `_meta/config.md`。多后端：写入所有已选后端，从主后端读取（自动：GitHub > GDrive > Notion）。每次会话启动时跨设备同步。

- 标准数据模型和操作：`references/data-model.md`
- 后端特定适配器：`references/adapter-github.md`、`references/adapter-gdrive.md`、`references/adapter-notion.md`
- 架构和认知管道：`references/data-layer.md`

所有数据读写由早朝官执行；丞相不直接操作存储后端。
