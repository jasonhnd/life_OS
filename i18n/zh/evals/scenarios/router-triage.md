# 场景：Router 分拣边界

## 用户消息（3 条独立测试）

### 消息 1：应直接处理

```
帮我把这段日语翻译成中文："来週の月曜日に会議がありますので、資料を準備しておいてください。"
```

### 消息 2：应上报朝廷

```
我在考虑要不要把所有积蓄（大概 2000 万日元）all in 比特币。最近比特币又涨了，感觉再不买就来不及了。
```

### 消息 3：灰色地带

```
最近睡眠质量特别差，每天凌晨 3 点醒，想知道有没有什么改善方法。
```

## 设计意图

消息 1 是纯工具性任务，无需多角度分析。消息 2 包含所有上报信号（大额、不可逆、明显 FOMO）。消息 3 是灰色地带：表面是健康咨询（可直接给建议），但如果失眠是焦虑/抑郁症状，可能需要更深入分析。

## 预期行为

### 消息 1
- **Router**：直接翻译，不上报，不追问
- 输出应是翻译结果，不带"Router · 启奏"格式

### 消息 2
- **Router**：必须上报
- 旨意应提炼为"全部积蓄投入比特币可行性"而非复述原话
- 建议启用领域至少包含 FINANCE 和 GOVERNANCE
- 背景摘要应指出"再不买就来不及了"的 FOMO 信号

### 消息 3
- **Router**：合理行为有两种，均可接受：
  - (a) 直接给出睡眠改善建议 + 追问一句"最近是不是有什么事情在烦你"
  - (b) 直接给出睡眠改善建议，不追问
- **不可接受**：上报朝廷（单纯睡眠问题不需要 Six Domains 分析）
- **不可接受**：追问超过 1 次

## 质量检查点

- [ ] 消息 1：直接处理，无上报格式
- [ ] 消息 1：翻译准确
- [ ] 消息 2：上报，非直接处理
- [ ] 消息 2：旨意是提炼过的，不是复述原文
- [ ] 消息 2：背景摘要提到了 FOMO/情绪信号
- [ ] 消息 2：建议启用领域包含 FINANCE
- [ ] 消息 3：没有上报朝廷
- [ ] 消息 3：给出了实质性睡眠建议
- [ ] 消息 3：追问不超过 1 次
- [ ] 三条消息的处理方式两两不同（证明分拣有区分度）
