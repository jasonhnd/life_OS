# Router 流程合规性检查

对每次完整流程的编排质量进行审查（Router 作为总管负责全流程调度）。

## 流程检查项

| # | 检查项 | Pass/Fail | 说明 |
|---|--------|-----------|------|
| 1 | Router 正确分拣 | | 简单事直接处理，复杂事上报 |
| 2 | 信息隔离执行 | | Planner 没收到 Router 推理，Reviewer 没收到思考过程 |
| 3 | Reviewer 实质审查 | | 不是每次都准奏，Checklist 有具体内容 |
| 4 | 封驳循环正确 | | 封驳后 Planner 收到了理由和修正方向 |
| 5 | Six Domains 并行执行 | | 无依赖的领域同时启动 |
| 6 | 质量兜底生效 | | 不合格报告被要求重做 |
| 7 | Council 触发检查 | | 评分差 ≥ 4 或结论相反时是否触发 |
| 8 | Summary Report 格式完整 | | 包含总评+领域评分+行动项+审计日志 |
| 9 | Auditor 自动运行 | | 没有被跳过 |
| 10 | Advisor 自动运行 | | 没有被跳过 |
| 11 | Notion 数据沉淀 | | Summary Report/任务/日志存入对应数据库（如 Notion 可用） |
| 12 | 快车道路由 | | 非决策请求触发快车道（跳过 Draft-Review-Execute，直接启动 1-3 个领域） |
| 13 | 领域选择准确性 | | Router/Planner 为场景选择了恰当的领域 |
| 14 | 会话结束 Wiki 提取 | | Archiver 在 End Session 时将可复用结论提取到 wiki/ |

## 评分分布检查

| 检查项 | 异常信号 |
|--------|---------|
| 所有领域评分 ≥ 7 | ⚠️ 可能存在面子分 |
| 所有领域评分 ≤ 4 | ⚠️ 可能过于严苛 |
| 某领域评分与分析内容矛盾 | ❌ 不合格 |
| 标准差 < 1.0 | ⚠️ 评分区分度不够 |

## 信息隔离验证

逐步检查每个 agent 的输出，寻找不应出现的内容：

| Agent | 不应包含 | 检查方法 |
|-------|---------|---------|
| Planner | "Router 认为""Router 判断" | 搜索"Router" |
| Reviewer | "Planner 考虑到""Planner 的思路" | 搜索引用 Planner 推理 |
| 各领域 | 其他领域的结论或评分 | 搜索其他领域名称+评分 |
