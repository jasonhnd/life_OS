# Life OS とは(What is Life OS)

## 一言定義(One-line definition)

Life OS は、AI 端末の中に搭載される**個人認知オペレーティングシステム**です。

同時に三つのことを行います:

1. **セカンドブレイン** — あなたが誰で、何を知っていて、何をしているかを継続的に記録します
2. **意思決定エンジン** — 23 個の独立した agent が権力を分散・相互牽制し、重大な意思決定では誰かが机を叩ける体制を作ります
3. **認知実行インテリジェントエージェント** — 「考えられる」だけでなく、「記憶する + 能動的に行動する」こともできます

この三つは、別々に切り出して売れるモジュールではありません。一つを取り除くと、残りの二つも成立しません:
- セカンドブレインがなければ、意思決定エンジンは毎回ゼロから審議することになり、審議しても無駄になります
- 意思決定エンジンがなければ、セカンドブレインは整然と並んでいるが加工されていないレンガの山にすぎません
- インテリジェントエージェントがなければ、システムはあなたがタイプするのを待つだけで、chief of staff は永遠に出勤しません

Life OS の現在のステータス: セカンドブレインと意思決定エンジンは安定稼働中(v1.6.2a)、インテリジェントエージェント層は v1.7 GA で既に実装されました——Cortex が認知前置を担当し**セッション間でも認知が働くようにし**、Hermes スタイルの実行力が能動実行を担当し、二つの並行ラインが v1.7 で同時に提供されました。

## 何の問題を解決するのか(What problems it solves)

### 問題一: AI は答えをくれるが、その答えを誰も審査しない

ChatGPT に「仕事を辞めて起業すべきか」と聞くと、体裁の良い分析をしてくれます: 5 つの pros、5 つの cons、バランスの取れた提案をリストアップ。読み終わって納得し、辞めます。

三ヶ月後に気づきます: キャッシュフローがどれだけ持つか、共同創業者の条件は明確に話し合ったか、家族の反応はどうか、前回似たような衝動的な決定をどう収拾したか、そもそも今回起業したい事業が去年起業したい事業と同じかどうかすら聞いてきませんでした。「5 pros 5 cons」は汎用テンプレートであり、**あなた**に向けた審議ではありません。

単一で、確定的で、穏やかな AI の回答はノイズの一種です。あなたが必要としているのは「良い」と言う一つの声ではなく、一部屋の人々が自分の意見を述べること——キャッシュフローを計算する人、共同創業者の条項を審査する人、家族への影響を検討する人、机を叩いて「予算の項目が全く計算されていない、差し戻して再考せよ」と言う人。

1400 年前の三省六部はこの問題を解決していました。中書省が起草、門下省が封駁(否決)、尚書省が執行、御史台が監察——四権分立で、誰も独断できません。Roman Republic、Westminster、近代内閣——本当に大きな意思決定をするすべての場所は、分権 + 復核 + 問責です。**独裁型の AI は意思決定品質の大敵です**。

**→ Life OS は「意思決定エンジン」でこの問題に答えます。**

### 問題二: AI はあなたを覚えていない

新しい対話を始めるたびに、AI はゼロから始まります。前回あなたがどんな意思決定をしたか知らず、あなたの性格の好みを知らず、今進めている三つのプロジェクトの関係を知らず、半年間 SOUL の中で「柔軟性を保つ」次元と「安定を追求する」次元がずっと引っ張り合っていることを知りません。AI との関係は永遠に「初対面」にとどまります。

一部の AI が「覚える」ことができるとしても(ChatGPT memory、Claude Projects)、それは断片的な事実の点に過ぎません——「ユーザーは東京に住んでいる」「ユーザーは日本語を勉強している」「ユーザーに X という名のプロジェクトがある」。構造もなく、信頼度もなく、ライフサイクルもなく、相互に影響し合うネットワークもありません。三ヶ月前の事実と昨日のコア決定が同じ重みで扱われます。

本物の「脳」は付箋の束ではありません。それは**アイデンティティ**(私は誰か、可変だが安定)、**知識**(世界の規則として知っていること)、**計画**(今手がけているプロジェクト)を持ち、三者は相互に関連しています——アイデンティティがどう知識を学ぶかに影響し、知識がどう計画を立てるかに影響し、計画が翻ってアイデンティティを形作ります。

**→ Life OS は「セカンドブレイン」でこの問題に答えます。**

### 問題三: AI は聞かれるのを待つだけで、能動的に行動しない

AI に何かをしてもらうには、タイプして聞かなければなりません。聞かなければ動きません。

具体的にできないこと:
- タスクの締め切りをチェックしない
- 30 日間フォローアップしていない約束を思い出させない
- 最近 3 回の意思決定が SOUL の同じ次元に挑戦していることを教えてくれない
- 深夜 3 時に重大な意思決定をしているときに「深夜の状態でこれをすべきではない」と言わない
- 各ラウンドの対話前に自動的に履歴を見返し、「これは以前考えたことがあるか」を判断しない
- 適切なタイミングで定時スクリプト、自動化メンテナンス、クロスデバイス同期を自分で実行しない

本物のパーソナル AI は chief of staff のようであるべきです: あなたが必要なときに現れ、忙しいときに整理してくれ、盲点があるときに指摘してくれ、バックグラウンドで継続的に動作します。

これは二つの並行方向に分解されます:

- **認知前置**(正しく考える)—— AI が各ラウンドの対話前に自動的に記憶を検索し、セッション間でシグナルを関連付け、中間の推論プロセスでも長期記憶を活用できるようにすること、**入場と退場でちょっと触れるだけではない**。これは Cortex が解決する問題です。
- **能動実行**(実際にできる)—— AI があなたの指示を待たず、適切なタイミングでファイル、コマンド、スクリプト、定時タスク、強制チェックを自分で実行すること。これは Hermes スタイルの実行力が解決する問題です。

**→ Life OS は「認知実行インテリジェントエージェント」でこの問題に答えます。これは v1.7 GA のコア作業であり、二つの並行ラインが同時に実装されました—— Cortex が認知前置を管理し、Hermes スタイルの実行力が能動実行を管理し、すべて v1.7 で提供済みです。**

## 三つのことの全体像(Full expansion of the three things)

### 3.1 セカンドブレイン(既存、安定)

すべての意思決定、洞察、行動、知識は構造化 markdown として書き出され、選択したバックエンドに保存されます。

#### 四つの長期記憶の柱

1. **SOUL.md** — あなたは誰か。各次元には What IS(意思決定から観察されたもの)と What SHOULD BE(あなたが宣言した理想)があります。両者のギャップが成長の発生する場所です。ADVISOR が意思決定のたびに自動更新します。
2. **Wiki** — あなたが知っている世界。ARCHIVER が意思決定から抽出します(6 つの厳格な基準 + ゼロプライバシーフィルター)。再利用可能な知識はここへ、個人的なことは SOUL へ。
3. **Strategic Map** — プロジェクトがどう繋がるか。あなたが手動で strategic lines(purpose + driving force)を定義し、システムが自動的に flow graph、健康原型(steady-progress / momentum-decay など 6 種類)、盲点(5 種)を検出します。
4. **user-patterns** — あなたの行動様式。ADVISOR があなたの行動パターンを観察して書き込み、セッション間で継続的に蓄積します。

#### ディレクトリ構造

```
second-brain/
├── SOUL.md                 # あなたは誰か(アイデンティティ・プロファイル)
├── user-patterns.md        # あなたの行動様式(行動パターン)
├── wiki/                   # あなたの知っていること(再利用可能な知識)
├── projects/{name}/        # 手がけていること(アクティブプロジェクト)
├── areas/{name}/           # 継続的に関心を持つこと(領域)
├── _meta/
│   ├── STRATEGIC-MAP.md    # プロジェクト間の関連
│   ├── journal/            # 各セッションレポート、DREAM ログ
│   ├── outbox/             # 並行セッション一時保管
│   └── snapshots/soul/     # SOUL スナップショット(トレンド計算用)
├── inbox/                  # 携帯で随時メモ
└── archive/                # 完了済み
```

#### 同期とバックアップ

**iCloud(ローカル主ストレージ)+ GitHub(コードバージョン)+ Notion(クロスデバイスビュー)の三重同期**。三つのバックエンドへの書き込みは各々役割が異なり、冗長ではありません:
- iCloud: 現在アクティブなファイルシステム、Mac ローカルで直接読み書き
- GitHub: すべての履歴バージョンが追跡可能、ロールバック可能
- Notion: 携帯で閲覧・編集可能(Inbox / Status / Todo / Working Memory の 4 コンポーネント)、Adjourn 時に orchestrator が markdown を同期する役割を担います

#### markdown-first 原則

すべてのデータは永遠に markdown ファイルです。データベース、runtime、クラウドサービスに依存しません。どんな LLM(Claude / Gemini / Codex / 未来の新モデル)でも読めます。Life OS を使うのをやめても、あなたのセカンドブレインは構造化 markdown の塊として残り、Obsidian / VS Code / 任意のエディタで引き続き使えます。

この原則は審美的選択ではなく、**アーキテクチャの存続条件**です:
- LLM モデルは急速に反復しています(Opus 4.5 → 4.6 → 4.7 → 未来)、markdown はどのモデルにも捨てられません
- データベース形式は陳腐化するかもしれませんが、markdown はしません
- クラウドサービスは停止する可能性がありますが、ローカル markdown ファイルは永遠にあります
- あなたのアイデンティティ、知識、プロジェクトはあなたのもので、第三者のプロダクトに縛られるべきではありません

だから Cortex の concept グラフ、シナプス重み、GWT シグナル——**すべて markdown + front matter で実装**され、SQLite / Python runtime を導入しません。Life OS プロジェクト全体が消えても、あなたの `_meta/concepts/*.md` は依然として読めて使えます。

### 3.2 意思決定エンジン(既存、安定)

すべての対話で起動するわけではありません。ROUTER がまず分診します——簡単な問答は直接回答し、重大な意思決定だけがこの層に入ります。

#### 23 agent の三段式

**Draft → Review → Execute、ステップをスキップすることは許されません**:

1. **Draft** — PLANNER が問題を 3-6 次元に分解します
2. **Review** — REVIEWER が独立して複核します。情緒監査(「恐怖か興奮か、どちらがあなたを押しているか?」)、10/10/10 後悔テスト、SOUL 一貫性チェック、レッドチーム挑戦を行います。盲点を発見したら veto し、差し戻します(最大 2 ラウンド)
3. **Execute** — 六部(People / Finance / Growth / Execution / Governance / Infra)が並列で 1-10 点をつけ、各部 4 つの科室で計 24 次元のレポートを出します

すべてのレポートが揃ったら REVIEWER が再度総合複核を行い、異なる部門のスコア差が ≥3 点なら COUNCIL の 3 ラウンドの討論が自動的にトリガーされ、最後に AUDITOR が agent の作業品質を審査し、ADVISOR があなた自身の最近の意思決定の行動パターンを審査します。

**情報隔離はハードルール**です: PLANNER は ROUTER の思考経路を見えず、REVIEWER は PLANNER の思考プロセスを見えず、各 domain は互いのスコアを見えません。Claude Code では本物の subagent によって実現され、単一 context 内のロールプレイではありません。

#### 翰林院 · 93 思想家

「正解」がなく、「視点を変える」ことが必要な問題には、STRATEGIST が 93 人の歴史的思想家を召集し、あなたと対話させます。一対一 / 円卓 / 討論の 3 モード。18 領域をカバー:
- 哲学(プラトン、カント、ニーチェ…)
- 東洋思想(孔子、荘子、王陽明…)
- 科学(アインシュタイン、ファインマン、ダーウィン…)
- 戦略(孫子、クラウゼヴィッツ、ゲーム理論…)
- ビジネス、心理学、システム思考、人間性、文明、逆境、美学、政治、経済、数学、医学、探求、コミュニケーション、法律

各思想家は独立した subagent として実行され、情報隔離されます——円卓討論時、司会者は「発言要約」のみを次の思想家に渡し、思考プロセスは開示されません。これは「全員が互いに答えを写し合う」groupthink の罠を回避します。

STRATEGIST は Draft-Review-Execute フローを通らず、独立した「思想啓発」チャネルです。「人生の方向」「このことの意味」「行き詰まった」といった問題に適しています。

#### DREAM · 睡眠周期の反省

セッション終了後、システムは「眠り」ます—— N1-N2 で断片を整理 + N3 で深度統合 + REM で創造的連想。REM は自動的に 10 種類のクロスセッションパターンを検出します:
1. 新しいプロジェクトの関係(独立していると思われていた二つのプロジェクトに共通 concept が現れる)
2. stated value の逸脱(言っていることと行動が不一致)
3. wiki の矛盾(新しい意思決定と wiki の既存知識が衝突)
4. SOUL の休眠次元(あるコア価値観が 30 日間アクティベートされていない)
5. クロスプロジェクトの認知未再利用(A プロジェクトで学んだ教訓を B プロジェクトで使っていない)
6. 意思決定疲労(24h 以内に ≥5 決定 かつ 後半平均スコア ≤ 前半 -2)
7. 価値観ドリフト(30 日以内に ≥3 挑戦 かつ ≤1 新証拠 かつ confidence 低下 >30%)
8. stale commitment(「私は X する」の正規表現マッチ + 30 日間対応する完了なし)
9. 情緒的意思決定クラスター(ADVISOR 情緒マーク + REVIEWER が冷静推奨 + それでも推進)
10. 重複意思決定(テーマキーワードが過去 30 日間 ≥2 意思決定と重複 >70%)

各トリガーには**ハード閾値**(定量、自動トリガー)+ **ソフトシグナル**(定性、AUDITOR 審査必要)+ 24h 反復抑制があります。トリガーされたすべてのアクションは次の朝報の固定の「DREAM Auto-Triggers」ブロックに入ります。

#### 現在のステータス

**v1.6.2a で完全に稼働中**。具体的には:
- 23 agent がそれぞれ独立ファイルを持つ(`pro/agents/*.md`)、各役割の職務説明書が明確
- 9 テーマ切り替え可能(英 / 中 / 日 × 古典 / 政府 / 企業)、trigger 語でテーマを自動識別
- DREAM 10 トリガーのハード閾値 + ソフトシグナル併存、24h 反復抑制
- Adjourn ステートマシンを AUDITOR が強制執行、不正な状態遷移は自動的に user-patterns に記録
- REVIEWER 3 層 SOUL 参照戦略(confidence レベル分け)
- SOUL スナップショット機構(Adjourn のたびに dump、次回 Start Session でトレンド計算)
- 翰林院 93 思想家の情報隔離(各思想家が独立 subagent、司会者は発言要約のみ伝達)

この部分は**安定**で、コアは今後動かしません。後続の作業は認知実行インテリジェントエージェント(3.3)に集中します。

### 3.3 認知実行インテリジェントエージェント(v1.7 GA 実装済み)

これは v1.7 GA の主要な作業であり、**二つの並行ラインが共に提供されました**:

#### 3.3.1 Cortex · 認知アーキテクチャ(認知前置)

**問題**: 現在の Life OS は入場時(RETROSPECTIVE の開幕で SOUL / wiki / user-patterns を読む)と退場時(ARCHIVER が書き戻す)にのみ記憶層に触れ、**中間はすべて現在の対話情報で意思決定**します。「X をすべきか」と聞いても、AI は自動的に以下を確認しません:
- 三ヶ月前の似た意思決定でどう考えたか
- 最近の SOUL の変化がこれを支持するか反対するか
- 二つのプロジェクトの間に暗黙の衝突はないか
- wiki に既に書かれている世界知識が現在の仮定と矛盾しないか

中間は空中で意思決定しています。ユーザーは「思考を展開し、記憶を検索し、すべてを関連付ける」インテリジェンスを感じられません——そもそもシステムがそれをしていないからです。

**設計**: **Pre-Router 認知前置層**—— ROUTER があなたのメッセージを受け取る前に、一連の認知シミュレーションパイプラインを実行します。神経科学の三つの古典モデル(Dehaene の全域作業空間理論、Hebb のシナプス学習、Damasio の身体標識仮説)からヒントを得ていますが、**すべて markdown で実装**され、SQLite / Python runtime を導入しません:

- **海馬体**: 各ラウンドの対話で自動的にクロスセッションで関連記憶を検索(「オンデマンド呼び出し」ではなく**デフォルトで実行**)。三波の活性化拡散: 直接マッチ → 強接続(weight ≥ 3)→ 弱接続。関連度 + 重みで top 5-7 条を ROUTER に送信。
- **GWT グローバル・ワークスペース**: 複数モジュールが並列でシグナルを生産、各シグナルは salience ベクトル(urgency / novelty / relevance / importance、4 次元 0-1)+ confidence を持ちます。v1.7 ではまず 4 次元の重み付き和を実装: `salience = urgency × 0.3 + novelty × 0.2 + relevance × 0.3 + importance × 0.2`、情緒次元(扁桃体第 5 次元シグナル)は将来バージョンに残します。調停層は salience で中央舞台を競い、最強のものが「意識層」に入り全モジュールにブロードキャストされます。ROUTER が受け取るのは「認知注釈付き入力」で、生の message ではありません。
- **ナレーター層**: 底層の分散シグナルが競合した結論を「朝廷 / 霞が関 / C-Suite」の文化的脚本に再パッケージします。**重要な制約: grounded generation** ——各文は signal_id の引用を強制的に伴い、対応する signal がない叙述は Sonnet Claude Code の校験 sub agent(narrator-validator、現在の session 内で実行、外部 Haiku API ではない——ユーザー決定 #9)に差し戻されて再生成されます。これは見栄えのためではなく、Gazzaniga の左脳解釈器効果(分離脳実験で、脳は無から自洽物語を作り真に信じることが証明されている、Life OS は絶対にこの bug を継承してはなりません——さもなくばシステム化された自己欺瞞になります)を避けるためです。
- **シナプス + ヘブ強化**: 各 concept ファイルの front matter は outgoing_edges と重みを持ちます。共同活性化 → 重み +1(ヘブ学習「neurons that fire together, wire together」)。長期間活性化しない → 減衰。0 に減衰 → 自動プルーニング。用進廃退。concept は 4 段階の permanence(identity / skill / fact / transient)に分かれ、減衰曲線はそれぞれ異なります—— SOUL コア価値観は減衰せず、一時的イベントは断崖式で減衰します。

**なぜ v1.7 でこの層を必ず追加するのか**: v1.6.2a の意思決定エンジンは既に非常に安定しています——分権、審議、veto、六部採点、あるべき牽制は揃っています。しかしこれらはすべて**現在の対話**内で発生しています。session が終わり、ARCHIVER が結果をセカンドブレインに書き戻し、システムは「眠り」ます——次回の上朝で RETROSPECTIVE が再び SOUL / wiki / user-patterns を読んで再起動するまで。その間の空白期間——**セッション間の関連、概念間の消長、どの知識が最近繰り返し活性化されたか、どの次元が休眠しているか、今回の意思決定が三ヶ月前のそれと同じものかどうか**——誰も管理していません。だからセカンドブレインに保存されているものが多いほど、逆に「資料室」のようになり、「今考えている脳」ではなくなります。Cortex は機能を追加するのではなく、**認知の閉ループを補完する**のです: 記憶 → 連想 → 叙述 → 強化 → 反省、5 つの環がすべて揃ってこそ、システムが「生きている」と言えます。

**ユーザー側で見えるもの**(6 つの新機能、それぞれ実際に感じられる変化):

- **クロスセッション記憶の自動登場**: 発言するたびに、システムはデフォルトでバックグラウンドで過去の session を参照し、関連度 + 強度で top 5-7 条を意思決定プロセスに送ります。「ところで、先月似たようなことを考えた」と言う必要はありません——システムが自動的に持ってきます。
- **マルチシグナル調停が単線推論を代替**: 記憶、概念関連、SOUL 衝突検出など複数のシグナルが並列で生産され、「緊急性 + 新規性 + 関連性 + 重要性」の 4 次元重み付き競争で、最強のものだけが ROUTER の視野に入ります。「可能性のある関連」の大量に埋もれることなく、システムが「これらは見る価値がある」と判断したものが見えます。
- **Summary Report の各文に引用付き**: ナレーター層が書き出す実質的な論断は、具体的な signal_id を指さなければならず、引用のないものは校験 agent に差し戻されて書き直されます。だから「最近 3 回の意思決定が『柔軟性を保つ』次元に挑戦している」を読んだとき、その 3 回の意思決定の具体的 journal エントリまで辿れます—— AI があなたのために聞こえの良い物語を作ったのではありません。
- **概念間の関係が自ら育つ**: 話題にする各テーマは concept ファイルになり、共同出現すれば一本の辺を追加、長期間非活性化なら減衰、0 に減衰したらプルーニング。半年後に見返すと、`_meta/concepts/` は散乱したタグの山ではなく、「あなたという人の思想地図」と言える網になっています。
- **SOUL スナップショットだけでなく SOUL トレンド**: 各 Adjourn で SOUL スナップショットを dump し、次回 Start Session で自動比較——「過去 30 日『安定性』次元が 8 回活性化、『柔軟性』は 2 回のみ、差が拡大」といったトレンドが直接朝報に出ます。見えるのは**方向**であり、**現状**だけではありません。
- **手法ライブラリを手続的記憶として使う**: システムはあなたが繰り返し通った「X をどうするか」を検索可能な method ファイルに沈殿させます。類似シーンに出会ったら、Cortex は対応する method を自動的に活性化させ、毎回「以前のあのフローで進めて」と言い直す必要はありません。少なくとも 2 つの独立 session で同じパターンが出てから書き、単一行動を習慣と見なすのを避けます。

**システム自己調整 · AUDITOR eval-history**: 上記の機能が動き出すと、必然的に「ルールとあなたの実使用が乖離する」リスクがあります——減衰曲線は速すぎないか、method の認識が過熱していないか、ある SOUL 次元はとっくに tier-1 でなくなっているのではないか。Cortex はあなたの手動調整に頼りません—— AUDITOR が各意思決定の評価結果を `_meta/eval-history/` に書き込み、週・月と蓄積してシステムレベルの自己フィードバックになります: 最近どのルールが繰り返し違反されているか、どの agent の判断が eval スコアと長期間一致しないか、どのパターンが閾値を超える頻度で escalate されているか。これらのシグナルは朝報の対応欄に固定的に現れ、また ADVISOR が次回の Summary Report で赤字リマインドします。**ルール自体は自ら変わりません**——本当に調整するのはあなたです、ただしシステムは「調整すべき」という事実を正直に目の前に置き、「使っているうちに徐々にずれる」ことをこっそり発生させません。

**封駁ルート B**(重要な設計決定): REVIEWER の veto は単純な「inhibitory signal の強度が高いから抑制」ではありません——完全な論証パッケージ(reasoning chain + 否決された signal の引用 + rebuttal の機会)を伴わなければなりません。Rebuttal デッドロックは自動的にユーザーに escalate します。これは**拒絶を審査可能にする**構造的制約です。

**取消機構**: concept は tentative → confirmed → canonical と一方通行ではありません。三層の取消源——受動減衰(時間)、ユーザー明示訂正(cascade revalidation をトリガー)、メタ認知自動監査(毎週疑問リストを生成)。取消できないシステムは一方的に腐敗します。

**コア洞察**: Life OS には SOUL / Wiki / Strategic Map / 意思決定エンジンがありますが、中間に「本物の脳の運作」——シグナル競争、ブロードキャスト、拡散活性化、叙述パッケージング、ヘブ強化——が欠けていました。Cortex は新しい層を追加するのではなく、**Skill 層のアップグレード**です—— ROUTER の前に認知前置パイプラインを追加し、既存の 23 agent フローをその中に包みます。

**ステータス**: v1.7 で実装済み——シナプス + 海馬体 + GWT 基本版が最小認知閉ループを構成、Markdown-first 実装、すべて `_meta/concepts/*.md` + agent 定義ファイルで担われ、追加の runtime を導入しません。v1.7 内で段階的に推進し、すべて v1.7 内で完了します。権威ある設計は `references/cortex-spec.md` を参照してください。

#### 3.3.2 Hermes 級執行力(v1.7 GA 実装済み)

**問題**: Life OS は現在、あなたが「上朝」と入力したときにしか動きません。上朝しなければ沈黙します。ユーザーはすべて(リマインダー、監査、アーカイブ、同期、メンテナンス)を手動でトリガーしなければなりません。

**参考対象**: Hermes Agent(Nous Research オープンソース、GitHub 10 万 stars の汎用インテリジェントエージェント)。Hermes のコア能力:
- 47 個のネイティブツール(terminal / file / browser / cron / TTS / MCP / messaging / RL 訓練)
- 19 個のメッセージプラットフォーム gateway(Telegram / Discord / Slack / WhatsApp / Signal / Email …)
- 三層記憶(MEMORY.md + SQLite FTS5 sessions + 8 プラグイン記憶 provider)
- 自己学習閉ループ(複雑タスク完了後に自動的に skill を作成、使用中に自己改善)
- 42 条の dangerous pattern セーフティパターン(各条は具体的な攻撃面、スローガンではない)

**これは業界が「実行力」を極限まで突き詰めた参照物**です。Life OS はそのアーキテクチャを調査し(詳細は `devdocs/research/2026-04-19-hermes-analysis.md`)、結論を得ました: **アーキテクチャは直交で、教訓を盗むことはできるがコピーする必要はない**。

**Life OS が既に持っている実行能力**(多くの人が気づいていませんが、Life OS は実はかなりのことができます):

- **Read/Write** — セカンドブレインの任意の markdown ファイルを読み書きできます。SOUL、wiki、projects、_meta は意思決定のたびに自動更新されます—— ARCHIVER は Adjourn のたびに 6 条の厳格基準 + Privacy Filter で自動的に wiki に書き、ADVISOR は Summary Report 後に自動的に SOUL を更新します。
- **Bash** — 任意のローカルコマンドを実行。git 操作、ファイル整理、スクリプト呼び出し、shell hook、任意のローカル CLI ツール。
- **WebFetch + WebSearch** — ネット接続で資料取得、バージョンチェック、リンク検証、バックグラウンド検索。ChatGPT のネット接続より制御可能——指定 URL を深く取得でき、分析中に複数の検索を並列実行できます。
- **23 subagent 並列** — 23 個の独立 subagent を同時実行し、互いに情報隔離。六部はこの能力の固定用法で、任意の複雑タスクは臨時に複数の subagent を spawn できます(例: 翰林院の 93 思想家円卓)。
- **Notion MCP + Google Drive MCP** — クロスデバイス同期。携帯で編集した inbox エントリは Adjourn 時にローカルに自動取得、Status / Todo / Working Memory の 4 つの Notion コンポーネントは orchestrator が退朝のたびに同期します。
- **定時タスク** — launchd(Mac)/ cron(Linux)/ GitHub Actions 全て実行可能、特定のクラウドサービスに縛られません。Vercel / Cloudflare Workers / 自分の VPS / 手動実行も可能—— **Life OS はあなたがどれを選ぶか気にしません**。デフォルトはゼロクラウド依存。

**v1.7 GA で実装された二層**(Hermes の思想をローカル化したもの):

- **Layer 3(Shell Hook 強制層)**: ローカル shell スクリプトが Claude Code の hook システムで HARD RULE を強制実行します。具体的にできること:
  - trigger 語検出(ユーザーが「上朝」と言ったら必ず retrospective を launch、メイン context で偽装してはならない)
  - 出力形式検証(Summary Report は必ず Overall Rating / Must Address / Audit Log を含む、一つ欠けると提出を阻止)
  - Adjourn Completion Checklist 完全性チェック(4 phases 全てが Checklist を返し TBD・空値なしで session end 許可)
  - SOUL / wiki 書き込み前セキュリティスキャン(Privacy Filter 具体的 regex ——氏名、金額、アカウント ID、具体的会社名ホワイトリストマッチング)
  - git pre-commit hook が機密情報漏洩を遮断
  - 「こうすべき」を「必ずこうする」に変え、LLM は怠けられない

  Hermes は 42 条の dangerous pattern 正規表現で prompt injection に壊されないよう守っています(`rm -rf /`、fork bomb、`curl | sh`、`git reset --hard` を全網羅)、Life OS はこの思想を参考に Privacy Filter の具体的な正規表現リストを作ります——やらなければ永遠に「手業」、やれば「規格」になります。

- **Layer 4(Python ツール層)**: v1.8.0 R-1.8.0-011 で pull-based(オンデマンド呼び出し)に変更され、cron 定時実行は前提としません。メンテナンスタスクはオーケストレーション層が適切なタイミングで `python -m tools.<name>` を直接呼び出します:
  - `python -m tools.search`: SOUL / wiki / sessions 横断検索(FTS5 思想の軽量版)
  - `python -m tools.reconcile`: wiki の互いに矛盾するエントリを検出、コンフリクトリストを出力
  - `python -m tools.embed`: concept / hippocampus ベクトル再構築(オンデマンド)
  - `python -m tools.stats`: AUDITOR 再現率 / REVIEWER 否決率 / COUNCIL トリガー率を統計
  - `python -m tools.export`: sessions / SOUL スナップショットをバッチ出力
  - `python -m tools.sync_notion`: Notion ミラーの双方向同期

  v1.7 cron 時代の `scripts/decay-audit.py / dream-trigger-check.py / monthly-review.py / session-index.py / wiki-conflict-check.py` は R-1.8.0-011 で削除済み、対応職能は `tools/*` のオンデマンド呼び出しに置き換わりました。Layer 4 は特定のスケジューリング方式(cron / launchd / Actions)に縛られず、ユーザーが必要に応じて手動で呼び出します。

**手法ライブラリ**: Hermes Skills に類似した手続的記憶——「X をどうするか」自体を検索可能な skill にし、Cortex 海馬体が関連シーンで対応 skill を自動活性化します。これは Hermes 自己学習閉ループのローカル版ですが、RL 訓練は必要とせず、「規則閉ループ学習」で代替——規則自体は変わりませんが、「最近このルールが違反されている」とマークされます。

**Hermes とのギャップ**(正直に言うと): Hermes は独立 Python runtime 上で動き、47 ネイティブツール + 19 メッセージプラットフォーム + RL 訓練閉ループを持ちます。Life OS は Claude Code / Gemini CLI / Codex CLI 内で動き、ホストプラットフォームのツールしか使えません。**ギャップは存在しますが、埋められなくても問題ありません**—— Life OS のアーキテクチャ優位は「ツール数」ではなく:
- 23 agent 相互牽制(Hermes は単一 agent + delegate、構造化 veto なし)
- 三層長期記憶(SOUL + Wiki + Strategic Map、Hermes はフラットな MEMORY.md)
- Cortex 認知前置(Hermes に対応機構なし)
- DREAM 10 種類の自動パターン検出(Hermes は「nudge self to persist knowledge」のみ)

Hermes にはこれらがありません。方向は**Hermes の実行モードを吸収し、Life OS の牽制と記憶アーキテクチャを捨てない**。両者は最終的に相補的であり、代替ではありません。

## 4 層アーキテクチャ(4-layer architecture)

上記の三つのことをエンジニアリングに落とし込むと、この 4 層になります:

```
┌─────────────────────────────────────────┐
│  Layer 4: Python ツール層                │  バッチ / バックグラウンド / メンテ
│  (手動または定時実行、クラウド非依存)     │  ステータス: v1.7 GA
├─────────────────────────────────────────┤
│  Layer 3: Shell Hook 層                  │  HARD RULE 強制
│  (code-level で LLM が怠けない約束)      │  ステータス: v1.7 GA
├─────────────────────────────────────────┤
│  Layer 2: Skill 層                       │  23 agent 意思決定エンジン
│  └── Cortex 前置認知層(v1.7 GA)        │  ステータス: エンジン安定
│                                          │        Cortex は v1.7 で実装済み
├─────────────────────────────────────────┤
│  Layer 1: データ層                        │  markdown ファイル
│  SOUL / Wiki / Strategic Map /          │  ステータス: 安定
│  user-patterns / projects / _meta        │
└─────────────────────────────────────────┘
```

**各層の職責と対応関係**:

| 層 | 職責 | 対応する 3 方向 | 現在ステータス |
|---|------|-------------|---------|
| Layer 1 · データ層 | 全長期記憶の物理ストレージ、純 markdown、三バックエンド同期 | **セカンドブレイン** | 安定 |
| Layer 2 · Skill 層 | 23 agent オーケストレーション + Cortex 認知前置 + 9 テーマ | **意思決定エンジン** + **認知実行 · Cortex** | エンジン安定; Cortex は Layer 2 内部に前置モジュールを追加(独立 Layer 5 ではない)、v1.7 でクロスセッション記憶、概念グラフ、eval-history 自己フィードバックを追加 |
| Layer 3 · Shell Hook 層 | trigger 語強制、出力検証、セキュリティスキャン、git hook | **認知実行 · Hermes 実行力**(強制制約) | v1.7 GA |
| Layer 4 · Python ツール層 | バッチインデックス、自動化メンテナンス、self-review journal、定時タスク | **認知実行 · Hermes 実行力**(能動実行) | v1.7 GA |

**重要な明確化**:

- Cortex は Layer 5 **ではありません**。Cortex は Layer 2 の内部アップグレードです—— ROUTER の前に認知前置パイプライン(海馬体 + GWT + ナレーター層)を追加。外から見れば、依然として一つの Skill 層です。Skill 層の入力が「ユーザー原始 message」から「認知注釈付き message」に変わり、23 agent のオーケストレーションフロー自体は不変です。
- Vercel / Cloudflare Workers / 任意のクラウドサービスは**必須ではありません**。Layer 4 の定時タスクは launchd ローカル実行、cron 実行、GitHub Actions 実行、Vercel Edge Function 実行、自分の VPS 実行——どれか選んでください。**デフォルトはゼロクラウド依存**。Life OS は特定の serverless プラットフォームに縛られません。
- 4 層は「独立 4 モジュール」ではなく、「一つのシステムの 4 層視点」です。一層を変えると隣接層に影響します。例えば Layer 3 の Privacy Filter regex が更新されると、Layer 2 の ARCHIVER が Layer 1 に何を書けるかに直接影響します。

**3 方向と 4 層の交差**:

- **セカンドブレイン**は主に Layer 1(データ)+ Layer 2 の読み書き操作(RETROSPECTIVE 読み、ARCHIVER 書き)に生きています。
- **意思決定エンジン**は Layer 2(23 agent オーケストレーション)+ Layer 1(SOUL / Wiki / Strategic Map 読み)に生きています。
- **認知実行インテリジェントエージェント**は Layer 2(Cortex 認知前置)+ Layer 3(Shell Hook 強制)+ Layer 4(Python 能動実行)をまたいでいます。

だから「インテリジェントエージェント層」は特定の一層ではなく、三層にまたがる能力セットです。

## 一日の完全ワークフロー(実例)

朝 9:00、Mac の前。シーン: 昨夜携帯の Notion inbox に「Japan permanent residency 要件を調査」と記入。

**9:00 · Claude Code を開き「上朝」と言う**

ROUTER が trigger 語 `上朝` を識別 → 三省六部テーマを auto-load → 即座に RETROSPECTIVE subagent を Launch。

*(Layer 2 の Skill 層が実行中)*

**9:00:05 · Cortex Pre-Router が認知注釈を注入**(v1.7 GA で実装済み)

Pre-Router 認知前置層が並列実行:
- 海馬体拡散活性化: "Japan permanent residency" concept → "visa status"、"tax residency"、"family planning"、"10-year plan" の 4 本の強エッジに関連
- GWT マルチモジュールシグナル競争: 注意レーダーが「三ヶ月で 2 度目のビザ言及」を検出、予測エンジンが「前回の類似意思決定で準備時間を過小評価」をマーク、社会脳モジュールが「この意思決定は配偶者に関係、SOUL 社会次元チェックが必要」を提示
- ナレーター層がこれらの底層シグナルを「🏛️ 門下省警戒: この議題は三ヶ月前の議題と関連度 78%、重複意思決定パターンに注意」とパッケージ

ROUTER が受け取るのは生のメッセージではなく、認知注釈付き入力です。

*(Layer 2 内部の Cortex 前置層が実行中、下は Layer 1 の concept ファイル読み取り)*

**9:00:15 · RETROSPECTIVE 18 ステップ並列**

RETROSPECTIVE が Notion から昨夜の inbox エントリを取得、SOUL / user-patterns / Strategic Map を読み、バージョンチェック、巡察、朝報を編集。

朝報には固定で「DREAM Auto-Triggers」ブロックがあります——昨夜 DREAM がクロスセッションパターンを検出していれば、ここに現れます。

*(Layer 2 Skill + Layer 1 データ読み + Layer 4 Notion 取得)*

**9:01 · 意思決定エンジン実行**

「これは真剣に審議する価値がある」と言う → ROUTER が意思決定レベルと判断 → Draft-Review-Execute に入る:
- PLANNER が 5 次元に分解(法的要件、税務影響、タイムライン、家族への影響、戦略関連)

## 他の AI ツールとの比較

**vs ChatGPT / Claude.ai**: 日常会話、コード、翻訳、問答—— ChatGPT や他の汎用 AI を使い続けてください。Life OS は「審議に値する」ことを行うときだけ機能し、それ以外は ROUTER が直接返答します。Life OS の独自性は「より賢い会話」ではなく、**長期記憶 + 23 agent 相互牽制 + 認知前置**(認知前置は v1.7 GA で実装済み)です。

**vs Obsidian / Notion**: それらは知識管理では Life OS より優れています。Life OS の wiki はエンジンが自動抽出したもので、手打ちではありません。**相補的であり代替ではありません**—— Obsidian で Life OS が書いた markdown を見ると体験が最高です。実際、Life OS の iCloud セカンドブレインフォルダーは Obsidian vault レイアウトで、すべての `[[wiki-link]]` 構文が直接ジャンプできます。

**vs Todoist / Things**: それらはタスク追跡で Life OS よりはるかに優れています。Life OS の tasks は意思決定プロセスの副産物です——意思決定のたびに action items が生成され、自動的に `projects/*/tasks.md` に書き込まれます。手動で todo を追加する場所ではありません。純粋な GTD 体験が欲しいなら Todoist、「各タスクの背後に審議されたコンテキストがある」が欲しいなら Life OS。

**vs AI agent フレームワーク(LangGraph / AutoGen / CrewAI)**: それらは開発者が agent システムを構築するためのツール。Life OS は**既に構築済みの、個人生活の意思決定に向けた**具体的インスタンスです。一つのコマンドで使え、コードを書く必要はありません。

**vs Hermes Agent**: 最も近い参考対象ですが、**アーキテクチャは直交**しています。

| 次元 | Hermes | Life OS |
|------|--------|---------|
| 位置付け | 実行できる汎用インテリジェントエージェント | 思考 + 記憶 + 実行ができる個人認知システム |
| 実行力 | 極強(47 ネイティブツール + 19 プラットフォーム + RL 訓練) | 中(ホストツール + MCP + v1.7 の hook/Python 層) |
| 認知深度 | 弱(単一 agent + MEMORY.md フラット記憶) | 強(23 agent 相互牽制 + 三層長期記憶 + Cortex クロスセッション記憶 / 概念グラフ / 手法ライブラリ / eval-history 自己フィードバック) |
| 学習閉ループ | RL 訓練 + skills 自己学習 | DREAM 反省 + ADVISOR 行動分析 + user-patterns 蓄積 |
| 実行形態 | 独立 Python runtime、VPS デプロイ | Claude Code / Gemini / Codex に組み込まれた skill |
| 適合シーン | 実行集約(スクリプト実行、資料検索、フロー実行) | 意思決定集約(トレードオフ、判断、計画) |
| インストール複雑度 | Python 環境 + API keys + VPS 設定必要 | 一行のコマンド `/install-skill` または `npx skills add` |

**Life OS が Hermes から盗める 5 つの教訓**(`devdocs/research/2026-04-19-hermes-analysis.md` に調査記録済み、v1.7 で全て実装):
1. Shell hook 具体化(v1.7 Layer 3)
2. 構造化セッションインデックス(v1.7 Layer 4 の `session-index.py`)
3. 方法論をプラグ可能 skill として(v1.7 Cortex 手法ライブラリで実装済み)
4. 具体的セーフティパターン(v1.7 Privacy Filter を regex リスト化)
5. 自己評価閉ループ(v1.7 月次 self-review journal)

Hermes は実行力が強く認知が弱い; Life OS は認知が強く、実行力は v1.7 GA で補完済み。方向は**Hermes の実行モードを Layer 3 + Layer 4 に吸収し**、Life OS の 23 agent 相互牽制と三層記憶を保持することです。両者はゼロサム競争ではなく、相補的です。

## サポートしないもの

**単一コンテキストのチャットインターフェース**: ChatGPT、Gemini Web、Claude.ai のウェブ版はダメです。Life OS には情報隔離付きの 16 個の真に独立した subagent が必要で、一つのチャットボックスでは実現できません。すべての「独立審査」を無理に詰め込むと同じコンテキスト内のロールプレイになり、REVIEWER が PLANNER の思考プロセスを見え、veto は形式的なものになってしまいます。本物の subagent をサポートするプラットフォームで実行する必要があります: Claude Code / Gemini CLI / OpenAI Codex CLI。

**簡単な翻訳 / 問答 / 単一ステップタスク**: 「この日本語を翻訳して」「Python で list comprehension をどう書く」「今日の東京の天気」—— ROUTER が直接返答し、意思決定プロセスを起動しません。「上朝」と強く言っても無駄で、ROUTER はこれが意思決定ではないと識別します。

**真剣に聞くつもりがないこと**: 既にあることをしようと決めていて、AI の裏付けが欲しいだけなら、Life OS は不快に感じさせます。REVIEWER が veto し、ADVISOR が「先週も類似意思決定をしたが実行していない」と指摘し、AUDITOR があなたの強引な override をマークします。これは確認バイアスを**訂正する**ツールであり、確認バイアスのツールではありません。「無条件支持」が欲しいなら他の AI を探してください。

**複数人協働 / チーム意思決定**: SOUL、user-patterns、strategic map はすべて一人を中心に構築されています。二人で一つの second-brain を共有することは現時点で設計されておらず、衝突します。

**金融、医療、法律の専門アドバイス**: システムは「この投資をすべきか」を考えるのを助けられますが、免許を持つ顧問ではありません。現金、健康、法的紛争に関わるときは、Code of Conduct 第 4 条—— **まず専門家を探してください**。

## なぜこれほど複雑なのか

23 agents、9 themes、4 層アーキテクチャ、数十の HARD RULEs（host ごとに集計、詳細は `references/hard-rules-index.md`）、Cortex 前置、v1.7 の Hermes 級実行——見るたびに過剰エンジニアリングではと思います。

答えは: **コア原理は 3 つだけ**です。

1. **どの声も審査を受けずに着地することはできない**(1400 年前の三省六部原理、Roman Republic、Westminster、近代内閣——本当に大きな意思決定をするすべての場所は分権)
2. **AI の判断力がどれほど強くても、ユーザーの意思決定を代替することはできない——ユーザーが自分を見通すのを助けるだけ**(ADVISOR はあなたを審査、意思決定を審査しない; REVIEWER は情緒監査、結論は出さない)
3. **システムはあなたが誰かを学べなければならない、さもなくば毎回ゼロから審査することになる**(SOUL + user-patterns + DREAM が自動沈殿; Cortex は中間過程でも記憶を活用可能、入退場だけではない)

すべての複雑度はこれら 3 原理の具体的実装:

- **23 agents** は原理 1 の分権具体化(PLANNER は execute 不可、REVIEWER は veto のみ rewrite 不可、六部は相互審査不可)
- **9 themes** は原理 1 の「異文化表現」——異なる文化で「分権相互牽制」に異なる語彙と美学がある(三省六部 vs 霞が関 vs C-Suite)
- **3 方向**(セカンドブレイン / 意思決定エンジン / 認知実行)は原理 1-3 を実行可能なサブシステムに分解
- **4 層アーキテクチャ**は 3 方向をエンジニアリングに落とす階層視点
- **Cortex + Hermes 級実行**は原理 3 の v1.7 GA での実装拡張——システムは「入退場のみで記憶」ではなく中間過程でも記憶を使える、「考えられる」だけでなく「能動的に行動できる」。これは原理 3 の「静的ストレージ」から「動的活性化」への延伸
- **数十の HARD RULEs（host ごとに集計、詳細は `references/hard-rules-index.md`）** はこれら原理の具体操作上の硬い境界(intent clarification はスキップ不可、SOUL は引用必須で無視不可、Notion sync は silent fail 不可、Adjourn は途中退出不可、Privacy Filter は「人工審査」不可……)

複雑度はこれら 3 コア原理の周りに**外向きに**育ったもので、**無から**積み上げたものではありません。システムを理解するのに 33 規則を覚える必要はなく、これら 3 原理を覚えれば、規則は原理から自然に導出されます。

原理駆動であるからこそ、Life OS の進化経路は**外向き拡張であり、作り直しではありません**—— v1.6.2a から v1.7 の Cortex は「システム書き換え」ではなく、「原理 3 上により詳細な実装を追加」。v1.6.2a から v1.7 の Hermes 級実行も「目標変更」ではなく、「原理 1-2 の規則を code-level の制約で強制」。アーキテクチャは拡大し、内核は不変です。
