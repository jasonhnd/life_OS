# トークン消費量の内訳

Life OSのトークン消費量は3つの変数に依存します：**モード**（Lite/Pro）、**シナリオの複雑さ**（何部が起動されるか）、**プロセスイベント**（封駁/政事堂が発動するか）。このドキュメントでは各ケースの消費量を分解します。

> Claude Max/Proサブスクリプションの場合、トークン単位の課金はありません。以下の分析はAPIユーザーおよびコスト意識の高いユーザー向けです。

---

## 基本概念

### 役職ごとのトークン構成

各役職の呼出 = **入力トークン**（システム命令 + 受信コンテキスト）+ **出力トークン**（役職の出力）

| 役職 | システム命令 | 典型的な入力コンテキスト | 典型的な出力 | 1回あたりの合計 |
|------|--------------------|-----------------------|----------------|---------------|
| 丞相 | 約500 | ユーザーメッセージ 約200 | 上奏 約150 / 直接対応 約300 | 約700-1,000 |
| 中書省 | 約400 | 題目 + 背景 + ユーザーメッセージ 約500 | 立案 約800 | 約1,700 |
| 門下省 | 約450 | 立案/報告 約800-2,000 | 審査結果 約600 | 約1,850-3,050 |
| 尚書省 | 約350 | 立案 約800 | 発令 約600 | 約1,750 |
| 六部（各部）| 約350 | 指示 + 背景 約500 | 評価報告 約800 | 約1,650 |
| 御史台 | 約350 | 完全なプロセス記録 約3,000 | 業績審査 約300 | 約3,650 |
| 諫官 | 約300 | 奏折 + ユーザーメッセージ 約1,500 | 諫言 約300 | 約2,100 |
| 早朝官 | 約350 | Notionデータ 約1,000 | ブリーフィング 約600 | 約1,950 |
| 翰林院 | 約250 | ユーザーメッセージ 約300 | 会話返答 約500 | 約1,050 |

> 上記は推定値です。実際の消費量はユーザー入力の長さとモデル出力の詳細度によって変動します。

### Lite vs Proのコスト差

| | Lite | Pro |
|--|------|-----|
| 呼出方法 | 単一コンテキスト、全役職の出力が蓄積 | 各役職が独立したAPI呼出 |
| 入力の繰り返し | 後の役職の入力 = 前の全役職の蓄積出力（増大） | 各役職は必要な入力のみ受け取る（固定サイズ） |
| 総消費量 | 低い（共有コンテキスト） | 高い（複数の独立呼出、ただし蓄積なし） |
| 典型的な比率 | 1x | 2-2.5x |

## 利用シナリオ別の内訳

### シナリオA：丞相が直接対応

```
ユーザー → 丞相 → 返答
```

最も軽い利用パターン。丞相が簡単と判断し直接対応。

| | Lite | Pro |
|--|------|-----|
| 呼出回数 | 1（メインコンテキスト） | 1（chengxiangエージェント） |
| トークン | 約700-1,000 | 約700-1,000 |
| コスト | 約$0.01-0.02 | 約$0.01-0.02 |

適用：雑談、翻訳、調べもの、記録、単一ステップのタスク。

---

### シナリオB：簡略プロセス（3部）

```
丞相 → 中書省 → 門下省 → 尚書省 → 3部 → 門下省最終審査 → 奏折 → 御史台 → 諫官
```

典型的なシナリオ：大きな買い物（戸部 + 兵部 + 刑部）

| ステップ | Lite | Pro |
|------|------|-----|
| 丞相 | 約1,000 | 約1,000 |
| 中書省 | 約1,700（蓄積） | 約1,700 |
| 門下省立案審査 | 約2,500（蓄積） | 約1,850 |
| 尚書省 | 約3,000（蓄積） | 約1,750 |
| 3部 | 約5,000（蓄積） | 約4,950（3x1,650、並列） |
| 門下省最終審査 | 約4,500（蓄積） | 約2,450 |
| 丞相奏折総括 | — | 約2,000 |
| 御史台 | 約4,000（蓄積） | 約3,650 |
| 諫官 | 約3,500（蓄積） | 約2,100 |
| **合計** | **約10,000-12,000** | **約21,000-23,000** |
| **コスト** | **約$0.20-0.25** | **約$0.50-0.60** |

---

### シナリオC：標準プロセス（4部）

```
丞相 → 中書省 → 門下省 → 尚書省 → 4部 → 門下省最終審査 → 奏折 → 御史台 → 諫官
```

典型的なシナリオ：投資判断（戸部 + 兵部 + 刑部 + 吏部）、健康管理（工部 + 兵部 + 戸部 + 刑部）

| | Lite | Pro |
|--|------|-----|
| 合計 | 約12,000-15,000 | 約25,000-28,000 |
| コスト | 約$0.25-0.35 | 約$0.60-0.75 |

---

### シナリオD：完全プロセス（6部）

```
丞相 → 中書省 → 門下省 → 尚書省 → 6部 → 門下省最終審査 → 奏折 → 御史台 → 諫官
```

典型的なシナリオ：キャリア転換、転居、年間目標、起業判断

| | Lite | Pro |
|--|------|-----|
| 合計 | 約15,000-18,000 | 約35,000-40,000 |
| コスト | 約$0.30-0.50 | 約$0.80-1.20 |

---

### シナリオE：完全プロセス + 封駁

```
丞相 → 中書省 → 門下省（封駁）→ 中書省（修正）→ 門下省（准奏）→ 尚書省 → 6部 → ...
```

封駁により中書省 + 門下省のサイクルが1回追加される。

| | Lite | Pro |
|--|------|-----|
| 封駁1回あたりの追加消費 | +3,000-4,000 | +5,000-6,000 |
| 合計（全6部 + 封駁1回） | 約18,000-22,000 | 約40,000-46,000 |
| コスト | 約$0.40-0.55 | 約$1.00-1.40 |

封駁は最大2回。封駁2回の場合：

| | Lite | Pro |
|--|------|-----|
| 合計 | 約21,000-26,000 | 約45,000-52,000 |
| コスト | 約$0.50-0.65 | 約$1.20-1.60 |

---

### シナリオF：完全プロセス + 封駁 + 政事堂

```
... → 6部 → 門下省最終審査（矛盾を発見）→ 政事堂（3ラウンドの議論）→ 奏折 → 御史台 → 諫官
```

政事堂は部の結論が矛盾する場合に発動される。関連する部が3ラウンドの議論を行う。3部が参加すると仮定。

| | Lite | Pro |
|--|------|-----|
| 政事堂の追加消費 | +3,000-5,000 | +8,000-10,000 |
| 合計（全6部 + 封駁 + 政事堂） | 約24,000-31,000 | 約48,000-62,000 |
| コスト | 約$0.55-0.75 | 約$1.50-2.00 |

これは1回のプロセスの理論上の最大消費量です。

---

### シナリオG：早朝レビュー

```
早朝官（独立プロセス、三省六部を経由しない）
```

| | Lite | Pro |
|--|------|-----|
| 合計 | 約1,500-2,500 | 約1,500-2,500 |
| コスト | 約$0.03-0.05 | 約$0.03-0.05 |

早朝官は1回の呼出。LiteとProのコストは同じ。Notionに接続している場合、データの読み取りにより入力トークンが増加。

---

### シナリオH：翰林院の深い対話

```
丞相（判断）→ 翰林院（マルチターン対話）
```

翰林院はマルチターン対話で、1ターンあたり約1,000-1,500トークン。

| | 3ターン | 5ターン | 10ターン |
|--|---------|---------|----------|
| 合計 | 約4,000-5,000 | 約7,000-9,000 | 約15,000-20,000 |
| コスト | 約$0.08-0.12 | 約$0.15-0.22 | 約$0.35-0.50 |

## トークン節約戦略

| 戦略 | 効果 | 備考 |
|----------|--------|-------|
| **簡単な案件で朝議を開かない** | 90%以上節約 | 丞相が直接対応、約1kトークン |
| **クイックモード** | 約1k節約 | 丞相トリアージをスキップし、直接中書省へ |
| **必要に応じて部を起動** | 20-50%節約 | 3部での大きな買い物 vs 全6部は2倍の差 |
| **日常レビューは早朝官を使用** | 80%以上節約 | 1回の呼出 約2k、完全プロセスなし |
| **Liteモードを使用** | 50-60%節約 | 情報隔離と引き換えにコスト削減 |
| **翰林院のターン数を制限** | 場合による | 追加ターンごとに+1-2k |

### 月間コスト見積もり

典型的なアクティブユーザーの月間利用パターンを想定：

| 利用タイプ | 月間頻度 | 1回あたり（Pro） | 月間合計（Pro） |
|------------|----------------|-------------------|---------------|
| 丞相が直接対応 | 30回 | 約1k | 約30k |

| 完全プロセス（6部） | 2回 | 約40k | 約80k |
| 早朝レビュー | 4回 | 約2k | 約8k |
| 翰林院 | 1回 | 約8k | 約8k |
| **月間合計** | | | **約226kトークン** |
| **月間コスト（Pro）** | | | **約$5-7** |
| **月間コスト（Lite）** | | | **約$2-3** |

> Claude Maxサブスクリプション（$100/月）には十分な利用枠が含まれています。上記のシナリオは十分に枠内に収まります。Proサブスクリプション（$20/月）もほとんどの日常利用をカバーします。

## API料金参考

Claude Sonnet 4.5 API料金を例に（2026年4月）：

| | 料金 |
|--|-------|
| 入力 | $3 / 100万トークン |
| 出力 | $15 / 100万トークン |

ProモードはOpus 4.5を使用：

| | 料金 |
|--|-------|
| 入力 | $15 / 100万トークン |
| 出力 | $75 / 100万トークン |

> 実際のコストは入力/出力の比率に依存します。Life OSは比較的出力が多く（部の報告 + 奏折）、出力トークンが総コストの60-70%を占めます。
