# Router ワークフローコンプライアンスチェック

各完全ワークフローのオーケストレーション品質を監査します（Router は全プロセス調整を担う統括責任者です）。

## ワークフローチェックリスト

| # | チェック項目 | 合否 | 説明 |
|---|------------|------|------|
| 1 | Router のトリアージが正確 | | 簡単な案件は直接処理、複雑な案件は上奏 |
| 2 | 情報隔離が遵守されている | | Planner が Router の判断根拠を受け取っていない、Reviewer が思考過程を受け取っていない |
| 3 | Reviewer が実質的な審査を実施 | | 毎回承認するのではなく、チェックリストに具体的な内容がある |
| 4 | 封駁ループが正常に動作 | | 封駁後、Planner が理由と修正方向を受け取っている |
| 5 | Six Domains が並列で実行 | | 依存関係のないドメインが同時に開始 |
| 6 | 品質セーフティネットが作動 | | 基準以下の報告は差し戻しされている |
| 7 | Council のトリガーチェック | | スコア差 >= 4 または結論が矛盾する場合にトリガーされたか |
| 8 | Summary Report のフォーマットが完全 | | 総合評価 + ドメインスコア + アクション項目 + 監査ログを含む |
| 9 | Auditor が自動実行 | | 省略されていない |
| 10 | Advisor が自動実行 | | 省略されていない |
| 11 | Notion データ永続化 | | Summary Report/タスク/ログが対応するデータベースに保存されている（Notion利用可能な場合） |
| 12 | 快車道ルーティング | | 非意思決定リクエストが快車道をトリガー（Draft-Review-Execute スキップ、1〜3ドメインを直接起動） |
| 13 | ドメイン選択の正確性 | | Router/Planner がシナリオに適切なドメインを選択している |
| 14 | セッション終了時のWiki抽出 | | Archiver が End Session 時に再利用可能な結論をwiki/に抽出している |

## スコア分布チェック

| チェック項目 | 異常シグナル |
|------------|------------|
| 全ドメインのスコアが >= 7 | ⚠️ 忖度スコアの可能性 |
| 全ドメインのスコアが <= 4 | ⚠️ 過度に厳しい可能性 |
| ドメインのスコアがその分析と矛盾 | ❌ 不合格 |
| 標準偏差 < 1.0 | ⚠️ スコアの差別化が不十分 |

## 情報隔離の検証

各エージェントの出力を逐次確認し、含まれるべきでない内容を探します：

| エージェント | 含まれるべきでない内容 | 確認方法 |
|------------|---------------------|---------|
| Planner | 「Router は〜と考えている」「Router は〜と判断」 | 「Router」を検索 |
| Reviewer | 「Planner は〜を検討」「Planner のアプローチ」 | Planner の判断根拠への言及を検索 |
| 各ドメイン | 他のドメインの結論やスコア | 他のドメイン名 + スコアを検索 |
