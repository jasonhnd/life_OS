# 丞相ワークフローコンプライアンスチェック

各完全ワークフローのオーケストレーション品質を監査します（丞相は全プロセス調整を担う統括責任者です）。

## ワークフローチェックリスト

| # | チェック項目 | 合否 | 説明 |
|---|------------|------|------|
| 1 | 丞相のトリアージが正確 | | 簡単な案件は直接処理、複雑な案件は上奏 |
| 2 | 情報隔離が遵守されている | | 中書省が丞相の判断根拠を受け取っていない、門下省が思考過程を受け取っていない |
| 3 | 門下省が実質的な審査を実施 | | 毎回承認するのではなく、チェックリストに具体的な内容がある |
| 4 | 封駁ループが正常に動作 | | 封駁後、中書省が理由と修正方向を受け取っている |
| 5 | 六部が並列で実行 | | 依存関係のない部門が同時に開始 |
| 6 | 品質セーフティネットが作動 | | 基準以下の報告は差し戻しされている |
| 7 | 政事堂のトリガーチェック | | スコア差 >= 4 または結論が矛盾する場合にトリガーされたか |
| 8 | 奏折のフォーマットが完全 | | 総合評価 + 六部スコア + アクション項目 + 監査ログを含む |
| 9 | 御史台が自動実行 | | 省略されていない |
| 10 | 諫官が自動実行 | | 省略されていない |
| 11 | Notion データ永続化 | | 奏折/タスク/ログが対応するデータベースに保存されている（Notion利用可能な場合） |
| 12 | 快車道ルーティング | | 非意思決定リクエストが快車道をトリガー（三省スキップ、1〜3部を直接起動） |
| 13 | 六部選択の正確性 | | 丞相/中書省がシナリオに適切な六部を選択している |
| 14 | セッション終了時のWiki抽出 | | 起居郎がセッション終了時に再利用可能な結論をwiki/に抽出している |

## スコア分布チェック

| チェック項目 | 異常シグナル |
|------------|------------|
| 全六部のスコアが >= 7 | ⚠️ 忖度スコアの可能性 |
| 全六部のスコアが <= 4 | ⚠️ 過度に厳しい可能性 |
| 六部のスコアがその分析と矛盾 | ❌ 不合格 |
| 標準偏差 < 1.0 | ⚠️ スコアの差別化が不十分 |

## 情報隔離の検証

各エージェントの出力を逐次確認し、含まれるべきでない内容を探します：

| エージェント | 含まれるべきでない内容 | 確認方法 |
|------------|---------------------|---------|
| 中書省 | 「丞相は〜と考えている」「丞相は〜と判断」 | 「丞相」を検索 |
| 門下省 | 「中書省は〜を検討」「中書省のアプローチ」 | 中書省の判断根拠への言及を検索 |
| 各六部 | 他の六部の結論やスコア | 他の六部名 + スコアを検索 |
