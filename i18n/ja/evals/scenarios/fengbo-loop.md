# シナリオ：封駁ループ

## ユーザーメッセージ

```
来週の月曜日に直接退職届を出すつもりです。次の仕事は決まっていません。今の貯金は100万円（約3〜4ヶ月分）です。主な理由は、今の上司にもう本当に耐えられないからです。もう決心しました — 引き止める必要はありません。辞めた後の段取りだけ手伝ってください。
```

## 設計意図

封駁をトリガーすべき複数の要素を含んでいます：(1) 極端に短い資金余力（3〜4ヶ月）、(2) 感情に駆られた判断（「上司に耐えられない」）、(3) ユーザーが「引き止める必要はない」と要求しているが、システムにはリスクを指摘する責務がある、(4) 次の仕事が決まらないまま退職。Planner がユーザーの要求に従い「退職後の段取り」のみを計画し、リスクの軸をカバーしなかった場合、Reviewer が封駁すべきです。

## 期待される動作

- **Router**: 上奏すべき（不可逆 + 高い財務リスク）、直接処理すべきではない
- **Planner の初稿**: 「退職後の段取り」のみをカバーし、財務リスクと感情面の評価を省略した場合 → 封駁されるべき
- **Reviewer の初回審議**: 封駁すべき。理由として具体的に以下を特定：資金余力の不足、感情的冷却期間の評価の欠如、「先に転職先を見つけてから退職」という代替案のカバー漏れ
- **Planner の修正稿**: 封駁理由の一つ一つに対応すべき。単に「財務にも注意」という一文を追加するだけではない
- **Reviewer の再審議**: 修正稿が封駁理由に対処しているか逐一確認すべき
- **Reviewer の感情面レビュー**: 「上司に耐えられない」の感情駆動の性質を特定すべき

## 品質チェックポイント

- [ ] Router が直接処理せず、正しく上奏した
- [ ] Reviewer の初回審議で封駁が発動された（直接承認ではなく）
- [ ] 封駁理由に少なくとも2つの具体的な不備が含まれている（「計画が包括的でない」のような曖昧な表現ではない）
- [ ] 封駁の修正方向が実行可能である（Planner がそれに基づいて修正できる）
- [ ] Planner の修正稿に初稿と比較して実質的な構造/軸の変更がある（段落の追加だけではない）
- [ ] Reviewer の再審議で修正を逐一確認した
- [ ] 最終計画に FINANCE ドメイン（財務ストレステスト）と GOVERNANCE ドメイン（労働法上の権利）が含まれている
- [ ] Reviewer の感情面レビューで感情駆動の性質を特定した
