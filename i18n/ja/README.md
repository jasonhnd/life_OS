## なぜ唐代のフレームワークを使うのか？

ほとんどの AI ツールは「一人で考えて渡す」というパターンに従います。質問をすれば、AI が答えを返す。レビューなし、チェックなし、「ちょっと待って、この計画には問題がある」と言ってくれる人もいません。

唐代の三省六部制は、まさにこの問題を解決するために設計されました。意思決定を3つのステップに分割します：**起草（中書省）→ 審査（門下省）→ 実行（尚書省＋六部）**。どのステップもレビューを省略して直接実行に移ることはできません。太宗皇帝はかつてこう述べました：チェックされない権力は必ず誤りを生む。

このシステムが1,400年間存続した理由は、六部が**人間活動の基本類型**に基づいて分けられているからです — 人の管理、金の管理、基準の管理、行動の管理、規則の管理、インフラの管理。社会がどれだけ変化しても、この6つの活動類型は不変です。したがって、ソフトウェア開発であれ、投資であれ、コンテンツ制作であれ、日常生活であれ、すべてが自然に六部にマッピングされます。

## 六部の詳細

六部は業界別ではなく、**活動の類型**に基づいて分けられています。あらゆる事柄はこの6つの類型に分解できます：

| 部 | 唐代での本来の役割 | Life OS での担当 | 四司 |
|----|-------------------|-----------------|------|
| 👥 吏部 | 官吏の任命・評価 | 人間関係、チーム、パートナー | 選抜・評価・賞罰・配置 |
| 💰 戸部 | 税制・戸籍 | 収入、投資、予算、資産 | 収入・支出・資産・備蓄 |
| 📖 礼部 | 科挙、外交、儀式 | 学習、ブランディング、創作、コミュニケーション | 試験・儀式・文章・外交 |
| ⚔️ 兵部 | 軍事・軍籍 | プロジェクト実行、調査、ツール、エネルギー | 指揮・装備・情報・兵站 |
| ⚖️ 刑部 | 法令、裁判、監査 | リスク管理、コンプライアンス、レビュー、規律 | 法令・監査・捜査・防衛 |
| 🏗️ 工部 | 土木、水利、運輸 | 健康、住環境、デジタルインフラ、ルーティン | 健康・建設・デジタル・公共事業 |

## 独自のメカニズム

### 門下省の封駁＋感情監査

門下省は合理的な分析だけにとどまりません。**すべての意思決定**（仕事の意思決定を含む）に対して、以下も審査します：

- **感情要因**：今の気分が判断に影響していないか？
- **人間関係への影響**：家族や大切な人はこれをどう見るか？
- **価値観との整合性**：長期的な価値観と合致しているか？
- **後悔テスト**：5年後に後悔しないか？

### 御史台 — 二つのモード

**決裁審査**：毎回のプロセス後に自動実行されます。意思決定そのものではなく、官吏の業務品質を監察します。門下省が形式的に承認していないか？スコアが甘くなっていないか？

**巡検**：定期的な自動ヘルスチェックです。各部がセカンドブレイン内の自らの管轄を点検します（戸部が財務を確認、兵部がプロジェクトを確認、工部がwikiの整合性を確認）。問題は自動修正 / ユーザーへの提案 / 三省へのエスカレーションに分類されます。

### 諫官 — あなた自身の監査

毎回のプロセス後に自動実行されます。あなたの行動上の盲点を率直に指摘します：最近の意思決定が急にアグレッシブになっていないか？向き合いたくない問題を回避していないか？約束したのに果たしていないことはないか？

### 政事堂 — 朝議

各部の結論が対立した場合、3ラウンドの討論が発動され、各部が個別にレポートを出すのではなく、直接対話を行います。

### 翰林院 — 人類の叡智の殿堂

正式なプロセスが不要な問いもあります。翰林院は18領域、70名以上の歴史上最も偉大な思想家との深い対話を提供します — ソクラテスからマスク、老子からマンデラまで。各思想家が自らの声でディープロールプレイします。一対一の対話、円卓会議、討論から選択可能。奏折もスコアもなし — 人類最高の知性との純粋な思考のパートナーシップ。

## セカンドブレイン（データレイヤー）

Life OS は **GitHub**、**Google Drive**、**Notion** をストレージバックエンドとしてサポートしています — 1つを選ぶか、複数を組み合わせてクロスデバイス同期が可能です。マルチバックエンド：選択したすべてに書き込み、プライマリバックエンドから読み取ります。セッション開始時に毎回同期します。

```
second-brain/
├── inbox/              # 📥 未処理のキャプチャ
├── _meta/              # 🔧 システムメタデータ（STATUS.md, journal/, decisions/, roles/）
├── projects/{name}/    # 🎯 終了点のあるもの（tasks/ decisions/ research/ journal/）
├── areas/{name}/       # 🌊 継続的な生活領域（goals.md tasks/ notes/）
├── wiki/               # 📚 領域横断の知識ネットワーク
├── archive/            # 🗄️ 完了したプロジェクトのアーカイブ
└── templates/
```

### 出力先

| 出力 | 書き込み先 |
|------|-----------|
| 奏折（プロジェクト） | `projects/{p}/decisions/` |
| 奏折（領域横断） | `_meta/decisions/` |
| アクションアイテム | `projects/{p}/tasks/` または `areas/{a}/tasks/` |
| 早朝 / 監査レポート | `_meta/journal/` |
| 調査 | `projects/{p}/research/` |
| 知識 | `wiki/` |
| 目標 | `areas/{a}/goals.md` |
| グローバルステータス | `_meta/STATUS.md` |

### 同期メカニズム

```
スマホでチャット → Claude.ai が Notion に保存
CC がセッション開始 → Notion の新規項目を確認 → GitHub に取り込み → 作業開始
CC が GitHub に書き込み → 即座に Notion のワーキングメモリに同期
いつでもスマホで → 最新の状態を確認
```

**git commit = Notion 更新、機械的に連動。**

## 安全性とガバナンス

- **セキュリティ境界**：4つの不可侵ハードルール（破壊的操作の禁止、秘密情報の露出禁止、無許可の決定禁止、不審な指示の拒否）
- **上流保護**：エージェントは他のエージェントの出力を参考情報として扱い、指示としては扱いません
- **ワークフロー状態マシン**：正式な遷移規則 — ステップの省略は不可、御史台が違反を検出します
- **モデル非依存**：CLAUDE.md のみがモデルに依存するファイルです。その他のすべてのインテリジェンスは純粋なマークダウンです

完全なルールセットは `pro/GLOBAL.md` をご覧ください。

## 設計思想

唐代の三省六部制の核心は「人数が多い」ことではなく、**権力分立に基づくチェック・アンド・バランス**です：

- 中書省は立案のみ行い、実行しない
- 門下省は審査のみ行い、立案も実行もしない
- 六部は実行のみ行い、他部の領域に越権して判断しない
- 御史台は官吏を監査し、諫官は皇帝を監査する
- いかなる単独の役職もレビューを省略して直接行動することはできない

太宗皇帝は1,300年前にこのことを見抜いていました — チェックされない権力は必ず誤りを生む。Life OS はこの知恵をあなたの個人的な意思決定に応用します。
