# 🏛️ Life OS — 三省六部制 個人内閣システム

🌍 [English](../../README.md) | [中文](../zh/README.md) | [日本語](README.md) | [한국어](../ko/README.md) | [Español](../es/README.md)

> 1,400年間機能し続けた統治システムで、あなたの人生を管理しましょう。

Life OSはAIをあなたの私的な朝廷に変えます。丞相（総管家）＋三省（立案/審査/実行）＋六部（人/金/学習/行動/規則/インフラ）＋御史台（官吏を監査）＋諫官（あなたを監査）＋翰林院（戦略アドバイザー）の体制で、生活、仕事、学習、財務、健康、人間関係を包括的に管理します。

**これはロールプレイではありません。権力分立に基づくチェック・アンド・バランスです。** 門下省は基準に満たない提案を封駁する権限を持ちます。御史台はすべての「官吏」の業務品質を監視します。諫官はあなた自身の行動上の盲点を率直に指摘します。

## なぜ唐代のフレームワークを使うのか？

ほとんどのAIツールは「一人で考えて渡す」というパターンに従います。質問をすれば、AIが答えを返す。レビューなし、チェックなし、「ちょっと待って、この計画には問題がある」と言ってくれる人もいません。

唐代の三省六部制は、まさにこの問題を解決するために設計されました。意思決定を3つのステップに分割します：**起草（中書省）→ 審査（門下省）→ 実行（尚書省＋六部）**。どのステップもレビューを省略して直接実行に移ることはできません。太宗皇帝はかつてこう述べました：チェックされない権力は必ず誤りを生む。

このシステムが1,400年間存続した理由は、六部が**人間活動の基本類型**に基づいて分けられているからです。人の管理、金の管理、基準の管理、行動の管理、規則の管理、インフラの管理。社会がどれだけ変化しても、この6つの活動類型は不変です。したがって、ソフトウェア開発であれ、投資であれ、コンテンツ制作であれ、日常生活であれ、すべてが自然に六部にマッピングされます。

## 六部の詳細

六部は業界別ではなく、**活動の類型**に基づいて分けられています。あらゆる事柄はこの6つの類型に分解できます：

| 部 | 唐代での本来の役割 | Life OSでの担当 | 四司 |
|----|-------------------|----------------|------|
| 👥 吏部 | 官吏の任命・評価 | 人間関係、チーム、パートナー | 選抜・評価・賞罰・配置 |
| 💰 戸部 | 税制・戸籍 | 収入、投資、予算、資産 | 収入・支出・資産・備蓄 |
| 📖 礼部 | 科挙、外交、儀式 | 学習、ブランディング、創作、コミュニケーション | 試験・儀式・文章・外交 |
| ⚔️ 兵部 | 軍事・軍籍 | プロジェクト実行、調査、ツール、エネルギー | 指揮・装備・情報・兵站 |
| ⚖️ 刑部 | 法令、裁判、監査 | リスク管理、コンプライアンス、レビュー、規律 | 法令・監査・捜査・防衛 |
| 🏗️ 工部 | 土木、水利、運輸 | 健康、住環境、デジタルインフラ、ルーティン | 健康・建設・デジタル・公共事業 |

## 独自のメカニズム

### 門下省の封駁＋感情監査

門下省は合理的な分析だけにとどまりません。**すべての意思決定**（仕事の意思決定を含む）に対して、以下も審査します：

- **感情要因**：今の気分が判断に影響していないか？
- **人間関係への影響**：家族や大切な人はこれをどう見るか？
- **価値観との整合性**：長期的な価値観と合致しているか？
- **後悔テスト**：5年後に後悔しないか？

例えば、学習コストの高い技術スタックを選んだ場合、門下省はこう指摘します：「これは今後3ヶ月の週末が学習に費やされ、家族との時間に影響が出ることを意味します。」

### 御史台 — 官吏の監査

毎回のプロセス後に自動実行されます。案件そのものではなく、「官吏の業務品質」を監察します：
- 門下省が形式的に承認していないか（精査せずに通していないか）？
- 各部の報告に実質的な内容があるか（「問題なし」だけになっていないか）？
- スコアが甘くなっていないか（面子を立てて高評価を付けていないか）？

### 諫官 — あなた自身の監査

毎回のプロセス後に自動実行されます。あなたの行動上の盲点を率直に指摘します：
- 最近の意思決定が急にアグレッシブになっていないか？
- 向き合いたくない問題を回避していないか？
- 約束したのに果たしていないことはないか？

### 政事堂 — 朝議

各部の結論が対立した場合（戸部は「資金不足」、兵部は「段階的に実行可能」）、3ラウンドの討論が発動されます。各部が個別にレポートを出すのではなく、直接対話を行います。

### 翰林院 — 戦略対話

正式なプロセスが不要な問いもあります。例えば「最近迷っていて、人生の方向性がわからない」。翰林院はあなたの非公開の戦略アドバイザーです。奏折もスコアもなく、ただあなたと共に深く考えます。

## セカンドブレイン（データレイヤー）

Life OSは**GitHubのセカンドブレイン**を主要なデータストア（ハードドライブ）として、**Notion**を軽量なワーキングメモリ（モバイル同期）として使用します。3つの方法論を組み合わせています：GTDが行動を駆動し、PARAが構造を整理し、Zettelkastenが知識を成長させます。

```
second-brain/
├── inbox/              # GTDのエントリーポイント
├── projects/{project}/ # 期限付きの目標駆動型（tasks/ decisions/ notes/を含む）
├── areas/{area}/       # 維持すべき継続的な生活領域（goals.md tasks/を含む）
├── zettelkasten/       # 知識の成長（fleeting/ literature/ permanent/）
├── records/            # ジャーナル、会議、連絡先、財務、健康
├── gtd/                # waiting/ someday/ reviews/
└── archive/            # 完了したプロジェクト
```

### 三省六部の出力先

| 出力 | 書き込み先 |
|------|-----------|
| 奏折 | `projects/{p}/decisions/` |
| アクションアイテム | `projects/{p}/tasks/` |
| レビュー/監査レポート | `records/journal/` |
| 調査 | `zettelkasten/literature/` |
| 目標 | `areas/{a}/goals.md` |

### 同期メカニズム

```
スマホでチャット → Claude.aiがNotionに保存
CCがセッション開始 → Notionの新規項目を確認 → GitHubに取り込み → 作業開始
CCがGitHubに書き込み → 即座にNotionのワーキングメモリに同期
いつでもスマホで → 最新の状態を確認
```

**git commit = Notion更新、機械的に連動。**

六部はAIの分析フレームワーク（固定）であり、Areasはあなたの実際の生活領域（カスタマイズ可能）です。両者は独立しています。

**[セカンドブレインアーキテクチャの全文はこちら ->](docs/second-brain.md)**（ディレクトリ構成、GTD/PARA/Zettelkastenのフロー、Notionメモリのセットアップを含む）

## 設計思想

唐代の三省六部制の核心は「人数が多い」ことではなく、**権力分立に基づくチェック・アンド・バランス**です：

- 中書省は立案のみ行い、実行しない
- 門下省は審査のみ行い、立案も実行もしない
- 六部は実行のみ行い、他部の領域に越権して判断しない
- 御史台は官吏を監査し、諫官は皇帝を監査する
- いかなる単独の役職もレビューを省略して直接行動することはできない

この設計により、すべての意思決定が分解 → 審査 → 実行 → 再審査 → 監査を経ることが保証されます。「一人で考えて渡す」のではありません。

太宗皇帝は1,300年前にこのことを見抜いていました。チェックされない権力は必ず誤りを生む。Life OSはこの知恵をあなたの個人的な意思決定に応用します。
