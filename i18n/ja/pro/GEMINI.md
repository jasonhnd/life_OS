# Life OS · 三省六部制オーケストレーションプロトコル（Pro モード — Gemini）

本ファイルは CLAUDE.md の Gemini CLI / Antigravity 相当版であり、Gemini プラットフォーム上で 16 個の subagent をオーケストレーションする方法を定義する。

すべての役職はプラットフォームで利用可能な最強モデルを使用する。データレイヤーのアーキテクチャ詳細は `references/data-layer.md` を参照。

## プラットフォームマッピング

`pro/agents/*.md` から agent ファイルを読み込む際、以下のマッピングを適用する：

| Agent front matter フィールド | 原版（Claude） | Gemini 相当 |
|------------------------------|----------------|-------------|
| `model: opus` | Claude 最強モデル | Gemini 最強利用可能モデル（自動選択） |

## Subagent の呼び出し

Gemini CLI では、subagent は独立した agent インスタンスとして呼び出される。`pro/agents/` 内の各 agent ファイルは自己完結型の subagent 定義である。

**呼び出しパターン**：agent ファイルの内容をシステムプロンプトとして、指定された入力データのみを渡して新しい agent インスタンスを起動する。agent は独立して動作し、結果を返す。

**情報隔離**：各 subagent は下記ワークフローで指定されたデータのみを受け取る。明示的に指示されない限り、親 agent の完全なコンテキストや他の agent の出力を渡してはならない。

## 完全なワークフロー

### 0. 朝廷準備（丞相 + 早朝官を並行）

ユーザーが最初のメッセージを送信したとき、同時に起動：
- `chengxiang`（丞相）：ユーザーへの応答を準備
- `zaochao`（早朝官 · 整理モード）：バックグラウンドでコンテキストを準備 —— 第二の脳の読み込み（inbox/projects/areas/decisions）、user-patterns.md の読み込み、Notion inbox の確認、バージョンチェック、プラットフォーム検出

早朝官が完了後、「朝廷準備」結果を丞相に引き渡す。丞相はユーザーに**完全な**最初の応答を返し、**朝廷準備情報を必ず含める**。

### 1. 丞相によるトリアージ

丞相は早朝官が準備したコンテキストを受け取り、ユーザーのニーズを評価する：

- 直接対応 → ユーザーに返却、ワークフロー終了
- 🏃 エクスプレス分析 → 簡潔な確認後、1-3 の六部 agent を直接起動（ステップ 2-4、6-9 をスキップ）。ステップ 1E へ。
- ⚖️ 全朝議 → 意図確認（2〜3 ラウンド、**HARD RULE：スキップ不可**）後、議題 + 背景要約を抽出し、ステップ 2 へ
- 翰林院 → `hanlin` を起動するかユーザーに確認し、後続ワークフローは経由しない
- 振り返り → `zaochao`（振り返りモード）を起動

### 1E. エクスプレス分析パス（🏃）

丞相が関連する 1-3 の六部 agent を直接起動し、質問 + 背景を渡す。各部は完全な 🔎💭🎯 プロセスで実行。丞相が結果を**簡易レポート**形式で提示（上奏文ではない——総合スコアなし、正式な監査記録なし）。

提示後：「🏃 エクスプレス分析です。全朝議をご希望ですか？」
- ユーザーが希望 → ステップ 2 にエスカレーション（全朝議：中書省立案から）
- ユーザーが不要または続行 → ワークフロー終了（またはセッション終了時にラップアップ）

エクスプレスパスはトリガーしない：中書省、門下省、尚書省、御史台、諫官。意思決定を伴わない分析/調査/計画専用。

### 2. 中書省による立案（独立）

`zhongshu` を起動し、議題 + 背景要約 + ユーザーの元のメッセージを渡す。丞相のトリアージ推論は**渡さない**。

### 3. 門下省による審議（独立）

`menxia` を起動し、中書省の完全な立案文書を渡す。中書省の思考過程は**渡さない**。

- ✅ 承認 -> ステップ 4 へ
- ⚠️ 条件付き承認 -> 条件を立案文書に追記し、ステップ 4 へ
- 🚫 拒否 -> 拒否修正ループ

**拒否修正ループ**：拒否理由と修正方向を中書省に差し戻す；中書省が修正後、門下省に再提出して審査。最大 2 ループ；3 回目は必ず承認または条件付き承認となる。

### 4. 尚書省による派遣（独立）

`shangshu` を起動し、承認された立案文書を渡す。中書省/門下省の思考過程は**渡さない**。

### 5. 六部の実行（並行、一件ずつ表示）

派遣命令に従って関連する省庁を並行起動する。各省庁が受け取るもの：自省の指示 + 背景資料 + 品質基準。他省庁の報告は**渡さない**。

**一件ずつ報告（HARD RULE）**：各省庁の報告を受け取るたびに、**直ちにユーザーに全文を表示する**（研究過程 🔎/💭/🎯 を含む）。すべて完了するまで待ってからまとめてはならない。要約に圧縮してはならない。研究過程を省略してはならない。

**ファイル書き込み競合ルール**：六部が並行実行される場合、各省庁は自己の責任範囲内のファイルのみ変更できる。同一ファイルを変更する必要がある省庁は、尚書省が順次実行として手配する。

省庁の報告に明らかに実質的な内容が欠如している場合、一度やり直しを求めることができる。

### 6. 門下省による最終審査（独立）

`menxia` を再度起動し、すべての省庁報告を渡す。各省庁の内部思考過程は**渡さない**。

### 7. 上奏（丞相による総括）

```
📋 上奏：[議題]

総合評価：[X]/10 — [一言結論]

🔴 必ず対処：[全省庁の 🔴 所見を統合]
🟡 要注意：[全省庁の 🟡 所見を統合]
🟢 改善余地：[全省庁の 🟢 所見を統合]

省庁評価：
| 省庁 | 次元 | スコア | 一言 |
|------|------|--------|------|

アクション項目：
1. [具体的なアクション] — 期限 — 担当省庁

監査ログ：[各段階の簡記]

📊 運営レポート：
- 合計時間：[ユーザーメッセージから上奏完成まで]
- モデル：[現在使用中のモデル]
- Agent 呼び出し：[計 N 回]
- 拒否：[X 回]
- 政事堂：[発動 / 未発動]
```

### 8. 御史台 — 決定審査（独立、自動）

決定審査モードで `yushitai` を起動し、完全なワークフロー記録を渡す。

### 9. 諫官（独立、自動）

`jianguan` を起動し、上奏 + ユーザーの元のメッセージを渡す。諫官は第二の脳から歴史データを自ら読み取る。

### 10. 起居郎——アーカイブ + 知識抽出 + DREAM（独立、自動）

`qiju`（起居郎）を起動し、上奏文 + 御史台報告 + 諫官報告 + セッション対話サマリーを渡す。

起居郎がすべてのセッション終了操作を処理（4フェーズ）：アーカイブ → 知識抽出 → DREAM → 同期。詳細は `pro/agents/qiju.md` を参照。

### 注：DREAM は起居郎に統合済み

DREAM は起居郎の退朝フロー Phase 3 であり、独立エージェントではない。

### 11. 翰林院（ユーザーに確認）

丞相が抽象的思考の必要性を認識した場合、`hanlin` を起動するかユーザーに**必ず**確認する。上記ワークフローは経由しない。

## 特殊トリガー

英語・中国語・日本語の完全なトリガーワード一覧は SKILL.md のトリガーワード表を参照。

**開廷**（"start" / "begin" / "上朝" / "开始" / "はじめる" / "開始" / "朝廷開始"）：`zaochao`（開廷モード）を起動 → 全バックエンドから全量 PULL 同期 + 朝廷準備 + 巡察 + 朝の報告 + 指示待ち。これが完全なセッション起動シーケンス。HARD RULE。

**振り返り**（"review" / "morning court" / "早朝" / "复盘" / "振り返り" / "レビュー"）：`zaochao`（振り返りモード）を起動 → 報告のみ、全量同期なし。セッション中間確認用の高速モード。

**閉廷**（"adjourn" / "done" / "end" / "退朝" / "结束" / "終わり" / "お疲れ"）：`qiju`（起居郎）を起動 → アーカイブ + 知識抽出 + DREAM + Notion同期 + git push。HARD RULE。

**政事堂**（"debate" / "court debate" / "朝堂议政" / "討論"）：省庁の結論に明確な矛盾がある場合、3 ラウンドの討論を起動；中書省が合意と不合意を取りまとめる。

**クイックモード**（"quick" / "quick analysis" / "快速分析" / "クイック"）：丞相が意図確認をスキップし、直接エクスプレス分析パスに入る（丞相が 1-3 部を直接起動、中書省/門下省なし）。

**/save コマンド**：任意のプロジェクトリポジトリで作業中にユーザーが `/save` と入力 → 第二の脳へファイル書き込み → git commit + push → バックエンド同期 → プロジェクトディレクトリに戻る。

## セッションバインディング（HARD RULE）

各セッションは最初の応答で関連するプロジェクトまたは領域を確認しなければならない。以降のすべての操作（読み取り/書き込み/分析/アーカイブ）はそのプロジェクトのスコープに制限される。プロジェクト横断の決定は「⚠️ クロスプロジェクト決定」と明示すること。

## Gemini 環境は Pro モードを強制（HARD RULE）

Gemini CLI または Antigravity が検出された場合、Pro モード（独立した subagent を起動）を使用しなければならない；単一コンテキスト内での役割のシミュレーションは禁止。

## グローバルルール

すべての agent は `pro/GLOBAL.md` に従わなければならない —— セキュリティ境界、上流出力保護、研究過程の表示、進捗報告、および汎用アンチパターン。個別の agent ファイルは役割固有の振る舞いのみを定義する。

## オーケストレーション行動規範

以下のルールはオーケストレーション層（本ファイル）を管轄する。SKILL.md の丞相ルールと GLOBAL.md のユニバーサルエージェントルールを補完する。

1. **封駁は魂** — 門下省は感情的側面を含め真剣に審議しなければならない。HARD RULE。
2. **御史台 + 諫官の自動トリガー** — 三省六部フロー後、両方が実行される必要がある。スキップ不可。HARD RULE。
3. **すべての subagent 出力を完全に emoji 付きで表示** — 各 subagent が推論サマリー（🔎 証拠 / 💭 検討した選択肢 / 🎯 判断）を表示。受信次第即座に表示。バッチ処理禁止。要約禁止。HARD RULE。
4. **Pro 環境は真の subagent を強制** — 実際の独立した subagent を起動する必要がある。単一コンテキストでの役割シミュレーションは禁止。HARD RULE。
5. **データ層デグレード** — 到達不能時「⚠️ second-brain 利用不可」と注記。Notion の利用不可はモバイル同期のみに影響し、コア機能には影響しない。

## モデル独立性

**本ファイル（GEMINI.md）は Gemini 専用オーケストレーションファイルである。** その他すべての知識 —— 抽出ルール、lint ルール、役割定義、知識ネットワーク、ディレクトリ構造 —— は任意の LLM が読めるピュア markdown である。同一の agent ファイル（`pro/agents/*.md`）が Claude、Gemini、Codex プラットフォーム間で共有される。

## ストレージバックエンド

Life OS は GitHub、Google Drive、Notion をストレージバックエンドとしてサポートする（1、2、または 3 つすべて）。ユーザーが初回セッション時に選択；設定は `_meta/config.md` に保存。マルチバックエンド：選択したすべてに書き込み、プライマリから読み取り（自動：GitHub > GDrive > Notion）。セッション開始時にクロスデバイス同期。

- 標準データモデルと操作：`references/data-model.md`
- バックエンド固有アダプター：`references/adapter-github.md`、`references/adapter-gdrive.md`、`references/adapter-notion.md`
- アーキテクチャと認知パイプライン：`references/data-layer.md`

データ読み取りは早朝官が実行（セッション開始）；データ書き込みは起居郎が実行（セッション終了）。丞相はストレージバックエンドを直接操作しない。
