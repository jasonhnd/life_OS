# 戦略マップ仕様

戦略マップはプロジェクト間の関係レイヤーです。プロジェクトがどのようにグループ化され、流動し、相互に影響し合うかを記録します。これにより Life OS はプロジェクト単位ではなく、ポートフォリオレベルで推論できるようになります。

認知科学研究に基づいて設計：Goal Systems Theory (Kruglanski 2002)、Recognition-Primed Decision model (Klein 1998)、Predictive Coding framework (Friston 2005)。

## 設計原則

1. **2層アーキテクチャ**：構造層（ユーザー定義、変化が遅い）+ 動的層（システム計算、毎回の上朝で更新）
2. **唯一の情報源**：`projects/{p}/index.md` の frontmatter がプロジェクトごとの戦略データを保存；`_meta/strategic-lines.md` がライン定義を保存；`_meta/STRATEGIC-MAP.md` はコンパイル出力（手動編集不可）
3. **ゼロから成長**：戦略データが存在しなければ、システムは通常通り動作する。ユーザーが関係を定義するまで、この機能は休眠状態
4. **ユーザー確認による構造変更**：新しい戦略ライン、プロジェクトの役割、フロー関係はユーザーの確認が必要（SOUL/wiki 候補と同様）
5. **パターンマッチング + ナラティブ評価**：数値スコアは使用しない。健全性アーキタイプにマッチし、何が起きているか、何を意味するか、何をすべきかについてのストーリーを書く
6. **クロスレイヤー統合**：戦略マップは SOUL.md、wiki/、DREAM と連携して一つの認知システムを構成する

## データ構造

### 戦略ライン（`_meta/strategic-lines.md`）

ユーザーのセカンドブレインに保存（Life OS リポジトリではない）。複数のラインは `---` で区切る。ユーザーが初めて戦略グルーピングを定義した時に作成される。

```yaml
---
type: strategic-line
id: "crypto-compliance"
name: "Crypto Compliance Pipeline"
purpose: "Build regulated crypto service infrastructure"
driving_force: "Establish first-mover advantage in regulated crypto services before market consolidation"
health_signals:
  - "Regulatory filings progressing"
  - "Partner onboarding on track"
  - "Legal review turnaround within expected windows"
time_window: 2026-09-30
area: "ventures"
created: 2026-04-15
---
```

| フィールド | 型 | 必須 | 説明 |
|-----------|------|------|------|
| id | string | はい | 一意識別子（kebab-case） |
| name | string | はい | 表示名 |
| purpose | text | はい | 一文の正式な目的 |
| driving_force | text | いいえ | このラインへの投資を本当に駆動するもの（purpose と異なる場合がある） |
| health_signals | text[] | いいえ | このラインが健全であることを示すシグナル（AI が driving_force に基づいて提案、ユーザーが確認） |
| time_window | date | いいえ | ライン全体に影響する期限 |
| area | string | いいえ | 関連する生活エリア |
| created | date | 自動 | 作成日 |

#### `driving_force` について

Goal Systems Theory (Kruglanski 2002) と自己欺瞞研究 (von Hippel & Trivers 2011) に基づく：人間は表向きの理由とより深い動機で目標を追求することが多い。`purpose` は公式な位置づけ；`driving_force` は実際にリソース配分を駆動するもの。

- driving_force = purpose の場合、空のままにする（システムは同一として扱う）
- driving_force はどの健全性シグナルが重要かに影響する。「関係性」に駆動されるラインは、コードのコミット数ではなくソーシャルアクティビティを監視すべき
- 諫官は行動が driving_force と一致しているかをチェックする（purpose だけではない）

#### `health_signals` について

- 戦略ラインの初回作成時、早朝官が driving_force に基づいて健全性シグナルを提案
- ユーザーが確認または修正
- 確認されたシグナルは保存され、以降の評価に使用される
- DREAM はラインの進化に伴いシグナルの更新を提案する可能性がある

### プロジェクト別戦略フィールド（`projects/{p}/index.md`）

既存の frontmatter に追加する。すべてのフィールドはオプションで、デフォルトは空/null。

```yaml
strategic:
  line: "crypto-compliance"
  role: "critical-path"
  flows_to:
    - target: "bittrade-jetro"
      type: "cognition"
      description: "Certification results reused"
  flows_from:
    - source: "ndfg"
      type: "cognition"
      description: "Industry knowledge input"
  last_activity: 2026-04-12
  status_reason: "Waiting for legal review, expected mid-May"
```

| フィールド | 型 | 値 | 説明 |
|-----------|------|------|------|
| strategic.line | string | strategic-line id を参照 | このプロジェクトが属する戦略ライン |
| strategic.role | enum | `critical-path` / `enabler` / `accelerator` / `insurance` | ライン内での役割 |
| strategic.flows_to[] | array | オブジェクト: {target, type, description} | 他のプロジェクトへの出力フロー |
| strategic.flows_from[] | array | オブジェクト: {source, type, description} | 他のプロジェクトからの入力フロー |
| strategic.last_activity | date | ISO 日付 | 最終更新日（起居郎が自動更新） |
| strategic.status_reason | text | 自由テキスト | このプロジェクトが現在のステータスにある理由（制御された待機と制御不能な停滞を区別する鍵） |

### 役割の定義

| 役割 | 意味 | 制約 |
|------|------|------|
| `critical-path` | これが停滞すると、ライン全体が停滞する | 各ラインに正確に1つ |
| `enabler` | critical-path が進む前に完了する必要がある | 複数可 |
| `accelerator` | ラインを加速するがブロッキングではない | 複数可 |
| `insurance` | 主要アプローチが失敗した場合のリスク軽減 | 複数可 |

### フロータイプの定義

| タイプ | 意味 | ブロック時の緊急度 |
|--------|------|-------------------|
| `cognition` | 結論や知識が別のプロジェクトの意思決定に情報を提供する | 中（非同期、wiki エントリ経由で運搬） |
| `resource` | 成果物、資産、コード、またはデリバラブルが別のプロジェクトに流れる | 高（具体的な依存関係） |
| `decision` | ある決定が別のプロジェクトの選択肢を制約または無効化する | クリティカル（即時同期が必要） |
| `trust` | あるプロジェクトで構築された関係資本が別のプロジェクトに利益をもたらす | 低（長期的な蓄積） |

## 評価方法

### 健全性アーキタイプ（数値スコアではない）

Klein の Recognition-Primed Decision model に基づく：専門家は経験した事例とのパターンマッチングによって状況を評価し、ナラティブ評価を生成する——加重平均の計算ではない。

| アーキタイプ | シグナル | 意味 |
|-------------|---------|------|
| 🟢 順調な進捗 | active + 最近の活動あり + タスクは予定通り | 介入不要 |
| 🟡 制御された待機 | on-hold + status_reason あり + 予想ウィンドウ内 | 正常だがタイムウィンドウを監視 |
| 🟡 モメンタム減衰 | active だが活動が減少 + タスクが蓄積 | 注意の漂流の可能性 |
| 🔴 制御不能な停滞 | on-hold + status_reason なし、または予想ウィンドウ超過 | 介入が必要 |
| 🔴 方向のドリフト | active だが行動が driving_force と不一致 | 間違ったことをしている |
| ⚪ 休眠 | insurance の役割 + 長期間非活動 | 想定内、懸念なし |

戦略ラインごとの評価プロセス：
1. すべてのシグナルを読む（status、activity、tasks、status_reason、driving_force、health_signals）
2. アーキタイプにマッチ
3. メンタルシミュレーション：このまま続けると、3 週間後にこのラインはどこにいるか？
4. ナラティブを書く：現在の状態 + 何を意味するか + 何を見守るか
5. ブラインドスポットをフラグ：どの情報が欠けているか？

### 減衰閾値

| 役割 | 警告 | アラート |
|------|------|---------|
| critical-path | 7 日間非活動 | 14 日 |
| enabler | 14 日 | 30 日 |
| accelerator | 30 日 | 60 日 |
| insurance | 60 日 | アラートなし（休眠が想定） |

### 意思決定の伝播

プロジェクトで意思決定がなされた際、flows_to をチェックして下流への影響を確認：

| フロータイプ | 下流への影響 | 深刻度 |
|-------------|------------|--------|
| `decision` | 下流プロジェクトの前提が無効化される可能性 | クリティカル——即座にフラグ |
| `cognition` | 変更された結論が下流の分析に影響 | 中——レビュー用にメモ |
| `resource` | デリバラブルの変更が下流の入力に影響 | 中——下流が必要なものをまだ持っているか検証 |
| `trust` | 関係の変化が下流の信頼資本に影響 | 低——認識のためにメモ |

チェック担当：
- **門下省**：審議中（意思決定前、封駁基準として）
- **起居郎**：第2フェーズ中（意思決定後、outbox 内の警告として）
- **早朝官**：コンパイル中（前のセッションからの古い警告）

### ブラインドスポット検出

Predictive Coding (Friston 2005) と Kahneman の不在の無視に関する研究に基づく：最も危険な認知の失敗は間違いを見つけることではなく、欠けているものに気づかないこと。

| ブラインドスポットの種類 | 検出者 | タイミング |
|------------------------|--------|-----------|
| 未定義の関係（どのラインにも属さないプロジェクト） | 早朝官 | 上朝——未所属プロジェクトをスキャン |
| 駆動力の軽視（行動が driving_force と不一致） | 諫官 | 毎回の三省フロー後 |
| ディメンションギャップ（すべての戦略ラインから欠落している生活エリア） | DREAM REM | 3 日間スキャン——生活ディメンションのカバレッジを確認 |
| 準備なしで迫るタイムウィンドウ | 早朝官 + 兵部 | 上朝ブリーフィング + 実行評価 |
| 壊れたフロー（定義されているが実際には流れていない） | 門下省 + 起居郎 | 審議——wiki 参照を確認；退朝——セッション資料を確認 |

## クロスレイヤー統合

### SOUL x 戦略マップ

`driving_force` は本質的に「この戦略ラインが SOUL のどの部分に奉仕するか」。

- SOUL.md が「家庭 > キャリア」（信頼度 0.8）と示しているのに、すべての戦略ラインがキャリア関連の driving force を持つ場合 → SOUL-戦略の不一致としてフラグ
- 門下省のチェック：この意思決定の戦略ラインは SOUL ディメンションと一致しているか？
- DREAM REM のチェック：driving force は時間の経過とともに SOUL と一貫しているか？

### wiki x 戦略マップ

プロジェクト間の認知フローは wiki コンテンツによって運搬される。

- 認知フローが定義され（A → B）、wiki に A のドメインからのエントリがある場合、それらのエントリがフローの実質的な内容
- 早朝官のクロスチェック：認知フローの wiki ドメインにコンテンツがあるか？下流プロジェクトがそれを参照しているか？
- 「壊れたフロー」 = wiki エントリは存在するが、下流プロジェクトが一度も引用していない

### DREAM x 戦略マップ

DREAM は3つの知識レイヤーを横断する統合エンジン。

戦略マップなし：DREAM REM はオープンエンドの「プロジェクト間のつながりはありますか？」を問う
戦略マップあり：DREAM REM には足場がある：
- **構造チェック**：定義されたフローはまだ有効か？古くなったり壊れたりしていないか？
- **SOUL x 戦略**：driving force は SOUL ディメンションと一貫しているか？
- **パターン x 戦略**：ユーザーの行動は戦略的優先順位と一致しているか？
- **wiki x フロー**：知識は実際にプロジェクト間で転送されているか？
- **構造を超えて**：戦略マップがまだ捕捉していないつながりは何か？

## コンパイル出力：`_meta/STRATEGIC-MAP.md`

早朝官が毎回の上朝時にコンパイル（ステップ 8.5）。手動編集不可。

### コンパイルアルゴリズム

```
1. _meta/strategic-lines.md を読取 → 全ライン定義
2. 全 projects/*/index.md を読取 → 戦略フィールドを収集
3. 各ラインについて：
   a. strategic.line が一致するプロジェクトを収集し、役割でソート
   b. 健全性アーキタイプをマッチ（status + activity + tasks + status_reason + health_signals を使用）
   c. ナラティブ評価を記述
   d. ライン単位のブラインドスポットを検出
4. クロスレイヤー検証：
   a. SOUL x driving_force の一致性
   b. wiki x 認知フローの完全性
   c. ユーザーパターン x 戦略的優先順位
5. 未所属プロジェクトを列挙
6. 全 flows_to/flows_from からフローグラフを構築
7. アクション推奨を生成（レバレッジランキング + 安全に無視可能 + 意思決定が必要）
8. _meta/STRATEGIC-MAP.md に書き込む
```

## アクション推奨ロジック

Shenhav, Botvinick & Cohen (2013) に基づく：脳は「コントロールの期待値」を計算する。単なる重要度ではなく、労力コストと機会コストで重み付けされた重要度。

### 最高レバレッジの選択

```
1. タイムウィンドウが迫っている（4 週間未満）戦略ラインはあるか？
   → そのラインの未完了作業が最高優先度を得る

2. critical-path プロジェクトが停滞しているか？
   → 待機中にそのラインの他のものを前に進められるか？
   → はいなら → enabler/accelerator を前進させる（待機期間を活用）

3. 軽視されている driving_force はあるか？
   → 行動パターンが戦略的優先事項の回避を示す → フラグ

4. 上記のいずれもない → 最も健全なラインの次のステップを前進させる
   （モメンタムには価値がある——良い勢いを無駄にしない）
```

### 安全に無視可能

低優先度項目の能動的抑制（Desimone & Duncan 1995：注意は主に無関係なシグナルの抑制によって機能する）。

プロジェクトが安全に無視できる条件：
- 順調（🟢 アーキタイプ）かつ critical-path ではない
- insurance の役割で、主要アプローチに失敗の兆候がない
- 健全な別の戦略ラインに属している

### 意思決定が必要

システムが構造的ギャップを検出した際に生成：
- どのラインにも割り当てられていないプロジェクト
- フローはあるが役割がないプロジェクト
- critical-path プロジェクトのない戦略ライン
- ユーザーがまだ確認していない health_signals

## コールドスタート

`_meta/strategic-lines.md` が存在しない場合：
- 早朝官は戦略コンパイルをサイレントにスキップ
- ブリーフィングはオリジナルのエリアステータスのフラットリスト形式にフォールバック
- 複数プロジェクトを持つ 3 回以上のセッション後、DREAM REM が提案する可能性：「N 個のアクティブなプロジェクトがありますが、戦略的関係が定義されていません。それらの関連を整理しますか？」
- ユーザーはいつでも丞相に伝えることで関係を定義できる

初回セットアップフロー：
1. ユーザーが「プロジェクトを整理しよう」と言う（または DREAM が提案）
2. 丞相が会話を導く：「どのプロジェクトが同じ目的に奉仕していますか？」→「それらの間で何が流れていますか？」→「これに取り組む本当の駆動力は何ですか？」
3. 起居郎が strategic-lines.md + 戦略フィールド付きの index-delta.md を書き込む
4. 早朝官が driving_force に基づいて health_signals を提案 → ユーザーが確認
5. 次の上朝 → 初回の戦略マップコンパイル

## グレースフルデグラデーション

- strategic-lines.md がない → 完全にスキップ、フラットなエリアステータスにフォールバック
- strategic-lines.md は存在するがプロジェクトに戦略フィールドがない → 空のマップをコンパイル、ブリーフィングではスキップ
- フローが存在しないプロジェクトを参照 → 警告で古いとフラグ
- 戦略ラインにプロジェクトが1つだけ → エラーではない、「シングルプロジェクトライン」と注記
- critical-path プロジェクトが完了に変わる → フラグ：「ラインに新しい critical-path の割り当てが必要かもしれません」
- SOUL.md が存在しない → SOUL x 戦略の一致性チェックをスキップ
- wiki/ が存在しない → wiki x フロー検証をスキップ
