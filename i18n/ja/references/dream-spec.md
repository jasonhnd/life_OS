# DREAM 仕様書

DREAMはシステムのオフラインメモリ処理 — 人間の睡眠サイクルから着想を得ています。**ARCHIVER（`pro/agents/archiver.md`）の Phase 3** として毎回の退朝時に実行され、直近3日間の活動をスキャンして整理、統合、発見を行います。

> **注意**：DREAM はもはや独立エージェントではありません。v1.4.4 でARCHIVERに統合されました。本仕様は三段階を定義し、ARCHIVERがそれを退朝フローに統合します。

## トリガー

```
ユーザーが「退朝」/ "adjourn" / "終わり" と言う
    ↓
RETROSPECTIVE: アーカイブ + 同期 + プッシュ（モード4）
    ↓
RETROSPECTIVE: DREAM エージェントを起動（最終ステップ）
    ↓
DREAM が3つのステージを実行
    ↓
ドリームレポートを _meta/journal/{date}-dream.md に書き込む
    ↓
セッション終了
```

DREAM が失敗またはタイムアウトした場合 → `_meta/sync-log.md` に警告を記録し、セッション終了をブロックしない。

---

## スコープ

**デフォルト：直近3日間（72時間以内）に変更されたファイル。** 3日間に変更がない場合、「前回のドリームレポート以降」に自動拡大（最新の `_meta/journal/*-dream.md` の日付を読取）。ドリームレポートが存在しない場合、直近7日間にフォールバック。

検出方法：
- GitHub バックエンド: `git log --since="3 days ago" --name-only --format=""` → 空の場合、`git log --since="{前回ドリーム日付}" --name-only --format=""`
- GDrive バックエンド: `modifiedTime > 3_days_ago` → 空の場合、`modifiedTime > 前回ドリーム日付`
- Notion バックエンド: `last_edited_time > 3_days_ago` → 空の場合、`last_edited_time > 前回ドリーム日付`

## 3つのステージ

人間の睡眠アーキテクチャから着想を得た：

### ステージ N1-N2: 整理 & アーカイブ

**人間の相当物**: 浅い眠り — 脳がその日の体験をカテゴリに整理する。

直近3日間の未解決事項をスキャン：
- `inbox/` の未分類アイテム → ターゲットプロジェクト/エリア/wikiを提案
- 抽出可能なインサイトを含む `_meta/journal/` エントリ → `user-patterns.md` の更新を提案
- 期限切れまたは重複した `projects/*/tasks/` → クリーンアップのためにフラグを立てる
- 孤立ファイル（作成されたがいずれの index.md からもリンクされていない）→ フラグを立てる

各発見について分類：
- **自動修正可能**（例：インデックスエントリの欠落）→ レポートに記録
- **ユーザー入力が必要**（例：このインボックスアイテムを分類する）→ 推奨アクションに追加

### ステージ N3: 深い統合

**人間の相当物**: 深い眠り — 重要な記憶が短期記憶から長期記憶へと移動する。

N3 は二つの問いを発する：
- **Q1（人について）**：このセッションはユーザーについて新しいことを明らかにしたか？ → SOUL 候補
- **Q2（世界について）**：このセッションは次回のための再利用可能な結論を生み出したか？ → Wiki 候補

直近3日間から、より深いパターンを抽出：
- 決定から再利用可能な結論 → Wiki 候補を提案する（以下の Wiki 候補フォーマットを参照）
  **重複チェック**：最新の outbox manifest を確認——`wiki: N`（N > 0）の場合、その session の退朝フローですでに wiki 候補を抽出済み。退朝フローが見逃した結論のみに集中し、再提案しないこと。
- wiki/INDEX.md をスキャン（存在する場合）：既存エントリを裏付ける新しい証拠 → evidence_count の更新を提案；既存エントリと矛盾する新しい証拠 → challenges の更新を提案
- 新しい証拠に基づいて更新が必要な `user-patterns.md` エントリ → 変更を提案
- **SOUL.md 候補エントリ** → 新規エントリまたは既存エントリへの更新を提案（以下のSOUL候補フォーマットを参照）

### ステージ REM: 創造的なつながり

**人間の相当物**: レム睡眠 — 脳が無関係な記憶の間に予期せぬつながりを作る。ここで夢が起き、洞察が生まれる。

このステージにチェックリストはない — データに語らせる：
- **クロスプロジェクトの関連**: プロジェクトAで学んだことがプロジェクトBに応用できるか？
- **行動と価値の整合性**: SOUL.md が存在する場合、最近の行動は表明された価値観と一致しているか？
- **ディメンションの盲点**: 最近の決定で完全に欠落していた人生のディメンションは何か？
- **予期せぬ洞察**: ユーザーを驚かせるようなパターンやつながりは何か？

量より質。1〜3の真の洞察。明白でないものが何も出てこない場合はそう言う — 洞察を捏造しない。

## SOUL 候補フォーマット

DREAMがSOUL.mdに記録する価値のある価値パターンを発見した場合：

```
🌱 SOUL 候補:
- ディメンション: [名前]
- 観察: [データで観察したこと]
- 証拠:
  - [日付] [決定/行動]
  - [日付] [決定/行動]
- 提案エントリ:
  - 実態: [観察されたパターン]
  - 理想: [ユーザーが確認する]
  - 差距: [明らかな場合]
```

ユーザーは次の登朝時に確認、編集、または拒否します。

## Wiki 候補フォーマット

DREAMが wiki/ に記録する価値のある再利用可能な結論を発見した場合：

```
📚 Wiki 候補:
- ドメイン: [ドメイン名]
- トピック: [短い識別子]
- 結論: [一文 — 再利用可能なテイクアウェイ]
- 証拠:
  - [日付] [決定/行動] — [説明]
  - [日付] [決定/行動] — [説明]
- 適用場面: [どのようなシナリオでこの知識を思い出すか]
```

既存の wiki エントリについては、更新を提案する：

```
📚 Wiki 更新:
- エントリ: wiki/[domain]/[file].md
- 変更: evidence_count +1（または challenges +1）
- 新しい証拠: [日付] [何が起こったか]
```

ユーザーは次の登朝時に確認、編集、または拒否します。

## 朝のブリーフィングへの統合

次の登朝時、RETROSPECTIVEは最新の未読ドリームレポートを読み取り、ブリーフィングに含めます：

```
💤 前回のセッションでシステムは夢を見ました:
- [主要な発見の1〜3行サマリー]
- [SOUL候補（もしあれば）のユーザー確認]
```

提示後、ドリームレポートを「提示済み」としてマークし、再度表示されないようにします。

## 制約

- **3日間のスコープは厳守** — 関連性が高そうでも古いファイルはスキャンしない
- **SOUL.md に直接書き込まない** — 候補を提案するのみ
- **wiki/ に直接書き込まない** — 候補と更新を提案するのみ
- **user-patterns.md に直接書き込まない** — 更新を提案するのみ
- **簡潔さ** — ドリームレポートは20〜50行であるべき、500行ではない
- **誠実さ** — 「重要な発見なし」は有効な夢。洞察を捏造しない
- **ブロックしない** — DREAMが失敗しても、セッションは通常通り終了する
