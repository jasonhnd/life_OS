# データレイヤーアーキテクチャ

全エージェントはデータの読み書き時にこのファイルを参照する。

## 設計原則

1. **全範囲カバー**: セカンドブレインは生活、家族、買い物、趣味、本業、副業 — すべてを網羅する
2. **モデル非依存**: 特定のモデルに縛られない。すべての「知能」はマークダウンファイルにエンコードされ、モデルの重みには依存しない
3. **AIなしでも動作**: マークダウンファイルをObsidianで開けば、読み書きとナビゲーションが可能。LLMはアクセラレータであり、前提条件ではない
4. **マークダウンが唯一の信頼源**: すべての知識は最終的に.mdファイルに集約される。Notionはトランスポートレイヤー（受信箱）であり、ストレージレイヤーではない
5. **Obsidianが閲覧レイヤー**: GitHubリポジトリをローカルにクローンし、Obsidianで開く。ウィキリンクと標準マークダウンリンクにより、知識グラフの自動可視化が可能

## モデル非依存

**CLAUDE.mdのみが特定モデルに紐づくファイルである。** その他すべて — 抽出ルール、検査ルール、ロール定義、知識ネットワーク、ディレクトリ構造 — は任意のモデルが読める純粋なマークダウンである。モデル切り替え時はCLAUDE.mdの参照を更新するだけでよい。

## 認知パイプライン

情報は5つのステージを通過し、それぞれが方法論に対応する：

```
知覚 → 捕捉 → 連想 → 判断 → 定着 → 創発
   ↑         ↑          ↑          ↑        ↑         ↑
 スマホ      GTD      Zettelkasten  三省六部   PARA    検査/御史台
 体験      inbox/    wiki/      デスクトップ  dirs    ヘルスチェック
```

### ステージ詳細

**知覚 → 捕捉 (GTD)**: モバイルでのゼロフリクション捕捉。ユーザーが何か言えば、スマホAIが受信箱に保存する。この段階では分類しない — 受信箱はGTDの収集バスケットである。

**捕捉 → 連想 (Zettelkasten)**: デスクトップCCが受信箱から取得し、最初のステップは連想の構築 — 既存のどのwiki記事が関連するか？ どのエンティティが言及されているか？ エンティティ間にどのような関係があるか？ ウィキリンクでバックリンクネットワークを構築する。

**連想 → 判断 (三省)**: すべての情報に判断が必要なわけではない。大規模なリソース配分、複数選択肢のトレードオフ、または取り消し困難な結果が伴う場合のみ、三省の意思決定モードを起動する。

**判断 → 定着 (PARA)**: 意思決定の結論はPARAに従って保存 — Projects（終点のあるもの）、Areas（継続的な領域）、Wiki（領域横断の知識）、Archive（完了済み）。

**定着 → 創発 (検査/御史台)**: 知識ネットワークの密度が臨界質量に達すると、創発が自動的に起こる。検査スキャンが領域横断の連想を発見し、新しいコンセプト記事を提案し、矛盾を発見する。

### モバイル vs デスクトップの分担

モバイルは知覚と捕捉のみを担当する（時折、ウェブ検索のような軽量な連想を行う）。デスクトップは連想、判断、定着、創発 — すべての重い処理を担当する。モバイルはパイプライン出力（STATUS.md、アーカイブ）を読めるが、書き込みは捕捉ステージのみ。

## 知識分類（7タイプ）

| タイプ | 保存先 | 例 |
|------|---------|---------|
| エンティティ知識 | wiki/ | ある企業が製品ラインを廃止した |
| 経験知識 | wiki/（主観と注記） | 素材Xは素材Yより手触りが良い |
| 関係知識 | wiki/（バックリンク） | 人物AはイベントBを通じて出会った |
| 意思決定記録 | areas/ または projects/ | プロジェクトがツールAからツールBに切り替えた |
| ToDo / 意向 | tasks/ | 次回製品Xを試す |
| インスピレーション / 直感 | inbox/（一時的） | XとYの間にチャンスがある |
| プロセス知識 | wiki/ | 日本で会社を登記する手順 |

これらの7タイプは、実際の使用状況に基づいて今後拡張される可能性がある。

## 知識抽出：4ステップトレーニング

1. **ユーザーが決定**: デスクトップCCが「抽出提案」を生成し、ユーザーが確認/修正する
2. **サンプル蓄積**: `_meta/extraction-log.md`に記録する
3. **LLMがルールを帰納**: ログから嗜好を帰納し、`_meta/extraction-rules.md`に書き込む（純粋なマークダウン、モデル非依存）
4. **定期的な修正**: ユーザーが月次でレビューし、誤分類を報告し、CCがルールを更新する

核心：「学習」の担体はマークダウンファイルであり、モデルの重みではない。モデル切り替え時はこれらのファイルを読むだけでよい。

## 御史台：2つの運用モード

御史台は三省制度内で2つのモードで運用される：

### モード1：決策レビュー（既存）

三省六部ワークフローの後に毎回、官吏の業務品質をレビューする。`pro/agents/yushitai.md`で既に定義済み。

### モード2：巡検（新規）

アイドル時に、各部が自身の管轄を巡検する。`_meta/roles/censor.md`で定義。

#### トリガーレベル

| トリガー | タイミング | 深度 |
|---------|------|-------|
| **起動時巡検** | デスクトップCCセッション開始時、`lint-state.md`で最終実行から>4h経過の場合 | 軽量、3行ブリーフィング |
| **同期後巡検** | 受信箱同期完了後 | 新コンテンツとwikiの整合性チェック、wiki記事が必要な新エンティティ、STATUS.md更新 |
| **深度巡検** | 週次または手動トリガー | 六部全体の巡回巡検 |

#### 六部巡検管轄

| 部 | 管轄 | チェック内容 |
|----------|-------------|--------|
| 戸部 | areas/finance/ | 投資戦略の陳腐化、財務数値の更新要否 |
| 兵部 | projects/ | プロジェクト活動状況、TODO完了率、リソース競合 |
| 礼部 | wiki/（人間関係） | 未履行の社交コミットメント、記録すべき新しい連絡先 |
| 工部 | wiki/ + _meta/ | 孤立ファイル、壊れたリンク、ルールの妥当性 |
| 吏部 | areas/career/ | キャリア方針と実際の行動の整合性 |
| 刑部 | 領域横断 | プロジェクト間の戦略矛盾、リスク評価が欠落した意思決定 |

#### 問題分類

| レベル | アクション | 例 |
|-------|--------|---------|
| **自動修正** | 御史台が直接対処 | インデックスエントリ欠落、バックリンク欠落、フォーマット問題 |
| **提案** | 受信箱に送信しユーザーへ | データ不整合、プロジェクト停滞の可能性、wiki提案 |
| **エスカレーション** | 三省意思決定モードを起動 | >¥1Mの財務矛盾、複数プロジェクト戦略競合、対人リスク |

#### 実装

- ロール定義は`_meta/roles/censor.md`に保存、CLAUDE.mdはそれを参照するのみ
- 巡検状態は`_meta/lint-state.md`に永続化（LLMのクロスセッションメモリ不足を解決）
- 巡検レポートは`_meta/lint-reports/`に保存、サマリーも受信箱に送信
- モデル切り替え時：ロールファイルは変更不要、CLAUDE.mdの参照のみ変更

## Notion（トランスポートレイヤー、3コンポーネント）

Notionはトランスポートレイヤーのみであり、ストレージレイヤーではない。

### 📬 受信箱（データベース）

モバイルとデスクトップ間で情報を受け渡すためのメッセージキュー。

| フィールド | タイプ | 説明 |
|-------|------|-------------|
| Content | Title | 一文またはキーポイント |
| Source | Select | Mobile / Desktop |
| Status | Select | Pending / Synced |
| Time | Created time | 自動 |

### 🧠 ステータスダッシュボード（ページ）

`_meta/STATUS.md`のミラー。各セッション終了時にCCが上書きする。

### 🗄️ アーカイブ

モバイルからの読み取り専用アーカイブアクセス。

## 早朝官のデータ操作

### 家事モード（会話開始時）

```
1. ~/second-brain/inbox/ を読む（未処理アイテム）
2. ~/second-brain/_meta/STATUS.md を読む（グローバルステータス）
3. ~/second-brain/_meta/lint-state.md を読む（巡検要否チェック：最終実行から>4h）
4. ~/second-brain/projects/{bound}/index.md を読む（紐づけプロジェクトのステータス）
5. ~/second-brain/projects/{bound}/decisions/ を読む（過去の意思決定）
6. ~/second-brain/projects/{bound}/tasks/ を読む（アクティブタスク）
7. user-patterns.md を読む
8. グローバル概観：全 projects/ と areas/ の index.md タイトル＋ステータスを一覧
9. Notion 📬 受信箱をチェック（モバイルからの新メッセージ）→ second-brain/inbox/ に取り込む
10. lint-state.md で最終実行から>4h → 御史台の軽量巡検をトリガー
11. プラットフォーム検出 + バージョンチェック
```

### ラップアップモード（プロセス終了時）

```
1. 奏折 → projects/{p}/decisions/ または _meta/decisions/（領域横断）
2. アクションアイテム → projects/{p}/tasks/ または areas/{a}/tasks/
3. 御史台レポート → _meta/journal/
4. 諫官レポート → _meta/journal/
5. _meta/STATUS.md を更新（グローバルステータススナップショット）
6. _meta/lint-state.md を更新（最終巡検時刻）
7. user-patterns.md を更新
8. git add + commit + push（second-brainリポジトリへ）
9. Notion同期：🧠 ステータスダッシュボードを更新
```

### レビューモード

```
1. _meta/STATUS.md を読んでグローバル状態を把握
2. projects/*/tasks/ を走査して完了率を算出
3. areas/*/goals.md を読んで目標進捗を確認
4. _meta/journal/ の最近のジャーナルを読む
5. _meta/lint-reports/ の最近の巡検レポートを読む
6. ファイルから算出したメトリクスダッシュボード
```

## 諫官のデータ取得

```
1. user-patterns.md を読む
2. _meta/journal/ の直近3件の諫官レポートを読む
3. projects/*/decisions/ + _meta/decisions/ の直近5件の意思決定を読む
4. projects/*/tasks/ を走査して完了率を算出
```

## 縮退ルール

- second-brainリポジトリにアクセス不可 → 「⚠️ second-brain利用不可」と注記
- Notionが利用不可 → モバイル同期にのみ影響、コア機能には影響しない
- 両方利用不可 → 通常通り動作、出力は会話内に表示されるが永続化されない
