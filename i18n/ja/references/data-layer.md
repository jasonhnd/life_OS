# データレイヤーアーキテクチャ

すべてのエージェントはデータの読み書き時にこのファイルを参照します。

## 設計原則

1. **完全網羅**: セカンドブレインは生活、家族、買い物、趣味、本業、副業 — すべてをカバーする
2. **LLM非依存**: 特定のモデルに縛られない。すべての「知能」はマークダウンファイルにエンコードされ、モデルの重みには依存しない
3. **AI不要でも動作**: マークダウンファイルをObsidianで開けば、読み書きやナビゲーションが可能。LLMはアクセラレーターであり、前提条件ではない
4. **マークダウンが唯一の真実の源**: すべての知識は最終的に .md ファイルに落ち着く。Notionはトランスポートレイヤー（インボックス）であり、ストレージレイヤーではない
5. **Obsidianが閲覧レイヤー**: GitHubリポジトリをローカルにクローンし、Obsidianで開く。Wikiリンクと標準マークダウンリンクにより、自動的にナレッジグラフの可視化が可能

## モデル非依存

**CLAUDE.mdは特定のモデルに紐づく唯一のファイルです。** それ以外すべて — 抽出ルール、lintルール、役割定義、知識ネットワーク、ディレクトリ構造 — は任意のモデルが読み取れる純粋なマークダウンです。モデルの切り替えはCLAUDE.mdの参照を更新するだけで済みます。

## 認知パイプライン

情報は6つのステージを流れ、それぞれが方法論にマッピングされます：

```
知覚 → 捕捉 → 判断 → 定着 → 連想 → 戦略化 → 創発
   ↑         ↑        ↑      ↓   ↘        ↑           ↑           ↑
 スマホ      GTD      3D6M  SOUL  Wiki   ROUTER+Wiki  戦略       DREAM REM
 体験      inbox/  デスクトップ (人)(知識) INDEX照合    MAP       領域横断
```

### ステージ詳細

**知覚 → 捕捉（GTD）**: モバイルでのゼロフリクション捕捉。ユーザーが何か言えば、スマホのAIがインボックスに保存する。この段階では分類なし — インボックスがGTDの収集バスケット。

**捕捉 → 判断（Draft-Review-Execute）**: デスクトップCCがインボックスから取り出す。すべての情報に決定が必要なわけではない。重大なリソース配分、複数オプションのトレードオフ、取り消し困難な結果が関わる場合のみ、Draft-Review-Executeの決定モードを起動する。

**判断 → 定着（SOUL + Wiki）**: 決定からの結論は二つのプールに定着する — SOUL（人について：価値観、パーソナリティ、行動パターン）と Wiki（世界について：再利用可能な知識、確立された結論）。DREAM が N3 段階で両方の候補を抽出し、ユーザーが次の上朝時に確認する。

**定着 → 連想（ROUTER + Wiki INDEX）**: ROUTERがセッション開始時に wiki/INDEX.md を読み取る。新しいリクエストが届くと、既存の知識が自動的に照合される — 「この領域について、すでに X がわかっています。」これにより蓄積された知識がアクティブなコンテキストに変わる。

**連想 → 戦略化（戦略マップ）**: ROUTERがセッション開始時に `_meta/STRATEGIC-MAP.md` を読む。リクエストが戦略的関係を持つプロジェクトに関わる場合、システムは下流の依存関係、ボトルネックの状況、意思決定伝播の警告を自動的に表面化する。これにより、プロジェクト単位の分析が戦略ライン対応の分析に変わる。`references/strategic-map-spec.md` を参照。

**戦略化 → 創発（DREAM REM）**: Wiki エントリと戦略的関係が蓄積されると、DREAM の REM ステージがフローグラフを足場として領域横断のつながりを発見する — ユーザーを驚かせる洞察。知識と関係が定着するほど、創発が増える。AUDITORの巡検も wiki の矛盾、ナレッジギャップ、プロジェクト間の戦略的矛盾を検出する。

### モバイル vs デスクトップの分業

モバイルは知覚と捕捉のみを担当（たまにWeb検索のような軽量な連想も）。デスクトップは連想、判断、定着、創発 — すべての重い処理を担当する。モバイルはパイプラインの出力（STATUS.md、アーカイブ）を読み取れるが、書き込みは捕捉ステージのみ。

## 知識分類（7タイプ）

| タイプ | 保存場所 | 例 |
|------|---------|---------|
| エンティティ知識 | wiki/ | ある企業がプロダクトラインを廃止した |
| 経験知識 | wiki/（主観マーク付き） | 素材Xの方が素材Yより良い感触 |
| 関係知識 | wiki/（バックリンク） | AさんはイベントBで知り合った |
| 決定記録 | areas/ または projects/ | プロジェクトがツールAからツールBに切り替えた |
| TODO / 意図 | tasks/ | 次回プロダクトXを試す |
| ひらめき / 直感 | inbox/（一時的） | XとYの間にチャンスがある |
| プロセス知識 | wiki/ | 日本で会社を登記する手順 |

これらの7タイプは実際の使用状況に基づき、将来拡張される可能性があります。

## 知識抽出: 4ステップトレーニング

1. **ユーザーが決定**: デスクトップCCが「抽出提案」を生成し、ユーザーが確認/修正
2. **サンプルを蓄積**: `_meta/extraction-log.md` に記録
3. **LLMがルールを帰納**: ログから好みを帰納し、`_meta/extraction-rules.md` に書き込む（純粋なマークダウン、モデル非依存）
4. **定期的な修正**: ユーザーが月次でレビューし、誤分類を報告、CCがルールを更新

核心: 「学習」の媒体はマークダウンファイルであり、モデルの重みではない。モデルの切り替えにはこれらのファイルを読み取るだけで良い。

## AUDITOR: 2つの運用モード

AUDITORはDraft-Review-Executeシステム内で2つのモードで運用されます：

### モード1: 決裁審査（既存）

すべてのDraft-Review-Executeワークフロー後に、公式作業の品質をレビューする。`pro/agents/auditor.md` で既に定義済み。

### モード2: 巡検（新規）

アイドル時に各部門が自身の管轄を巡検する。`_meta/roles/censor.md` で定義。

#### トリガーレベル

| トリガー | タイミング | 深度 |
|---------|------|-------|
| **起動時巡検** | デスクトップCCセッション開始時、`lint-state.md` が最終実行から4時間以上を示す場合 | 軽量、3行ブリーフィング |
| **同期後巡検** | インボックス同期完了後 | 新しいコンテンツとwikiの整合性チェック、新エンティティのwiki記事必要性、STATUS.md更新 |
| **深度巡検** | 週次または手動トリガー | 六領域全体の巡検 |

#### 六領域巡検の責任

| 部門 | 管轄 | チェック項目 |
|----------|-------------|--------|
| FINANCE | areas/finance/ | 投資戦略の陳腐化、財務数値の更新必要性 |
| EXECUTION | projects/ | プロジェクトの活動状況、TODOの完了率、リソースの競合 |
| GROWTH | wiki/（人間関係） | 未履行の社会的コミットメント、新しい連絡先の記録 |
| INFRA | wiki/ + _meta/ | 孤立ファイル、壊れたリンク、ルールの有効性 |
| PEOPLE | areas/career/ | キャリアの方向性と行動の一致 |
| GOVERNANCE | 横断 | プロジェクト間の戦略的矛盾、リスク評価が欠けている決定 |

#### 問題分類

| レベル | アクション | 例 |
|-------|--------|---------|
| **自動修正** | AUDITORが直接処理 | インデックスエントリの欠落、バックリンクの欠落、フォーマットの問題 |
| **提案** | ユーザーのインボックスに送信 | データの不整合、プロジェクトの停滞の可能性、wiki提案 |
| **エスカレーション** | Draft-Review-Executeの決定モードを起動 | 100万円超の財務的矛盾、マルチプロジェクトの戦略的競合、対人リスク |

#### 実装

- 役割定義は `_meta/roles/censor.md` に保存、CLAUDE.mdはそれを参照するのみ
- 巡検状態は `_meta/lint-state.md` に永続化（LLMのクロスセッションメモリの欠如を解決）
- 巡検レポートは `_meta/lint-reports/` に保存、サマリーもインボックスに送信
- モデルの切り替え: 役割ファイルは変更なし、CLAUDE.mdの参照のみ変更

## RETROSPECTIVEのデータ操作

すべての操作は標準インターフェースを使用します。ユーザーが設定したバックエンドに応じて呼び出しを適応させます。

### ハウスキーピングモード（会話開始時）

```
0. _meta/config.md を読取 → バックエンドリストと「このプラットフォーム」の最終同期タイムスタンプを取得
1. 設定済みの各バックエンドの MCP 可用性を確認（利用不可のものは SKIPPED とマーク）
2. マルチバックエンド同期（複数バックエンドが設定・利用可能な場合）:
   - 各利用可能な同期バックエンドに、このプラットフォームの last_sync_time 以降の変更をクエリ
   - 比較、競合解決（data-model.md を参照）
   - プライマリバックエンドに変更を適用
   - 同期バックエンドにプッシュ
3. Outbox マージ: _meta/outbox/ 内の未マージ session をスキャン
   - _meta/.merge-lock が存在し 5分未満 → マージをスキップ
   - .merge-lock を書き込む → 各 outbox をマージ → STATUS.md をコンパイル → commit + push → .merge-lock を削除
   - マージされた session をブリーフィングで報告
4. インボックス（未処理アイテム）を読取 — プライマリバックエンド経由
5. _meta/STATUS.md（グローバルステータス）を読取
5. _meta/lint-state.md を読取（巡検が必要か確認: 最終実行から4時間以上）
6. ReadProjectContext(バインドされたプロジェクト) — タスク、決定、ジャーナル
7. user-patterns.md を読取
8. グローバル概観: List Project + List Area（タイトル + ステータスのみ）
8.5. 戦略マップのコンパイル: `_meta/strategic-lines.md` が存在する場合 → 全 `projects/*/index.md` の戦略フィールドを読取 → `_meta/STRATEGIC-MAP.md` をコンパイル（アーキタイプマッチング、ナラティブ評価、クロスレイヤー検証、アクション推奨）。`references/strategic-map-spec.md` を参照。
9. lint-state.md が4時間以上を示す場合 → 軽量AUDITOR巡検をトリガー
10. プラットフォーム検出 + バージョンチェック
```

### ラップアップモード（プロセス終了時）

```
1. session-id を生成：date コマンドを実行して実際のタイムスタンプを取得し、{platform}-{YYYYMMDD}-{HHMM} 形式にする。捏造禁止——システムクロックを使用。HARD RULE。
2. _meta/outbox/{session-id}/ を作成
3. Save Decision / Save Task / Save JournalEntry → _meta/outbox/{session-id}/ へ（メインディレクトリには書き込まない）
4. index-delta.md を書く（projects/{p}/index.md への変更）
5. patterns-delta.md を書く（ADVISORが提案を持つ場合、user-patterns.md に追記する内容）
6. manifest.md を書く（session メタデータ）
7. git add _meta/outbox/{session-id}/ → commit → push（outbox ディレクトリのみ）
8. Notion に outbox を同期（設定されている場合）
9. _meta/config.md のこのプラットフォームの last_sync_time を更新
10. バックエンド障害 → _meta/sync-log.md にログ、ブロックしない

注意: projects/、STATUS.md、user-patterns.md には直接書き込まないこと。マージは次の上朝時に行う。
```

### レビューモード

```
1. _meta/STATUS.md からグローバル状態を読取
2. List Task（全プロジェクト）→ 完了率を計算
3. List Area → 目標を読取
4. List JournalEntry（最近）→ ジャーナルと巡検レポート
5. 結果からメトリクスダッシュボードを計算
```

## ADVISORのデータ取得

```
1. user-patterns.md を読取
2. List JournalEntry (type: remonstrator, limit: 3) → 直近3件のレポート
3. List Decision (limit: 5) → 最近の決定
4. List Task → 完了率を計算
```

## 唯一の情報源ルール

**`projects/{p}/index.md` が各プロジェクトのバージョン、フェーズ、ステータスの権威ある情報源である。** `_meta/STATUS.md` はコンパイルされたダッシュボード — index.md ファイルから生成しなければならず、手書きしてはならない。

| データ | 権威ある情報源 | コンパイルされたビュー |
|--------|-------------|-------------------|
| プロジェクトのバージョン/フェーズ/ステータス | `projects/{p}/index.md` | `_meta/STATUS.md` |
| エリアの目標/ステータス | `areas/{a}/index.md` | `_meta/STATUS.md` |
| タスク完了度 | `projects/{p}/tasks/*.md` | メトリクスダッシュボード |
| 行動パターン | `user-patterns.md` | ADVISORレポート |
| 戦略的関係 | `projects/{p}/index.md` 戦略フィールド + `_meta/strategic-lines.md` | `_meta/STRATEGIC-MAP.md` |

**書込順序の強制**：常に権威ある情報源を先に更新し、次にダッシュボードをコンパイルする。STATUS.md にプロジェクトレベルの情報を直接書き込んではならない。

**AUDITOR lint ルール**：巡検時に `_meta/STATUS.md` の各プロジェクトのバージョン/ステータスが `projects/{p}/index.md` と一致しているか確認する。不一致 → 🔴 を報告、権威ある情報源を正とする。

## デグレードルール

- プライマリバックエンドに到達不能 → 「⚠️ プライマリバックエンド利用不可」と注記
- 同期バックエンドに到達不能 → ⚠️ と注記、ログ、次のセッションでリトライ
- すべてのバックエンドが利用不可 → 通常通り動作、出力は会話に表示されるが永続化されない
