# データレイヤーアーキテクチャ

すべてのエージェントはデータの読み書き時にこのファイルを参照します。

## 設計原則

1. **完全網羅**: セカンドブレインは生活、家族、買い物、趣味、本業、副業 — すべてをカバーする
2. **LLM非依存**: 特定のモデルに縛られない。すべての「知能」はマークダウンファイルにエンコードされ、モデルの重みには依存しない
3. **AI不要でも動作**: マークダウンファイルをObsidianで開けば、読み書きやナビゲーションが可能。LLMはアクセラレーターであり、前提条件ではない
4. **マークダウンが唯一の真実の源**: すべての知識は最終的に .md ファイルに落ち着く。Notionはトランスポートレイヤー（インボックス）であり、ストレージレイヤーではない
5. **Obsidianが閲覧レイヤー**: GitHubリポジトリをローカルにクローンし、Obsidianで開く。Wikiリンクと標準マークダウンリンクにより、自動的にナレッジグラフの可視化が可能

## モデル非依存

**CLAUDE.mdは特定のモデルに紐づく唯一のファイルです。** それ以外すべて — 抽出ルール、lintルール、役割定義、知識ネットワーク、ディレクトリ構造 — は任意のモデルが読み取れる純粋なマークダウンです。モデルの切り替えはCLAUDE.mdの参照を更新するだけで済みます。

## 認知パイプライン

情報は5つのステージを流れ、それぞれが方法論にマッピングされます：

```
知覚 → 捕捉 → 連想 → 判断 → 定着 → 創発
   ↑         ↑          ↑          ↑        ↑         ↑
 スマホ      GTD      Zettelkasten  3D6M    PARA    Lint/御史台
 体験      inbox/    wiki/      デスクトップ   dirs    ヘルスチェック
```

### ステージ詳細

**知覚 → 捕捉（GTD）**: モバイルでのゼロフリクション捕捉。ユーザーが何か言えば、スマホのAIがインボックスに保存する。この段階では分類なし — インボックスがGTDの収集バスケット。

**捕捉 → 連想（Zettelkasten）**: デスクトップCCがインボックスから取り出し、最初のステップは連想の構築 — どの既存wikiの記事が関連するか？どのエンティティが言及されているか？エンティティ間にどのような関係があるか？Wikiリンクでバックリンクネットワークを構築する。

**連想 → 判断（三省）**: すべての情報に決定が必要なわけではない。重大なリソース配分、複数オプションのトレードオフ、取り消し困難な結果が関わる場合のみ、三省の決定モードを起動する。

**判断 → 定着（PARA）**: 決定からの結論はPARAに従って保存 — Projects（終点のあるもの）、Areas（継続的な領域）、Wiki（横断的知識）、Archive（完了済み）。

**定着 → 創発（Lint/御史台）**: 知識ネットワークの密度が臨界質量に達すると、創発が自動的に起こる。Lintスキャンが横断的な連想を発見し、新しいコンセプト記事を提案し、矛盾を見つける。

### モバイル vs デスクトップの分業

モバイルは知覚と捕捉のみを担当（たまにWeb検索のような軽量な連想も）。デスクトップは連想、判断、定着、創発 — すべての重い処理を担当する。モバイルはパイプラインの出力（STATUS.md、アーカイブ）を読み取れるが、書き込みは捕捉ステージのみ。

## 知識分類（7タイプ）

| タイプ | 保存場所 | 例 |
|------|---------|---------|
| エンティティ知識 | wiki/ | ある企業がプロダクトラインを廃止した |
| 経験知識 | wiki/（主観マーク付き） | 素材Xの方が素材Yより良い感触 |
| 関係知識 | wiki/（バックリンク） | AさんはイベントBで知り合った |
| 決定記録 | areas/ または projects/ | プロジェクトがツールAからツールBに切り替えた |
| TODO / 意図 | tasks/ | 次回プロダクトXを試す |
| ひらめき / 直感 | inbox/（一時的） | XとYの間にチャンスがある |
| プロセス知識 | wiki/ | 日本で会社を登記する手順 |

これらの7タイプは実際の使用状況に基づき、将来拡張される可能性があります。

## 知識抽出: 4ステップトレーニング

1. **ユーザーが決定**: デスクトップCCが「抽出提案」を生成し、ユーザーが確認/修正
2. **サンプルを蓄積**: `_meta/extraction-log.md` に記録
3. **LLMがルールを帰納**: ログから好みを帰納し、`_meta/extraction-rules.md` に書き込む（純粋なマークダウン、モデル非依存）
4. **定期的な修正**: ユーザーが月次でレビューし、誤分類を報告、CCがルールを更新

核心: 「学習」の媒体はマークダウンファイルであり、モデルの重みではない。モデルの切り替えにはこれらのファイルを読み取るだけで良い。

## 御史台: 2つの運用モード

御史台は三省システム内で2つのモードで運用されます：

### モード1: 決裁審査（既存）

すべての三省ワークフロー後に、公式作業の品質をレビューする。`pro/agents/yushitai.md` で既に定義済み。

### モード2: 巡検（新規）

アイドル時に各部門が自身の管轄を巡検する。`_meta/roles/censor.md` で定義。

#### トリガーレベル

| トリガー | タイミング | 深度 |
|---------|------|-------|
| **起動時巡検** | デスクトップCCセッション開始時、`lint-state.md` が最終実行から4時間以上を示す場合 | 軽量、3行ブリーフィング |
| **同期後巡検** | インボックス同期完了後 | 新しいコンテンツとwikiの整合性チェック、新エンティティのwiki記事必要性、STATUS.md更新 |
| **深度巡検** | 週次または手動トリガー | 六部全体の巡検 |

#### 六部巡検の責任

| 部門 | 管轄 | チェック項目 |
|----------|-------------|--------|
| 戸部 | areas/finance/ | 投資戦略の陳腐化、財務数値の更新必要性 |
| 兵部 | projects/ | プロジェクトの活動状況、TODOの完了率、リソースの競合 |
| 礼部 | wiki/（人間関係） | 未履行の社会的コミットメント、新しい連絡先の記録 |
| 工部 | wiki/ + _meta/ | 孤立ファイル、壊れたリンク、ルールの有効性 |
| 吏部 | areas/career/ | キャリアの方向性と行動の一致 |
| 刑部 | 横断 | プロジェクト間の戦略的矛盾、リスク評価が欠けている決定 |

#### 問題分類

| レベル | アクション | 例 |
|-------|--------|---------|
| **自動修正** | 御史台が直接処理 | インデックスエントリの欠落、バックリンクの欠落、フォーマットの問題 |
| **提案** | ユーザーのインボックスに送信 | データの不整合、プロジェクトの停滞の可能性、wiki提案 |
| **エスカレーション** | 三省の決定モードを起動 | 100万円超の財務的矛盾、マルチプロジェクトの戦略的競合、対人リスク |

#### 実装

- 役割定義は `_meta/roles/censor.md` に保存、CLAUDE.mdはそれを参照するのみ
- 巡検状態は `_meta/lint-state.md` に永続化（LLMのクロスセッションメモリの欠如を解決）
- 巡検レポートは `_meta/lint-reports/` に保存、サマリーもインボックスに送信
- モデルの切り替え: 役割ファイルは変更なし、CLAUDE.mdの参照のみ変更

## 三省の出力先

すべての出力は `references/data-model.md` の標準オペレーションを使用します。ユーザーが選択したバックエンドのアダプターがこれらをプラットフォーム固有の呼び出しに変換します。

| 出力 | 標準オペレーション |
|--------|-------------------|
| 決裁の奏折 | Save Decision |
| アクションアイテム | Save Task |
| 早朝 / 御史台 / 諫官レポート | Save JournalEntry |
| 巡検レポート | Save JournalEntry (type: inspection) |
| リサーチ / 知識 | Save WikiNote |
| 目標 | Update Area (goals フィールド) |
| グローバルステータス | アダプター固有のSTATUSメカニズムで更新 |

## ストレージバックエンド

Life OS は3つのストレージバックエンドをサポートしています。ユーザーは1つ、2つ、または全3つを選択します。

| バックエンド | 最適な用途 | アダプター | 形式 |
|---------|----------|---------|--------|
| GitHub | テクニカルユーザー、Claude Code | `references/adapter-github.md` | .md + フロントマター |
| Google Drive | 一般ユーザー、ゼロセットアップ | `references/adapter-gdrive.md` | .md + フロントマター |
| Notion | Notion ユーザー | `references/adapter-notion.md` | Notion データベース |

標準データ型とオペレーション: `references/data-model.md`

マルチバックエンドルール（同期、競合、削除、障害処理）: `references/data-model.md`

## 早朝官のデータ操作

すべての操作は標準インターフェースを使用します。ユーザーが設定したバックエンドに応じて呼び出しを適応させます。

### ハウスキーピングモード（会話開始時）

```
1. _meta/config.md を読取 → バックエンドリストと最終同期タイムスタンプを取得
2. マルチバックエンド同期（複数バックエンドが設定されている場合）:
   - 各同期バックエンドに last_sync_time 以降の変更をクエリ
   - 比較、競合解決（data-model.md を参照）
   - プライマリバックエンドに変更を適用
   - 同期バックエンドにプッシュ
3. インボックス（未処理アイテム）を読取 — プライマリバックエンド経由
4. _meta/STATUS.md（グローバルステータス）を読取
5. _meta/lint-state.md を読取（巡検が必要か確認: 最終実行から4時間以上）
6. ReadProjectContext(バインドされたプロジェクト) — タスク、決定、ジャーナル
7. user-patterns.md を読取
8. グローバル概観: List Project + List Area（タイトル + ステータスのみ）
9. lint-state.md が4時間以上を示す場合 → 軽量御史台巡検をトリガー
10. プラットフォーム検出 + バージョンチェック
```

### ラップアップモード（プロセス終了時）

```
1. Save Decision / Save Task / Save JournalEntry → プライマリバックエンド経由
2. _meta/STATUS.md を更新
3. _meta/lint-state.md を更新
4. user-patterns.md を更新
5. コミット（GitHubバックエンドの場合: git add + commit + push）
6. すべての同期バックエンドに同期（出力 + STATUS を書き込む）
7. いずれかのバックエンドが失敗 → _meta/sync-log.md にログ、ブロックしない
8. _meta/config.md の last_sync_time を更新
```

### レビューモード

```
1. _meta/STATUS.md からグローバル状態を読取
2. List Task（全プロジェクト）→ 完了率を計算
3. List Area → 目標を読取
4. List JournalEntry（最近）→ ジャーナルと巡検レポート
5. 結果からメトリクスダッシュボードを計算
```

## 諫官のデータ取得

```
1. user-patterns.md を読取
2. List JournalEntry (type: remonstrator, limit: 3) → 直近3件のレポート
3. List Decision (limit: 5) → 最近の決定
4. List Task → 完了率を計算
```

## デグレードルール

- プライマリバックエンドに到達不能 → 「⚠️ プライマリバックエンド利用不可」と注記
- 同期バックエンドに到達不能 → ⚠️ と注記、ログ、次のセッションでリトライ
- すべてのバックエンドが利用不可 → 通常通り動作、出力は会話に表示されるが永続化されない
