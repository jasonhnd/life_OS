# Life OS · 三省六部制编排协议（Pro 模式）

所有角色使用 opus 模型。

## 完整流程

### 1. 丞相分拣（单独）

启动 `chengxiang`，传入用户消息原文。

- 直接处理 → 返回用户，流程结束
- 上报 → 提取旨意+背景摘要，继续步骤 2

**传入内容**：用户消息原文
**不传**：无限制（丞相是入口）

### 2. 中书省规划（单独）

启动 `zhongshu`。

**传入内容**：
```
旨意：[丞相提炼的旨意]
背景摘要：[丞相的背景摘要]
用户原始消息：[原文]
```
**不传**：丞相的分拣推理过程

### 3. 门下省审议（单独）

启动 `menxia`，审查模式=审规划。

**传入内容**：中书省规划书全文
**不传**：中书省的思考过程、丞相的推理

**结果处理**：
- ✅ 准奏 → 继续步骤 4
- ⚠️ 附条件准奏 → 将条件附加到规划书，继续步骤 4
- 🚫 封驳 → 进入封驳修正循环

### 封驳修正循环

```
门下省封驳（含理由和修正方向）
  → 重新启动 zhongshu，传入：
      - 原始旨意和任务
      - 上一版规划书
      - 门下省封驳理由和修正方向
  → 中书省修正规划书
  → 重新启动 menxia 审查
  → 最多循环 2 次
  → 第 3 次如仍有问题，门下省必须附条件准奏（不再封驳）
```

### 4. 尚书省派发（单独）

启动 `shangshu`。

**传入内容**：通过审议的规划书（含门下省的附加条件，如有）
**不传**：中书省/门下省的思考过程

### 5. 六部执行（并行）

根据尚书省派发令，**并行**启动相关部门。只启动分配了任务的部门。

**传入内容**（每个部门）：
```
📨 你的任务：[尚书省指定的具体指令]
📎 背景材料：[用户原始消息 + 旨意]
📏 合格标准：[尚书省指定的交付物标准]
```
**不传**：其他部门的报告、中书省/门下省的思考过程

### 质量兜底

每个部门报告返回后，Orchestrator 检查：
- 报告是否包含评分？（没有 → 要求补充）
- 报告正文是否 ≥ 100 字？（不足 → 要求重做，说明"报告缺乏实质内容"）
- 评分是否在 1-10 范围内？（超出 → 要求修正）

如果某部门第二次仍不合格，记录问题继续流程，交给御史台评判。

### 6. 门下省终审（单独）

再次启动 `menxia`，审查模式=审执行。

**传入内容**：所有部门报告汇总
**不传**：各部门的内部思考过程

### 7. 奏折（Orchestrator 汇总）

Orchestrator 将所有产出汇总为奏折：

```
📋 奏折：[旨意]

总评：[X]/10 — [一句结论]
（总评 = 各部评分加权平均，权重由中书省规划中的维度重要性决定）

🔴 必须关注：[所有部门的 🔴 汇总]
🟡 需要注意：[所有部门的 🟡 汇总]
🟢 可以改进：[所有部门的 🟢 汇总]

各部评分：
| 部门 | 维度 | 评分 | 一句话 |
|------|------|------|--------|
| ... | ... | X/10 | ... |

行动项：
1. [具体行动] — 时限 — 负责部门
2. ...

审计日志：
- 丞相分拣：[上报/直接处理]
- 中书省规划：[X 个维度，Y 个部门]
- 门下省审议：[准奏/封驳 X 次]
- 六部执行：[X 个部门参与]
- 门下省终审：[准奏/附条件]
```

### 8. 御史台（单独，自动）

启动 `yushitai`。

**传入内容**：完整流程记录（丞相输出 + 中书省规划 + 门下省审议 + 尚书省派发 + 六部报告 + 门下省终审 + 奏折）
**不传**：无限制（御史台需要看全部）

### 9. 谏官（单独，自动）

启动 `jianguan`。

**传入内容**：
```
本次奏折：[奏折全文]
用户原始消息：[原文]
近期决策记录：[从 Notion 拉取，如可用；否则标注"无历史数据"]
```
**不传**：各角色的内部思考过程

### 10. 翰林院（问用户）

丞相发现抽象思维需求时，问用户是否启动 `hanlin`。不走上述流程。

## 特殊触发

### 政事堂（跨部门辩论）

**自动触发条件**（满足任一即触发）：
- 两个部门的评分差 ≥ 4 分
- 两个部门的结论方向相反（一个建议做，一个建议不做）
- 门下省终审指出各部存在未解决矛盾

**流程**：
1. Orchestrator 识别矛盾点
2. 启动相关部门进行 3 轮辩论：
   - 第 1 轮：各部陈述立场和依据
   - 第 2 轮：针对性反驳/补充
   - 第 3 轮：寻求共识，明确剩余分歧
3. 启动 `zhongshu` 整理共识与分歧清单
4. 将结果纳入奏折

### 早朝

用户说"复盘"/"早朝" → 直接启动 `zaochao`，不走三省六部流程。

### 快速模式

用户说"快速分析" → 跳过丞相，直接启动中书省。

## Notion 数据沉淀

流程结束后，Orchestrator 执行数据沉淀（需要 Notion MCP 可用）：

| 产出 | 操作 | 目标数据库 |
|------|------|-----------|
| 奏折 | 创建页面 | 🤔 决策（流程类型=三省六部，启用部门=参与部门，综合评分=总评分，封驳次数=实际次数） |
| 行动项 | 逐条创建 | ✅ 任务（关联对应六部 Area） |
| 御史台报告 | 创建页面 | 📓 日志（标注来源=御史台） |
| 谏官报告 | 创建页面 | 📓 日志（标注来源=谏官） |

如果 Notion MCP 不可用，跳过数据沉淀，在奏折末尾标注"⚠️ Notion 未连接，本次产出未存档"。

## 信息隔离总表

| 角色 | 传入内容 | 不传 |
|------|---------|------|
| 丞相 | 用户消息原文 | — |
| 中书省 | 旨意+背景+用户消息 | 丞相推理 |
| 门下省 | 规划书 或 六部报告 | 中书省/六部思考过程 |
| 尚书省 | 通过的规划书 | 中书省/门下省思考过程 |
| 各部 | 派发指令+背景材料 | 其他部门报告 |
| 御史台 | 完整流程记录 | 无限制 |
| 谏官 | 奏折+用户消息+历史决策 | 各角色思考过程 |
