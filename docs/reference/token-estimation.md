# Token 消耗拆解

Life OS 的 Token 消耗取决于两个变量：**场景复杂度**（激活几个部门）和**流程事件**（是否触发 Veto / 政事堂）。本文按场景拆解。

> Claude Max / Pro 订阅用户不按 Token 计费。以下分析面向 API 用户和成本敏感用户。

---

## 基本概念

### 每个角色的 Token 构成

每次角色调用 = **输入 Token**（系统指令 + 上下文）+ **输出 Token**（角色产出）

| 角色 | 系统指令 | 典型输入上下文 | 典型输出 | 每次调用总计 |
|------|---------|---------------|---------|------------|
| 丞相（Prime Minister） | ~500 | 用户消息 ~200 | 呈递 ~150 / 直接处理 ~300 | ~700-1,000 |
| 中书省（Secretariat） | ~400 | 诏令 + 背景 + 用户消息 ~500 | 方案 ~800 | ~1,700 |
| 门下省（Chancellery） | ~450 | 方案/报告 ~800-2,000 | 审查结果 ~600 | ~1,850-3,050 |
| 尚书省（Dept. of State Affairs） | ~350 | 方案 ~800 | 派遣令 ~600 | ~1,750 |
| 六部（每部） | ~350 | 指令 + 背景 ~500 | 评估报告 ~800 | ~1,650 |
| 御史台（Censorate） | ~350 | 完整流程记录 ~3,000 | 绩效评估 ~300 | ~3,650 |
| 谏官（Remonstrator） | ~300 | 奏折 + 用户消息 ~1,500 | 进谏 ~300 | ~2,100 |
| 早朝官（Morning Court） | ~350 | Notion 数据 ~1,000 | 简报 ~600 | ~1,950（仅会话开始） |
| 起居郎（Court Diarist） | ~300 | 会话上下文 ~1,500 | 日志 + wiki 抽取 ~500 | ~2,300 |
| 翰林院（Hanlin Academy） | ~250 | 用户消息 ~300 | 对话回复 ~500 | ~1,050 |

> 以上为估算。实际消耗随用户输入长度和模型输出详细程度变化。

## 按使用场景拆解

### 场景 A：丞相直接处理

```
用户 → 丞相 → 回复
```

最轻量使用模式。丞相判断简单、直接处理。

| | Tokens |
|--|--------|
| 调用次数 | 1（router agent） |
| Tokens | ~700-1,000 |
| 成本 | ~$0.01-0.02 |

**适用场景**：闲聊、翻译、查询、记录、单步任务。

---

### 场景 A2：快速分析（🏃 Express）

```
丞相 → 1-2 个部门 → 奏折总结
```

当请求不涉及决策（只是分析、研究、规划），丞相跳过完整三省流程，直接启动 1-2 个相关部门。

| | Tokens |
|--|--------|
| Tokens | ~15,000-20,000 |
| 成本 | ~$0.35-0.50 |

**适用场景**：研究请求、规划任务、市场分析、信息收集。

---

### 场景 B：精简流程（3 个部门）

```
丞相 → 中书省 → 门下省 → 尚书省 → 3 个部门 → 门下省终审 → 奏折 → 御史台 → 谏官
```

典型场景：大额购买（户部 + 兵部 + 刑部）。

| 步骤 | Tokens |
|------|--------|
| 丞相 | ~1,000 |
| 中书省 | ~1,700 |
| 门下省方案审查 | ~1,850 |
| 尚书省 | ~1,750 |
| 3 个部门 | ~4,950（3×1,650，并行） |
| 门下省终审 | ~2,450 |
| 丞相奏折总结 | ~2,000 |
| 御史台 | ~3,650 |
| 谏官 | ~2,100 |
| **总计** | **~21,000-23,000** |
| **成本** | **~$0.50-0.60** |

---

### 场景 C：标准流程（4 个部门）

```
丞相 → 中书省 → 门下省 → 尚书省 → 4 个部门 → 门下省终审 → 奏折 → 御史台 → 谏官
```

典型场景：投资决策（户部 + 兵部 + 刑部 + 吏部）、健康管理（工部 + 兵部 + 户部 + 刑部）。

| | Tokens |
|--|--------|
| 总计 | ~25,000-28,000 |
| 成本 | ~$0.60-0.75 |

---

### 场景 D：完整流程（6 个部门）

```
丞相 → 中书省 → 门下省 → 尚书省 → 6 个部门 → 门下省终审 → 奏折 → 御史台 → 谏官
```

典型场景：职业转换、搬家、年度目标、创业决策。

| | Tokens |
|--|--------|
| 总计 | ~35,000-40,000 |
| 成本 | ~$0.80-1.20 |

---

### 场景 E：完整流程 + Veto

```
丞相 → 中书省 → 门下省（Veto）→ 中书省（修正）→ 门下省（通过）→ 尚书省 → 6 个部门 → ...
```

每次 Veto 增加一轮 中书省 + 门下省。

| | Tokens |
|--|--------|
| 每次 Veto 额外消耗 | +5,000-6,000 |
| 总计（6 部门 + 1 Veto） | ~40,000-46,000 |
| 成本 | ~$1.00-1.40 |

**最多 2 次 Veto**。若 2 次 Veto：

| | Tokens |
|--|--------|
| 总计 | ~45,000-52,000 |
| 成本 | ~$1.20-1.60 |

---

### 场景 F：完整流程 + Veto + 政事堂

```
... → 6 个部门 → 门下省终审（发现矛盾）→ 政事堂（3 轮辩论）→ 奏折 → 御史台 → 谏官
```

政事堂在部门结论相互矛盾时触发；相关部门进行 3 轮辩论。假设 3 个部门参与。

| | Tokens |
|--|--------|
| 政事堂额外消耗 | +8,000-10,000 |
| 总计（6 部门 + Veto + 政事堂） | ~48,000-62,000 |
| 成本 | ~$1.50-2.00 |

这是单个流程的理论最大消耗。

---

### 场景 G：早朝复盘

```
早朝官（独立流程，不经过三省六部）
```

| | Tokens |
|--|--------|
| 总计 | ~1,500-2,500 |
| 成本 | ~$0.03-0.05 |

早朝官是会话开始时的单次调用。若连接 Notion，读取数据会增加输入 Token。

---

### 场景 H：翰林院深度对话

```
丞相（判断）→ 翰林院（多轮对话）
```

翰林院是多轮对话，每轮约 1,000-1,500 Token。

| | 3 轮 | 5 轮 | 10 轮 |
|--|------|------|-------|
| 总计 | ~4,000-5,000 | ~7,000-9,000 | ~15,000-20,000 |
| 成本 | ~$0.08-0.12 | ~$0.15-0.22 | ~$0.35-0.50 |

## 节省 Token 的策略

| 策略 | 效果 | 注意 |
|------|------|------|
| **简单事项不开朝会** | 节省 90%+ | 丞相直接处理，~1k Token |
| **使用快速分析路径** | 节省 60-75% | 跳过三省，直接启动 1-2 个部门 |
| **Quick 模式** | 节省 ~1k | 跳过丞相分流，直接进入中书省 |
| **按需激活部门** | 节省 20-50% | 大额购买 3 部门 vs 全部 6 部门，差 2 倍 |
| **用早朝官做日常复盘** | 节省 80%+ | 单次调用 ~2k，不走完整流程 |
| **限制翰林院轮数** | 因情况而异 | 每增加一轮 +1-2k |

## API 定价参考

以 Claude Sonnet 4.5 API 定价为例（2026 年 4 月）：

| | 价格 |
|--|------|
| 输入 | $3 / 1M tokens |
| 输出 | $15 / 1M tokens |

Pro 模式使用 Opus 4.5：

| | 价格 |
|--|------|
| 输入 | $15 / 1M tokens |
| 输出 | $75 / 1M tokens |

> 实际成本取决于输入/输出比。Life OS 输出相对较多（部门报告 + 奏折），输出 Token 占总成本的 60-70%。

## 三个主题的成本差异

主题切换本身不额外消耗 Token——主题文件（~60 行）在会话开始时读一次即可，之后的调用都使用相同的角色 ID。因此：

- Roman Republic、US Government、C-Suite 三个英文主题成本**基本相同**
- 三省六部 vs 中国政府 vs 公司部门三个中文主题成本**基本相同**
- 明治政府 vs 霞が関 vs 企業三个日文主题成本**基本相同**
- **中 / 日主题略贵于英文**——中日字符的 Token 密度通常是英文的 1.5-2 倍（因为 UTF-8 编码），但对整体场景影响在 20% 以内

## 如何检查实际用量

- **Claude Code**：`/cost` 命令查看累积用量
- **Gemini CLI**：`gemini usage` 查看当前账单状态
- **Codex CLI**：通过 OpenAI dashboard 查看 API 用量
- **API 直接使用**：通过 Anthropic / OpenAI / Google AI Studio 的使用报表

记录你自己最常用的场景类型，就能更准确预估自己的月度成本。
