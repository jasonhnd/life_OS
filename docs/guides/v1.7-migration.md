# v1.7 升级指南 · 从 v1.6.2a 到 v1.7

> Life OS v1.7 是一次**大版本升级**：新增 Cortex 认知前置层、Hermes 风格的本地执行层（shell hooks + Python tools）、方法库、评估历史自反馈闭环。本指南面向已经在使用 v1.6.2a 的用户，讲清楚"升级时我需要做什么，升级后我会看到什么"。

---

## 升级前必读：三个关键前提

1. **Markdown-first 保留**：所有真理源仍然是 `second-brain/**/*.md`。Cortex 新加的东西（概念图、session 索引、快照）也是 markdown。**随时可退回 v1.6.2a**——你的数据不会被锁进数据库。
2. **用户决策已锁定（见 `references/cortex-spec.md` 和 `docs/architecture/roadmap.md`）**：
   - 所有工具从 Claude Code Bash 本地触发，**不做 Telegram bot / 独立常驻 agent / 远程 cron / GitHub Actions**
   - 不做 embedding / 向量 DB，语义检索用 LLM judgment
   - 不设 "3 个月到期必砍" 的硬性 kill criteria，改为 AUDITOR eval-history 持续监测
3. **spec 阶段**：v1.7 所有 spec 已写完；程序实现还未开始。本指南描述的是 "v1.7 正式发布后" 的体验——**你目前运行的还是 v1.6.2a**。

## 二、升级步骤（约 5 分钟）

### 1. 拉最新 Life OS 代码

```bash
cd ~/.claude/skills/life_OS  # 或 Life OS 的 clone 路径
git fetch origin
git checkout v1.7.0   # 正式发布时的 tag
```

### 2. 同步依赖

```bash
cd ~/.claude/skills/life_OS
uv sync                # 读 pyproject.toml + uv.lock，创建 .venv
```

验证：
```bash
uv run life-os-tool --help
# 应看到 16 个子命令：stats / rebuild-session-index / rebuild-concept-index /
#                     backup / extract / seed-concepts / reindex / reconcile /
#                     search / daily-briefing / research / export / sync-notion /
#                     embed / migrate / seed
```

### 3. 安装新版 shell hooks

```bash
bash scripts/setup-hooks.sh
# v1.7 新增 5 个 hook：pre-prompt-guard / post-response-verify /
#                      pre-write-scan / stop-session-verify / pre-read-allowlist
# 已有的 lifeos-version-check.sh（v1.6.2 基线）保留，不冲突
```

验证：
```bash
jq '.hooks | keys' ~/.claude/settings.json
# ["PreToolUse", "PostToolUse", "Stop", "UserPromptSubmit"]
```

### 4. 跑迁移脚本（关键步骤）

```bash
cd ~/second-brain

# 推荐：通过统一 CLI 入口（tools/cli.py §2 单二进制）
uv run --directory ~/.claude/skills/life_OS life-os-tool migrate \
  --from v1.6.2a --to v1.7

# 等价的直接调用（migrate.py 模块的 docstring 形式）
uv run --directory ~/.claude/skills/life_OS tools/migrate.py \
  --from v1.6.2a --to v1.7
```

migrate.py 接受的完整 flag 列表：
- `--from {version}`（必填，例如 `v1.6.2a`）
- `--to {version}`（必填，例如 `v1.7`）
- `--dry-run`（只打印计划不落盘）
- `--root PATH`（second-brain 根，默认 cwd；从 `~/second-brain` 跑时可省略）
- `--verbose` / `-v`（stderr 输出进度）

先用 `--dry-run` 看一遍计划，确认无问题再去掉 flag 正式跑。

**migrate.py 做什么**（详见 `references/tools-spec.md §6.7`）：
- 扫最近 **3 个月** 的 `_meta/journal/*.md`（用户决策 #7 的窗口，WINDOW_DAYS=90）
- 合成 `_meta/sessions/{session_id}.md`（每个历史 session 一个摘要，session_id 形如 `claude-YYYYMMDD-HHMM`）
- 编译 `_meta/sessions/INDEX.md`（hippocampus 的 Wave 1 扫描对象）
- 提取候选概念：activation_count ≥ 3 的进 `_meta/concepts/{domain}/{cid}.md`（status: confirmed 或 ≥10 为 canonical），低于 3 的不输出；tentative 走 `_meta/concepts/_tentative/{domain}/{cid}.md`
- 编译 `_meta/concepts/INDEX.md` + `_meta/concepts/SYNAPSES-INDEX.md`
- 从 SOUL delta 合成历史 snapshot 到 `_meta/snapshots/soul/{YYYY-MM-DD-HHMM}.md`（frontmatter 标 `provenance: synthetic`）
- 从 journal 语言模式抽取 top-5 候选 method 到 `_meta/methods/_tentative/{slug}.md`（status: tentative，confidence: 0.3）
- **不**回填 `_meta/eval-history/`（那是 v1.7 新的自评闭环，v1.7 day-one 开始累积）
- 写 `_meta/cortex/bootstrap-status.md` 记录迁移结果

Migration 是**幂等**的——重跑会覆盖编译产物，但不会重复创建概念/session 摘要。

### 5. 首次 "上朝" 验证

打开 Claude Code session，说"上朝"。你会看到：

```

🌅 Trigger: 上朝 → Launching retrospective subagent  ← pre-prompt-guard hook

[retrospective Mode 0 完整 briefing...]

```

如果 `bootstrap-status.md` 说一切正常、briefing 里能看到从历史数据合成出来的概念和 session，**升级成功**。

## 三、升级后的新体验

### 1. 每次用户消息都会触发 Cortex Pre-Router（Step 0.5）

用户说话 → **hippocampus** 扫 `_meta/sessions/INDEX.md` 找相关历史 → **concept lookup** 在 `_meta/concepts/` 做扩散激活 → **SOUL 维度检查** 复用 RETROSPECTIVE 的 Health Report → **GWT arbitrator** 用 salience 公式仲裁，输出 top-5 信号 → **ROUTER 收到带认知标注的输入**。

你在 ROUTER 的分诊里会看到类似：

```
[COGNITIVE CONTEXT — reference only]
⚠️ SOUL CONFLICT: "risk-tolerance" (core, confidence 0.82) 与当前请求冲突
📎 相关历史: S:claude-20260315-1420 (score 8.2) — 3 个月前决定过类似问题
🧬 活跃概念: C:finance:company-a-holding (canonical, weight 47)
💡 观察: 该主题过去 30 天出现 3 次（可能 repeated-decision）
[END COGNITIVE CONTEXT]

你的原始消息: {原话}
```

**失败降级**：如果任何子 agent 超时或失败，orchestrator 自动退回到 v1.6.2a 行为（ROUTER 收原始消息，无认知标注），不 block session。

### 2. Full Deliberation 末尾会多一步 Narrator Validation（Step 7.5）

ROUTER 产出 Summary Report 后，narrator-validator（Sonnet Claude Code subagent，**不是外部 Haiku API**）会扫一遍，每一个 substantive claim 必须带 `signal_id` citation。格式：

```
根据你过去在类似决策中偏保守的模式 [S:claude-20260315-1420][D:governance-score-4]，
这次 FINANCE 的建议也建议走保守路线 [D:finance-score-7]。
```

如果 narrator-validator 发现无 citation 的 substantive claim，会让 narrator 重写（最多 3 轮，超时或 3 次失败后降级为无 citation 的 raw Summary Report）。

你想核实任何一句话时，可以问 "这句话背后的 signal 是什么？" —— 系统会展开 trace 给你看（详见 `references/narrator-spec.md §15` 和 §17 Trace UX）。

### 3. 退朝时 archiver Phase 2 做的事更多了

v1.6.2a 的 archiver Phase 2：wiki 自动写入 + SOUL 自动写入 + snapshot dump。

v1.7 新增：
- **概念抽取 + Hebbian 更新**：session 里出现的实体/概念 → 写 `_meta/concepts/_tentative/`；同时共激活的概念对 → 边权重 +1
- **方法候选检测**：session 语言模式识别"可复用工作流" → 写 `_meta/methods/_tentative/`
- **SYNAPSES-INDEX 重建**：概念图的反向索引重新编译
- **decay pass**：按 permanence tier 衰减边权重 + 降级 canonical 概念（详见 `references/cortex-spec.md §Design Principles → Three-tier undo`）
- **session summary 写入 `_meta/sessions/{session_id}.md`**：次日 "上朝" 时 retrospective 会扫它
- **eval-history 写入 `_meta/eval-history/{date}-{project}.md`**：AUDITOR 的自评

这些都在 archiver 一次 invocation 里完成，不会拆分到多个 session。

### 4. 每次 "上朝" 会看到系统性问题检测

retrospective Mode 0 读最近 10 条 eval-history 做模式检测：

```
⚠️ 系统性问题检测：
- adjourn 完整度连续 3 次 ≤ 6/10 → 检查 archiver 子 agent 启动
- wiki 抽取质量从 4/15 起下降 → 本次重点关注
- 同类 violation 30 天内发生 5 次 → 升级为 user-patterns 追踪
```

如果没问题，这个块直接省略（不是 "一切正常" 的占位符）。

### 5. 可选：每日本地 briefing

如果你配了 launchd 或 crontab：

```bash
# ~/Library/LaunchAgents/com.lifeos.daily-briefing.plist
# 06:00 每天跑 daily_briefing.py → 写入 _meta/briefings/{date}.md
# 同时可选触发 macOS 原生通知（osascript / terminal-notifier）
```

下次 "上朝" 时 retrospective 会把 briefing 的未读项呈给你。

**注意**：Life OS v1.7 **不做** Telegram bot / Discord / 跨平台消息推送 / 独立常驻 agent。本地通知通过 macOS 原生机制，也只是"提醒你打开 session"的作用，不替代 session。

## 升级后的第一周：日常体验对标

前一节讲"v1.7 新增了哪些机制"，这一节从**用户视角**讲这些机制落到你日常使用时是什么样——每天"上朝 → 决策 → 退朝"的循环里，有哪几个瞬间和 v1.6.2a 不一样，你要留意看什么、做什么。

### 1. 上朝第一刻：先看一段"Cognitive Context"再决定怎么走

v1.6.2a：你说"上朝"或发一条新消息，ROUTER 立刻分诊、直接分派 agent。

v1.7：你发消息后，ROUTER **分诊前**会先插入 Step 0.5——hippocampus 扫 `_meta/sessions/INDEX.md` 捞相关历史、concept lookup 在概念图做扩散激活、SOUL check 看有没有核心维度冲突，gwt-arbitrator 把 3-5 条高 salience 信号合成一段 `[COGNITIVE CONTEXT]` 块（格式见 `references/gwt-spec.md §8`），**贴在你原消息前面**送给 ROUTER。你在 session 日志里会看到它，可以在 ROUTER 分派前纠正（"这条历史其实不相关，别参考"），也可以沉默放行。

示例（对话片段）：

```
你："我在考虑把 Company A 的仓位再加 20%"

[Life OS 在 ROUTER 分派前显示]
[COGNITIVE CONTEXT — reference only]
⚠️ SOUL CONFLICT: "risk-tolerance" (core, confidence 0.82) 正被挑战
📎 相关历史: S:claude-20260315-1420 — 3 个月前你也想加仓，最后决定不加
🧬 活跃概念: C:finance:company-a-holding (canonical, weight 47)
💡 观察: 过去 30 天这个话题出现 3 次（可能 repeated-decision）
[END COGNITIVE CONTEXT]

ROUTER → Full Deliberation（FINANCE + GOVERNANCE + REVIEWER）
```

你如果觉得"3 个月前那次情况不一样，这次不必参考"，直接说一句"那条历史不算数"，下一轮 ROUTER 就会忽略它。这是一次**显式的上下文确认**，不是黑箱。

### 2. 决策过程中：信号打架时系统会说清楚用了谁、忽略了谁

v1.6.2a：多 agent 意见不一致时，REVIEWER 给个合并的 Summary，过程不透明。

v1.7：当 hippocampus 召回的历史、SOUL 的趋势、当前对话里用户说的新事实**互相矛盾**时，gwt-arbitrator 不会"取平均"，而是按 salience 公式排序，top-5 入选，低于 per_signal_floor (0.3) 的被丢弃（见 `references/gwt-spec.md §9`）；narrator 再把入选信号编进 Summary Report 时，**每一句 substantive claim 都带 `[signal_id]` citation**（Step 7.5 强制）。你想知道"某条被忽略的信号去哪了"，直接问："为什么 Summary 没提上周那个 decision？"——ROUTER 识别 trace 请求，narrator trace UX（`references/narrator-spec.md §17`）会回答"salience 0.22 低于阈值被丢弃"或"那是 connective tissue，没有 signal 支撑"。

示例：

```
REVIEWER Summary 片段：
"FINANCE 建议 +10% 仓位 [S:claude-20260420-0930][D:finance-score-7]，
但和你 SOUL 中 'risk-tolerance' 维度 [S:SOUL-dim-risk-tolerance] 相冲突，
建议打折到 +5%。"

你："上周那次加仓讨论呢？为什么没提？"

系统（trace UX 回答，不跑 LLM，只读文件）：
"S:claude-20260414-1510 salience 0.24，低于 floor 0.3 被 arbitrator 丢弃。
原因：recency 高但 concept overlap 低（那次讨论是 Company B 不是 Company A）。"
```

**这是 Gazzaniga 反面**——系统不会编一个自洽故事糊弄你，每个引用都能追溯到具体文件和行号。

### 3. 报告输出：Summary Report 里每句话都可点击回溯

v1.6.2a：REVIEWER 的 Summary 是散文段落，没有 citation。

v1.7：narrator-validator 在 Step 7.5 强制所有 substantive claim 带 `[S:...]` 或 `[D:...]` citation（S = session signal，D = decision score），不带的回炉重写最多 3 轮（3 次失败后降级为无 citation 的 raw report，标记 degraded）。你读 Summary 时遇到不确定的结论，选中那段问"signal 是什么"，trace UX 会给你**源文件路径 + 内容片段**，<2 秒返回（`narrator-spec.md §17 Trace UX` 要求），不再跑 LLM 生成——避免"解释的解释的解释"递归 confabulation。

示例：

```
Summary 原文：
"根据你过去在类似决策中偏保守的模式 [S:claude-20260315-1420][D:governance-score-4]，
这次 FINANCE 的建议也建议走保守路线 [D:finance-score-7]。"

你（鼠标选中或复制）："[S:claude-20260315-1420] 是什么？"

系统：
📄 _meta/sessions/claude-20260315-1420.md （行 12-18）
摘要: "用户考虑提高 Company B 仓位，最终决定维持现状"
key_decisions: ["保持 Company B 仓位不变"]
activated_concepts: [C:finance:risk-tolerance, C:finance:company-b-holding]
```

这条 trace 是纯文件读取，不改变任何状态、不计入 AUDITOR、不写 journal——随便问几次都没副作用。

### 4. 退朝后：系统递上 2-5 个候选"概念"和"方法"等下次确认

v1.6.2a：archiver Phase 2 做 wiki/SOUL auto-write + snapshot，就结束。

v1.7：archiver Phase 2 额外做**概念抽取 + 方法检测**（`cortex-spec.md §Step 10`），把本次 session 里新冒出来的实体写入 `_meta/concepts/_tentative/`、可复用工作流写入 `_meta/methods/_tentative/`——但**不直接 promote**。这些候选在**下次上朝的 retrospective Mode 0** 里会一次性展示，问你：`confirm` / `reject` / `rename`。这是赫布强化（Hebbian learning）的入口——你的 confirm 是突触权重 +1 的触发器；reject 会把候选移到 `_archive/`。

示例（下次上朝时）：

```
🧬 待确认概念（2 个）:
  1. C:finance:company-a-holding-strategy (tentative)
     来源: 过去 3 周 4 次 session 共激活
     [confirm / rename "仓位策略-A" / reject]
  2. C:personal:weekend-review-habit (tentative)
     来源: 连续 2 周五晚 session 提及
     [confirm / rename / reject]

📐 待确认方法（1 个）:
  1. M:finance:triple-check-before-allocate
     来源: archiver 检测到语言模式 "先看 X，再算 Y，最后决定 Z" 出现 3 次
     [confirm / rename / reject]
```

**不要全接受也不要全拒绝**——挑那些你真在用的确认，剩下的 reject 或留着观察。30 天未 confirm 的 tentative 会被 decay pass 自动清理（`cortex-spec.md §Design Principles → Three-tier undo`）。

### 5. 一个月后：AUDITOR eval-history 会主动提示系统性问题

v1.6.2a：AUDITOR 评分写进 journal，但需要你自己翻日志找趋势。

v1.7：AUDITOR 每次 session 写一条 `_meta/eval-history/{date}-{project}.md`，打分 5-7 个维度（见 `cortex-spec.md §Metrics` 和 `hippocampus-spec.md §307`：`cognitive_annotation_quality`、`citation_groundedness`、`adjourn_completeness` 等）。retrospective Mode 0 读**最近 10 条**做模式检测，发现恶化趋势时在上朝 briefing 顶部提示——不需要你主动查。

示例（第 4 周的某次上朝）：

```
⚠️ 系统性问题检测（最近 10 次 session）:
- cognitive_annotation_quality 从 7.2 → 5.1，连续 3 次下滑
  → 原因候选: 你最近 3 次上朝里 Step 0.5 召回的历史里有 30% 被你手动
    标为"不相关"。建议调整 hippocampus 衰减参数:
      _meta/cortex/config.md → decay_curves.fact.half_life_days: 90 → 60
    或把 _meta/sessions/INDEX.md 里 3 个月前的旧 session 归档到 _archive/
- narrator citation_groundedness 稳定 9.4/10 ✅
- adjourn_completeness 稳定 8.7/10 ✅
```

**这就是 Life OS 的"自反馈闭环"**——不是你监督系统，是系统帮你监督它自己。你只需要决定"按建议调参"还是"忽略这次波动"。如果趋势持续恶化，用户决策 #4（见 `roadmap.md`）明确"AUDITOR 持续监测但不预设硬性 kill"——判断权始终在你手上。

## 五、破坏性变更（Breaking Changes）

### 1. Snapshot 格式变更

v1.6.2a 的 snapshot frontmatter 用 `taken_at` 和 `type: soul-snapshot`；v1.7 改成 `captured_at` 和 `dimensions[]` + `tier` 字段（详见 `references/snapshot-spec.md`）。

**影响**：迁移后 **老 snapshot 仍然可读**（retrospective 兼容两种格式），但新 snapshot 用新格式。建议不要回退到 v1.6.2a，否则新 snapshot 会无法被旧版 retrospective 解析。

### 2. SOUL tier 命名

v1.6.2a 用 `Tier 1 / Tier 2 / Tier 3`；v1.7 改成 `core / secondary / emerging / dormant` 4 级。

**影响**：REVIEWER 报告的语言变了（"核心维度"、"活跃价值"），但 SOUL.md 的内容格式不变。用户不需要修改任何已有文件。

### 3. wiki 的 domain 命名

v1.6.2a 示例 domain 用 `tech`；v1.7 用 `technical`（对齐 concept-spec 和 method-library-spec）。

**影响**：如果你的 wiki 用了 `wiki/tech/` 目录，可以保留（不强制改名），但 migration 会建议统一到 `wiki/technical/`。

### 4. 首次启用会有一批候选

migration 跑完，`_meta/concepts/_tentative/` 会有几十个候选概念，`_meta/methods/_tentative/` 会有几个候选方法。retrospective 下次会一次性呈现让你确认（或拒绝）。**不必全部接受** —— 不相关的直接拒绝。

## 六、回退方案

如果你发现 v1.7 不适合，回退到 v1.6.2a：

```bash
cd ~/.claude/skills/life_OS
git checkout v1.6.2a

cd ~/second-brain
# v1.7 新增的目录不会影响 v1.6.2a（v1.6.2a 不读它们）
# 但如果想完全干净：
rm -rf _meta/sessions/ _meta/concepts/ _meta/methods/ _meta/eval-history/ \
       _meta/cortex/ _meta/audit/ _meta/ambiguous_corrections/ _meta/compliance/
# 或 git reset：
git checkout backup-pre-v1.7
```

你的核心数据（SOUL / wiki / projects / decisions / journal）完全不变，因为 v1.7 的 migration **只追加新目录，不修改已有文件**。

## 七、FAQ

**Q: 升级需要联网吗？**
A: 需要一次 `git pull` 和 `uv sync`。之后所有工具本地跑，不需要联网（Claude Code 本身需要联网调用 LLM，但那是已有的）。

**Q: 我还能不能离线用？**
A: 能。v1.7 和 v1.6.2a 一样，所有数据本地 markdown，断网也能看/编辑/git commit（只是不能跑 Claude Code 的 LLM 部分）。

**Q: 迁移会动我的 SOUL.md / wiki/ / projects/ 吗？**
A: **不动**。migration 只写新目录。SOUL.md 被读取作为 snapshot 合成的输入，但不会被修改。

**Q: 迁移要多长时间？**
A: 3 个月 journal 数据约 60 秒（tools-spec §6.7）。`--dry-run` 先跑一遍看计划。

**Q: 迁移中断会怎样？**
A: migration 是幂等的。中断后重跑即可——它会跳过已处理的文件，继续没处理完的。

**Q: v1.7 的 Claude API 成本会涨吗？**
A: 每次用户消息多一次 Step 0.5（hippocampus + concept lookup + GWT），粗估增加 5-10k tokens/message（Opus），成本约 $0.05-0.10/session。narrator validator 在 Full Deliberation 末尾加一次 Sonnet call。总体 v1.7 比 v1.6.2a 贵约 30-50%。这是用户已接受的代价（用户决策 #8 把成本 scope 定在 $0.2-0.5/month）。

**Q: 我之前的 session journal 会被扔掉吗？**
A: 不会。`_meta/journal/` 保留原样。v1.7 的 `_meta/sessions/` 是**并存**的摘要索引，不是 journal 的替代品。journal 继续是"每次 session 的详细报告"，sessions 是"为 hippocampus 检索服务的精简摘要"。

**Q: 能不能只启用 Cortex 不启用执行层，或反之？**
A: 能。编辑 `_meta/cortex/config.md` 的 feature switches（详见 `references/cortex-spec.md §Cortex Runtime Files`）。把 `cortex_enabled: false` 就完全退到 v1.6.2a 认知行为，但保留 Python tools。

## 八、下一步

升级完跑几天 v1.7 session 后，关注这些指标（retrospective 会自动呈现）：

1. **hippocampus 命中率** — 下游 agent 真的引用了 retrieved sessions 吗？目标 >20%
2. **cognitive_annotation_quality** — eval-history 里这个分数持续 ≥ 6/10？
3. **citation_groundedness** — narrator citation 能不能真实映射到 signal？目标 ≥ 9/10
