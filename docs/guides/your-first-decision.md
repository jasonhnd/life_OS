# 你的第一个决策 · 端到端走一遍

这篇文档陪你走完第一次完整决策流程。你会看到从打字问个问题开始，一直到系统把奏折写到硬盘上为止，中间多个岗位具体在做什么。

读完这篇，你就知道这套系统不是玄学，每一步都有人干活、每一步都留下痕迹。

---

## 开始前你需要什么

- 已经跑过一次 `开始` 或 `上朝`（系统已经绑定到某个项目）
- 一个真正纠结的问题，最好是涉及钱、时间、关系、或者不可逆的那种

如果你的问题是"怎么关闭通知"、"怎么把 PDF 转 Word"——这不是本文要走的流程。系统会直接回答你，不会上朝议事。

---

## 示例场景

本文用下面这个真实场景贯穿全流程：

> 用户：我在东京做了 3 年产品经理，年薪 900 万日元。最近有一个硅谷初创公司给我 offer，股票 + 15 万美元基础工资，要我搬到美国。我该去吗？

这是一个典型的"上朝议事"问题：涉及钱、职业、家庭、风险，所有六部都要开口。

---

## 第 1 步 · 打字

你不需要背任何口令。直接说你的困惑：

```
我该接这个硅谷 offer 吗
```

系统判断这不是简单问答，也不是"给我做个学习计划"那种可以走快车道的事。这是决策。

## 第 2 步 · 丞相（ROUTER）的意图澄清

丞相不会一听到"offer"就立刻叫六部上朝。规矩是：**至少 2-3 轮澄清再升堂**。

第一轮——丞相先复述一遍，确认理解没跑偏：

```
🏛️ 丞相

让我先确认一下——你现在的真实纠结，是"去还是不去美国"，
还是"硅谷这个特定 offer 值不值"？这两个问题答案可能不一样。
```

你回答："是去还是不去美国，已经想了半年了。"

第二轮——丞相切中要害。它知道问题类型是"决策"，所以要问清标准和约束：

```
🏛️ 丞相

那我想问一个关键的：你最担心的是哪一个？
a) 钱不够稳，怕踩雷
b) 家人反对或跟不过去
c) 在新环境能不能立足
d) 其他

这个决定了六部重点查哪个方向。
```

你回答："主要是 b 和 c。钱我算过能撑，但老婆刚有新工作，我父母身体也开始出状况。"

第三轮——丞相确认约束条件：

```
🏛️ 丞相

理解了。最后问一个：这个 offer 有没有 deadline？
以及你和老婆真的谈过吗，还是你自己想的她会支持？
```

你回答："Deadline 是下周三。老婆知道有这个 offer 但没具体谈。"

## 第 3 步 · 丞相升堂（生成奏章）

澄清够了，丞相写升堂文书：

```
🏛️ 丞相 · 升堂

📂 Scope: projects/career-transition
📋 Subject: 硅谷 offer 去留决策
📌 Type: 职业
💡 Suggested Domains: 户部(财务)、吏部(人事)、兵部(执行)、刑部(风控)、工部(基建)
📝 Background Summary:
东京产品经理 3 年，年薪 900 万。硅谷初创 offer，15 万美元 + 股票。
用户纠结点：家庭（妻子新工作、父母健康）+ 在新环境立足。
Deadline：下周三。妻子未深谈。

🔍 Intent Clarification Notes:
用户已自行排除了"钱是否撑得住"这个维度，但未深谈家庭。
这是"silent dimension"——吏部重点审。
```

然后升堂。

## 第 4 步 · 中书省（PLANNER）出规划文书

中书省**看不到**丞相的澄清过程——信息隔离防止偏见串。它只收到 Subject + 背景。

```
📜 中书省 · 规划文书
Subject: 硅谷 offer 去留决策
Intent: 在保留家庭稳定的前提下，判断这个 offer 是否值得

1. 财务可持续性 → 户部 — 24 月现金流 + RSU 日程
2. 家庭影响评估 → 吏部 + 工部 — 同行/分居/放弃 3 方案
3. 职业立足评估 → 兵部 + 礼部 — 6/12/24 月里程碑
4. 法律与移民风控 → 刑部 — H1B + RSU cliff + 退出路径
5. 健康与基建 → 工部 — 住房 + 医疗 + 远程照护 + 3 条红线

⚠️ Risk Warning: "妻子未深谈"是最大盲点
📋 户部、刑部、工部并行；吏部需先与兵部同步
```

## 第 5 步 · 门下省（REVIEWER）的第一次封驳

关键——REVIEWER 不是盖章机器。它拿到规划文书就开始找毛病。

```
🔍 门下省 · 审议意见

❓ 意图审视:
- "已排除钱是否撑得住"——真的排除了吗？没看到具体测算，用户说"算过"
  不等于经得起户部推敲
- 最大的盲点：妻子。规划里虽然列了吏部评估家庭影响，但顺序错了——
  你还没问过她就让六部评估，这是在"为她决定"

🔎 情感维度审查:
- 情绪信号: deadline 下周三 → 时间压力下的决定容易失真
- 10/10/10 Rule:
  10 分钟后: 这个 offer 的新鲜感会不会让你高估风险承受？
  10 月后: 如果去了且失败，妻子怎么想？如果没去且后悔，怎么自处？
  10 年后: 这个决定会决定你"是谁"吗？还是只是一次工作变动？
- 红色信号: 父母身体"开始出状况"——这是新增的信息，未在户部议程中

🔮 SOUL Reference:
SOUL.md 尚未建立。本次决策可能开启维度：
「决策时对家庭意见的权重」——当前证据：用户提到担心老婆但没深谈

🚫 封驳
- 失败维度: 意图澄清不彻底
- 核心问题: 用户让六部帮他"想清楚"，但还没跟妻子谈过
- 修订方向: 先让丞相再追一次：你打算什么时候跟妻子谈？
  谈完之前，这个议程不应进入执行。或者把议程改为
  "如何让这场对话有结构地发生"
- 缺失信息: 妻子的立场、父母健康的具体情况
```

## 第 6 步 · 回到中书省（封驳循环）

中书省重写规划文书。这次新增：

```
📜 规划文书 v2
Intent: 先结构化夫妻对话，再基于对话结果评估 offer

0. 前置任务: 90 分钟夫妻对话（议程执行前必完成）
1. 对话框架 → 礼部 + 兵部 (Assist)
2-5. 户部/刑部/工部/吏部/兵部 — 同 v1，但结果取决于前置任务
```

门下省审议：`✅ Approved（条件：前置任务必成，若妻子反对议程终止）`

## 第 7 步 · 尚书省（DISPATCHER）派遣令

```
📨 派遣令
并行: 礼部 / 户部 / 刑部 / 工部
串行: 吏部（待工部完）| 兵部
```

## 第 8 步 · 六部逐一上报

**关键规矩**：每个部报告进来，丞相必须**立刻全文展示给你**，不能攒着、不能压缩。你会一条一条看到。

### 户部报告

```
💰 户部 · 财务评估

🔎 查了 RSU 4 年 vesting，加州税率 13.3%，联邦 24-32%
💭 A 同行 / B 分居 12 月 / C 留东京
🎯 A: +$180k 24 月 | B: +$140k + 关系损耗 | C: 机会成本 $200k/年

评分: 8/10 — 钱不是问题，但 RSU 前 1 年无 vesting，离职成本高
```

### 工部报告

```
🏗️ 工部 · 基建评估

🔎 对比了 Palo Alto 和中央线沿线的住房、医疗、通勤
🎯 住房 3x (¥200k/月 vs $4500/月)
   父母远程照顾难 — 时差 17h，无熟悉医生
   三条红线:
   1. 父母任一需 24h 照顾 → 必须至少 1 人在东京
   2. 湾区住房超 $5500/月 → 现金流吃紧
   3. 连续 2 周 < 6h 睡眠 → 基建崩溃

评分: 6/10 — 远程照护是最大风险
```

### 刑部报告

```
⚖️ 刑部 · 风控评估

🔎 H1B 抽签率 28%，RSU cliff 条款
🎯 硬约束:
1. H1B 不中 → offer 作废
2. 1 年 cliff 前离职 → 股票归零
3. 东京合同竞业？需查

退出路径: 签证 9 月前知结果 / 失业 H1B 60 天宽限

评分: 5/10 — 签证是最大不可控
```

### 吏部报告（组 1 完成后）

```
👥 吏部 · 人事评估

🎯 "妻子有新工作"权重极高——新工作 6-18 月是关键期
三种关系成本:
- 同行但她不愿: 5 年关系损耗 40-60%
- 分居 12 月: 15-25%（跨国夫妻研究）
- 放弃: 你对她的"亏欠感" 6-12 月内抱怨

🔴 红色: 用户还没跟她谈。任何方案在对话前都是假的。

评分: 4/10 (未谈) → 谈完可重评
```

### 兵部、礼部略（内容类似）。

丞相逐条展示。你读到吏部那一段可能已经动摇了——那一刀切得很准。

## 第 9 步 · 门下省最终审议

六部都报完后，门下省再审一次。这次它看的是有没有矛盾：

```
🔍 门下省 · 最终审议

📊 各部得分:
- 户部: 8/10
- 刑部: 5/10
- 工部: 6/10
- 吏部: 4/10
- 兵部: 7/10
- 礼部: 7/10

得分差: 8 - 4 = 4 ≥ 3 → 触发政事堂议政

⚠️ 冲突点:
户部说"钱够"，吏部说"关系不够"——这不是数值冲突，是维度冲突。
不同的人生优先级会给出不同答案。

→ 自动召开政事堂议政
```

## 第 10 步 · 政事堂议政（COUNCIL）

三轮辩论，户部和吏部对拍。

**第一轮**：户部先说"机会难得，18 个月后 RSU 足以覆盖搬家风险"。吏部回："你在算股票价值，但没算妻子的 18 个月"。

**第二轮**：户部让步"那用分居方案"。吏部"分居是缓冲，不是解法——24 个月后还得面对同一个问题"。

**第三轮**：户部"那按用户的 SOUL 来说"，吏部"SOUL 还没建立，这是他第一次重大决策"。

中书省汇总共识与分歧：

```
共识: 没跟妻子谈过之前，所有方案都是空中楼阁
分歧: 假如妻子支持，户部认为值得去，吏部认为风险仍被低估
```

## 第 11 步 · 丞相出奏折

```
📋 奏折：硅谷 offer 去留决策

总评: 5/10 — 不是 offer 值不值的问题，是你还没问过最重要的人

Must Address:
- 下周三之前，先完成结构化夫妻对话（礼部框架）
- 无论对话结果如何，本议程结论才有效

Needs Attention:
- 父母健康的具体医疗需求（缺失信息）
- Offer 的签证路径（刑部未能闭环，需追问 HR）

Room for Improvement:
- 若同行：住房预算红线 $5500/月
- 若分居：明确 12 个月重议日

部门评分:
| 部门 | 维度 | 评分 | 一句话 |
|------|------|------|--------|
| 户部 | 可持续性 | 8 | 钱不是障碍 |
| 刑部 | 风控 | 5 | 签证不可控 |
| 工部 | 基建 | 6 | 远程照护是关键风险 |
| 吏部 | 关系 | 4 | 未谈，不可评 |
| 兵部 | 执行 | 7 | 立足可行 |
| 礼部 | 表达 | 7 | 框架已出 |

Action Items:
1. 本周六晚上 90 分钟夫妻对话，用礼部框架 — 截止本周六 — 礼部
2. 联系 offer HR 确认签证路径 — 截止明天 — 刑部
3. 查父母最近体检记录 — 截止本周三 — 工部
4. 如对话后妻子支持 → 重开议程定"同行/分居"细节 — 视情况
5. 如对话后妻子反对 → 议程转为"如何告知 HR"— 视情况

Audit Log: 封驳 1 次，议政 1 次（户部 vs 吏部）

Operations Report:
- Total Time: 14 分钟
- Model: opus
- Agent Calls: 11
- Vetoes: 1
- COUNCIL: Triggered
```

## 第 12 步 · 御史台（AUDITOR）审查

御史台自动跟着上来，审这个流程本身：

```
🔱 御史台 · 审查报告

流程合规性: ✅
- 澄清 3 轮 √
- 封驳 1 次，未超上限 √
- 六部全报全展示 √
- 政事堂正确触发 √

质量问题:
- 刑部报告有"需追问"未闭环，应在派遣时就告诉用户"HR 信息缺失将影响精度"
- 吏部给了 4/10 但"未谈"——这不是"评不出"而是"评了"，可能误导用户
  建议未来规则：信息不足时应为 N/A 而非 4/10

无违规。
```

## 第 13 步 · 谏官（ADVISOR）

谏官看的不是这个决定，看的是**你这个人**：

```
💬 谏官 · 行为审查

这是你第一次上朝。我注意到几点：
1. "钱我算过能撑"——没给户部过目，单向屏蔽
2. 涉及妻子的决定推到六部之后才想起她——silent dimension 信号
3. Deadline 下周三你周一上朝——留 2 天消化做得好 📈

🔮 SOUL Candidate:
- Dimension: "决策时对家庭意见的权重"
- What IS: 倾向先自己想清楚再与家人沟通
- What SHOULD BE: (留空，等你填)
```

## 第 14 步 · 起居郎（ARCHIVER）归档

你说"退朝"。丞相立刻调起居郎，中间不插话。起居郎跑 4 阶段：

- **Phase 1**：决策、tasks、报告写到 `_meta/outbox/`
- **Phase 2 知识提取**：扫会话找 wiki / SOUL 候选
  - wiki 候选"分居 12 月关系损耗 15-25%"——个人推论非公开研究，证据 1 不够 2，**discard**
  - SOUL 候选"决策时家庭意见权重"——**自动写入** confidence 0.3，What SHOULD BE 留空
- **Phase 3 DREAM**：首次决策，REM 写"基线建立"
- **Phase 4 Sync**：git push + Notion 推送

输出 Completion Checklist + Notion sync 结果后会话结束。

## 快车道 vs 全套

"快速分析" 跳澄清直接走快车道——但**快车道不做决策**，只做研究/分析/规划。

- "帮我做一个 Rust 学习计划" → 快车道
- "分析这份合同的风险点" → 快车道
- "应该接这个 offer 吗" → **必须**走全套

---

## 这套流程为什么这样

- **PLANNER 看不到 ROUTER 推理** — 信息隔离防止偏见串
- **REVIEWER 可以否决两次** — 没有它你会把 ROUTER 假设当事实
- **COUNCIL 靠分差 ≥3 触发** — 自动，不用手动
- **ARCHIVER 是独立 subagent** — 防止流程被压缩

下次决策时这些机制**自动**运行。你只管说出问题。

---

## 常见问题

**Q：封驳后我可以不改吗？** 可以，但丞相会拒绝继续。你要么改要么强行继续（后者被御史台记录）。

**Q：六部报告太长不想全看？** 报告必须全展示（硬规矩）。读完只执行 Action Items 也行，但不能省略展示。

**Q：COUNCIL 议不出结论？** 三轮后中书省写"共识 + 分歧"，你拍板。系统的价值是把分歧显性化。

**Q：下次同样问题再问？** 早朝官晨报会提示"上次说要和妻子谈，谈了吗？"——这是 DREAM action #8 (stale-commitment)。
