## 4. Dependency Graph (which spec depends on which)

```
cortex-spec
  ├── hippocampus-spec          ← retrieval mechanism
  ├── gwt-spec                   ← arbitration mechanism
  ├── narrator-spec              ← grounded generation
  ├── concept-spec               ← synapse + hebbian mechanism
  ├── session-index-spec         ← hippocampus data source
  ├── snapshot-spec              ← SOUL trend data source
  ├── eval-history-spec          ← AUDITOR self-feedback
  └── method-library-spec        ← procedural memory

hooks-spec                       ← Layer 3 enforcement (independent)
tools-spec                       ← Layer 4 batch tools (reads all _meta/ artifacts)

data-model                       ← types consumed by tools-spec dataclasses
data-layer                       ← physical layout, backend adapters
wiki-spec / soul-spec / dream-spec  ← existing specs; 4-tier / positioning / Cortex-integration updates
```

**No cyclic dependencies**. cortex-spec is the root. tools-spec reads all artifact specs but nothing reads tools-spec (tools are consumers, not producers of schema).

### Precedence rules when two specs appear to conflict

The spec package is audited for consistency — contradictions are bugs. But implementers occasionally hit ambiguous cases. Precedence for resolving them:

1. **User decisions (§3) win over any spec wording.** The 16 locked decisions are the root. If a spec contradicts them, fix the spec.
2. **Per-artifact spec wins on its own artifact's schema.** For field definitions of SessionSummary / Concept / SoulSnapshot / EvalEntry / Method, the authoritative spec is the per-artifact spec (session-index-spec, concept-spec, snapshot-spec, eval-history-spec, method-library-spec). `data-model.md` is the short-form summary for tool implementers; if it disagrees with the per-artifact spec, the per-artifact spec wins.
3. **Per-mechanism spec wins on its own mechanism's behavior.** For retrieval algorithm → hippocampus-spec. For arbitration formula → gwt-spec. For citation format → narrator-spec. For concept graph rules → concept-spec.
4. **cortex-spec wins on cross-mechanism integration.** When multiple mechanisms interact (Step 0.5 orchestration, archiver Phase 2 ordering, undo mechanism), cortex-spec is authoritative.
5. **hooks-spec wins on Layer 3 enforcement.** tools-spec wins on Layer 4 Python tools. If one spec describes a behavior also touched by the other, read both but treat the owning spec as canonical.
6. **Architecture docs are non-authoritative.** `docs/architecture/*.md` describe rationale and orientation. If they disagree with any `references/*.md` spec, the spec wins. Architecture docs will be updated to match.
7. **Freshness tiebreaker**: when two authoritative specs disagree despite above rules, the more recently updated file wins (check `git log`). Open a spec-bug issue and sync them.

## 5. Cross-Spec Invariants (must hold across all specs)

Any implementation must satisfy these invariants. If any is violated, a spec is wrong.

### 5.1 Write ownership invariants

- **archiver writes, retrospective reads** — archiver Phase 2 is the only writer of concepts, snapshots, session summaries, method candidates, eval-history entries, decay-log blocks. retrospective Mode 0 compiles INDEX files but never modifies per-session artifacts.
- **ROUTER owns user-facing output** — narrator is part of ROUTER's Step 7.5 composition; no separate `pro/agents/narrator.md` file (`references/narrator-spec.md §6`).
- **Session summary write is Phase 2, not Phase 1** — the summary's frontmatter references Phase 2 outputs (concepts_activated, methods_discovered); Phase 1 only generates the session_id.
- **Concept decay is archiver's write, retrospective's read** — retrospective regenerates `INDEX.md` but does not mutate edge weights or status.

### 5.2 Path / placeholder invariants

- Session index file: `_meta/sessions/INDEX.md` (not `_meta/INDEX.md`)
- Placeholder style: `{project}` / `{area}` / `{session_id}` / `{method_id}` / `{concept_id}` (underscore form, full words; never `{p}` / `{session-id}`)
- Migration log path: `_meta/cortex/bootstrap-status.md` (canonical; `_meta/migration-log-*` is historical and deprecated)

### 5.3 Naming invariants

- **SOUL tier**: `core` / `secondary` / `emerging` / `dormant` (not `Tier 1/2/3`)
- **Salience dimensions**: 4 (`urgency / novelty / relevance / importance`); `emotional_weight` is amygdala-tier, out of scope for v1.7
- **Validator model**: Sonnet Claude Code subagent (not "Haiku-tier", not external API)
- **Domain taxonomy** across wiki/concept/method: `finance / startup / personal / technical / method / relationship / health / legal` plus user-defined
- **Method status enum**: `tentative / confirmed / canonical` (files in `_tentative/` have `status: tentative`, not `candidate`)
- **Concept status enum**: `tentative / confirmed / canonical`

### 5.4 Scope / non-goals invariants

- **No Telegram bot / Discord / cross-platform message gateway** — Life OS is a local Claude Code decision engine, not an independent bot product
- **No independent daemon / always-on agent** — everything runs inside Claude Code sessions
- **No remote cron / GitHub Actions** — local launchd or crontab only
- **No database as source of truth** — markdown + YAML frontmatter is authoritative; any index layer (if ever added) is a rebuild-able cache
- **No kill criteria with pre-committed numeric thresholds** — AUDITOR eval-history monitors; user decides module-level changes through normal spec revision
