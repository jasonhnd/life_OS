# 快车道 vs 全朝议 · ROUTER 怎么分诊

> 本文档是作者自用参考。写给懂系统内部的人看。

## 一句话区分

**判断标准只有一个：有没有要做的决策？**

- **有**（权衡、选项二选一、不可逆、大额）→ 全朝议（⚖️ 全套）
- **没有**（分析、研究、规划、总结、审查）→ 快车道（🏃 Express）

这是 `pro/agents/router.md` 写的唯一分诊线：

> The key test: is there a decision to make? If yes → full court. If no → express.

## 快车道是什么

ROUTER 跳过 PLANNER / REVIEWER / DISPATCHER / AUDITOR / ADVISOR，**直接启动 1-3 个领域 agent**，每个领域做完整的研究/考虑/判断流程。ROUTER 把结果以**简报形式**呈现——不是奏折。

简报和奏折的区别：
- 简报：没有整体评分、没有 Must/Needs/Room 三层分类、没有正式审计日志
- 奏折：有整体评分（X/10）、有分层结论、有领域评分表、有审计日志、有运营报告

## 快车道跳过了什么

| 步骤 | 全朝议 | 快车道 |
|------|--------|--------|
| 0. 预备会 | ✅ | ✅ |
| 1. ROUTER 分诊 | ✅ | ✅ |
| 意图澄清 | 2-3 轮 HARD | 简短（或 Quick Mode 完全跳过） |
| 2. PLANNER 规划 | ✅ | ❌ |
| 3. REVIEWER 初审 | ✅ | ❌ |
| 4. DISPATCHER 分配 | ✅ | ❌（ROUTER 直接派单） |
| 5. 领域执行 | 六领域并行 | 1-3 领域 |
| 6. REVIEWER 终审 | ✅ | ❌ |
| 7. ROUTER 奏折 | ✅ 奏折 | ✅ 简报（简化） |
| 8. AUDITOR | ✅ | ❌ |
| 9. ADVISOR | ✅ | ❌ |
| 10. ARCHIVER | ✅ | 视情况（简报也可归档为 journal） |

跳过的核心是 **REVIEWER 封驳 + AUDITOR 审计 + ADVISOR 建议**。这三件事是决策质量保障——分析类请求不需要决策质量保障，只需要领域专业度。

## 什么情况走快车道

ROUTER 的判断框架（来自 `pro/agents/router.md`）：

> Express analysis (🏃 快车道): When the request needs domain-level expertise but does NOT involve a decision — no trade-offs, no choosing between options, no irreversibility, no strong emotional weight.

四个"没有"全部满足才走快车道：
1. 没有权衡（不在 A 和 B 之间选）
2. 没有选项切换（不是"做哪个"）
3. 没有不可逆性（结果可调整）
4. 没有强情绪权重（不牵扯价值观、家庭、人际核心）

具体例子：

| 请求 | 类型 | 理由 |
|------|------|------|
| "帮我分析这份合同的风险" | 🏃 快车道 | governance 一个领域，只是检视，没有决策 |
| "总结下项目进度" | 🏃 快车道 | execution 一个领域，只是现状总结 |
| "做个 3 个月学编程的学习计划" | 🏃 快车道 | growth 一个领域，计划不是决策 |
| "这只股票值得买吗" | ⚖️ 全朝议 | 决策（买/不买）+ 有钱风险 |
| "要不要搬去东京" | ⚖️ 全朝议 | 不可逆 + 多领域权衡 |
| "帮我分析我目前的财务状况" | 🏃 快车道 | finance 一个领域，是诊断不是决策 |
| "该不该辞职" | ⚖️ 全朝议 | 决策 + 不可逆 + 情绪权重 |
| "这个技术栈选哪个合适" | 边界情况（看上下文） | 如果是项目选型决策 → 全朝议；如果是学习方向规划 → 快车道 |

## 快车道不要触发的边界

以下请求**看似像"分析"但本质是决策**，绝对不能走快车道：

- "帮我分析要不要……" — "要不要"是决策信号
- "对比一下 A 和 B 哪个好" — 对比后用户会做选择
- "这个 offer 值不值得接" — "值不值"是判断，直接通向决策
- "我该怎么选" — 明确是决策

ROUTER 要能识别这些**伪装成分析请求的决策问题**。用户会用"分析"来降低心理压力，但系统不能被这种措辞骗过去。

## 如何从快车道升级到全朝议

ROUTER 呈现完简报后，固定问一句：

> 🏃 This is an express analysis. Want a full court deliberation instead?

（对应中文："这是快车道分析。要不要升级成全朝议？"）

三种用户回应：

1. **说"是" / "升级"** → ROUTER 把简报里的关键信息 + 用户原话 + 补充问题重组成 Subject，进入第 2 步（PLANNER 规划）。简报内容作为背景材料传给 PLANNER。
2. **说"不用" / 继续聊别的** → 工作流结束（或过渡到下一话题）
3. **不回应（直接问下一个问题）** → 结束快车道，视新问题类型重新分诊

升级时 ROUTER 不需要重跑领域——简报里的分析结果可以作为 PLANNER 的输入。PLANNER 可能扩展新维度、叫新领域、复用快车道已做的工作。

## Token 和时间权衡

**快车道典型成本**（1 领域）：
- 时间：30 秒 – 2 分钟
- Token：5k – 15k（领域单份研究报告规模）
- 角色数：ROUTER + 1 领域 = 2 个独立 subagent

**全朝议典型成本**（6 领域）：
- 时间：3 – 8 分钟
- Token：80k – 200k+（PLANNER + REVIEWER x2 + DISPATCHER + 6 领域 + AUDITOR + ADVISOR + ARCHIVER）
- 角色数：最少 12 个独立 subagent

**倍率大约 10-15 倍**。这就是为什么不能把所有请求都当决策——会极大浪费 token 和时间，用户等不住。

另一方面，如果**该做决策却走了快车道**，代价不是 token，是**决策错误**。快车道没有 REVIEWER 封驳、没有 AUDITOR 审计、没有 SOUL 对齐检查——决策质量显著下降。宁可多花 10 倍 token，不能做错决策。

## Quick Mode · 用户显式强制

如果用户说"quick" / "快速分析" / "クイック"，ROUTER 进入 Quick Mode：
- 跳过意图澄清（即使有决策信号）
- 直接走快车道（1 领域）
- 简报极简

**Quick Mode 是用户知情选择的后果**。ROUTER 不对决策质量负责——用户要求快，系统就快。

但如果 Quick Mode 下 ROUTER 依然检测到强情绪信号（"崩溃"、"撑不下去"），会先回一句"听起来不太好，要不要暂停一下"——不启动 Quick，留空间给用户。情绪优先于速度。

## 实战场景举例

### 场景 1：合同风险审查（快车道典型）

用户："帮我看看这份服务协议有没有法律风险。"

ROUTER 判断：
- 没有决策（不在签/不签之间）
- 只是风险检视
- governance 单领域即可

ROUTER 直接启动 governance，governance 做完整审查，出简报。
之后问："要不要升级成全朝议"——如果用户回"不，我只是先看看"，工作流结束。
如果用户回"等等，这看起来有大问题，我该不该签" → 升级成全朝议，六领域（finance + governance + people + execution）重新分析。

### 场景 2：年度规划（全朝议，不可走快车道）

用户："帮我做 2026 年的年度计划。"

ROUTER 判断：
- 看似"规划"，但包含优先级取舍（家庭 vs 事业 vs 健康）
- 多领域权衡（finance + people + growth + execution + governance + infra）
- 有价值观权重

这是场景 4（年度目标规划），必须全朝议。ROUTER 意图澄清 3 轮，PLANNER 分 6 维度，六领域并行。

### 场景 3：单问题快速答复

用户："我这个月信用卡账单是多少？"

ROUTER 判断：
- 信息查询，不是分析也不是决策
- 不需要 finance 深度研究

ROUTER **直接处理**（甚至不走快车道），通过工具查询 second-brain，直接回答。

### 场景 4：表面分析、本质决策

用户："帮我分析一下跳槽到这家公司划不划算。"

ROUTER 判断：
- 虽说"分析"，"划不划算"是判断
- 本质是决策问题（跳/不跳）
- 不可逆、大额（薪水变化长期影响）

**不能走快车道**。ROUTER 进入意图澄清 3 轮，然后全朝议。

## 陷阱 · ROUTER 容易犯的错

1. **被"帮我分析"骗** — 用户说"分析"≠ 真的只是分析。看有没有决策。
2. **怕用户嫌慢就走快车道** — 质量优先于速度，决策必须全朝议。
3. **快车道后不问升级** — HARD RULE：必须问 "Want a full court deliberation instead?"。
4. **情绪信号强还走快车道** — 情绪强 → 先分离情绪，分离后再判断路径。
5. **Quick Mode 触发时还做意图澄清** — Quick Mode 就是跳过澄清，别违反用户的明确选择。

## 快速自查

写 Petition 前问自己：

- [ ] 用户这次会做一个选择/动作吗？（YES → 全朝议）
- [ ] 结果可逆吗？（不可逆 → 全朝议）
- [ ] 涉及大额、家庭、人际核心吗？（YES → 全朝议）
- [ ] 情绪权重明显吗？（YES → 先分离情绪，再判断）
- [ ] 只是"帮我看看"、"帮我总结"、"帮我研究"？（YES → 快车道）

三个 NO + 最后一个 YES → 快车道。其他情况 → 全朝议。
