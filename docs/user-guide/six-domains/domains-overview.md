# 六大领域总览

Life OS 把个人生活切成六个领域，每个领域有一个专门的 subagent 负责分析。领域之间不是并列的——它们是**职权边界**，用来解决"这个问题应该谁来答"。

## 六个领域

| 领域 | 管什么 | Emoji |
|------|--------|-------|
| PEOPLE | 管人 | 👥 |
| FINANCE | 管钱 | 💰 |
| GROWTH | 管成长、形象、对外表达 | 📖 |
| EXECUTION | 管推进和执行 | ⚔️ |
| GOVERNANCE | 管规矩和风险 | ⚖️ |
| INFRA | 管基础设施（含身体） | 🏗️ |

每个领域下面分四个 division（司），共 24 个细分职能。比如 FINANCE 分为 Income / Spending / Assets / Reserves（收入 / 支出 / 资产 / 储备）。

## 为什么是六个，不是五个或七个

这个划分是试出来的。关键是**相互独立、共同穷尽**：

- 一个决策落地，哪个领域都不能绕开吗？→ 加一个领域
- 两个领域经常打架抢地盘？→ 合并或重划边界

身体为什么在 INFRA 而不是独立一个领域？因为身体和数字基础设施、居住环境一样，都属于"长期资产，需要持续维护，崩了一切停摆"。INFRA 统一管这类事。

## 职权边界规则：按根因路由（root-cause routing）

**原则：按根本原因分类，不按表面症状分类。**

同一句话"我最近很累"，可能是三个不同领域的问题：

| 用户说 | 真正的根因 | 负责领域 |
|--------|-----------|---------|
| "连续加班三周，身体快散架了" | 身体透支 | INFRA/Fitness（主） + EXECUTION/Logistics（辅） |
| "每天忙到很晚，复盘发现没干出实质东西" | 精力分配 + 工作流低效 | EXECUTION/Logistics |
| "早上起不来晚上睡不着，整个节奏乱了" | 作息崩了 | INFRA/Routines（主） + INFRA/Fitness（辅） |
| "焦虑导致失眠，白天没精神" | 心理健康 | INFRA/Fitness |

**快判规则**：身体病了 → Fitness，节奏乱了 → Routines，工作低效 → Logistics。

另一个高频场景"我一直拖延"：

| 用户说 | 真正的根因 | 负责领域 |
|--------|-----------|---------|
| "这个任务太大了，不知道从哪下手" | 没拆解 | EXECUTION/Logistics |
| "明知道该做，拿起手机刷了两小时" | 坏习惯 | GOVERNANCE/Discipline（主） + EXECUTION/Logistics（辅） |
| "每次开会前一晚临时抱佛脚，已经第五次了" | 反复违约 | GOVERNANCE/Discipline |
| "一想到那个任务就焦虑，所以一直不开始" | 心理障碍 | INFRA/Fitness |

**快判规则**：不会做 → Logistics（方法问题），不想做 → Discipline（习惯问题），不敢做 → Fitness（心理问题）。

## DISPATCHER 如何处理依赖

DISPATCHER 是调度员。它拿到 PLANNER 批准的规划文档后，决定：

1. **哪些领域要出手**。不是每次都六个都上。一个 $50 的订阅要不要取消——只需要 FINANCE/Spending，别的领域都来就是过度分析。
2. **并行还是串行**。默认并行。但如果两个领域要改同一个文件（比如 FINANCE 和 EXECUTION 都要修改 `projects/x/index.md`），DISPATCHER 会排成串行，避免写冲突。
3. **主辅关系**。"Japanese client visiting, 要带什么礼物" 这种场景是跨文化商务礼仪，GROWTH/Diplomacy 主导，PEOPLE/Relations 作为辅助嵌入一个小节。

## 主辅机制（assist）

当某个场景标注"辅助"时：

- **主领域**（lead）负责出方案、承担决策责任
- **辅领域**（assist）提供专业输入，嵌入主领域报告的专门小节里
- 如果主领域和辅领域意见相悖，升级到 COUNCIL（朝堂议政）进行结构化辩论

## 典型案例：一个决策如何被拆给六个领域

**用户问："我在考虑辞职去做独立开发者，帮我看看。"**

DISPATCHER 判断这是全员问题：

- **FINANCE**：Runway 有多少个月？副业收入能否顶上？社保断档怎么处理？
- **PEOPLE**：谁会支持、谁会反对？客户资源能带走吗？导师怎么看？
- **GROWTH**：技能结构够不够撑起个人品牌？写作/社交媒体是否就位？
- **EXECUTION**：前 90 天的里程碑？产品路线图？市场定位？
- **GOVERNANCE**：劳动合同有没有竞业条款？知识产权归属？税务身份切换？
- **INFRA**：独立工作的作息节奏？居家工位？健保方案？

六个报告并行出来，REVIEWER 汇总，如果任何两个领域的评分差 ≥ 3 分（比如 FINANCE 给 4 分但 EXECUTION 给 9 分），自动触发 COUNCIL 辩论。

## 和谁配合

| 上游 | 输入 | 下游 | 输出 |
|------|------|------|------|
| DISPATCHER | 调度令（含主辅标签） | REVIEWER 终审 | 领域报告（分数 + 🔴🟡🟢 发现 + 结论） |

每个领域报告必须显式展示：
- 评分（X/10）+ 校准说明
- 🔴 必须处理 / 🟡 需要注意 / 🟢 尚可的发现分类
- 结论（一句话，不要拐弯）

## 信息隔离（每个领域只看到自己该看的）

这是 Pro 模式的硬规则。每个领域 subagent 只拿到：

- **自己的 dispatch 指令**（DISPATCHER 写的）
- **背景材料**（PLANNER 整理的）
- **如果该项目有战略角色**，本项目的战略上下文（flows only，不是全图）

**不会拿到**：
- 其他领域的报告
- ROUTER / PLANNER / REVIEWER 的思考过程
- 用户和 ROUTER 的闲聊

这样做的原因：每个领域独立思考，避免"看到 FINANCE 给了 4 分，我也往低打"这种从众偏差。REVIEWER 终审时才汇总所有报告。

## 分数分布与 COUNCIL 触发

六个领域给出的评分如果出现：
- 两个领域之间评分差 ≥ 3 分
- 或一个领域明确说"可以"，另一个说"不要"

→ REVIEWER 自动触发 COUNCIL（朝堂议政），进行 3 轮结构化辩论。这是决策系统的纠错机制：当六个专业视角内部出现严重分歧，说明问题还没看透，不要急着汇总。

## Express 模式跳过六领域全体

不是每个问题都值得六个领域全上。用户问"这个 $10 订阅要不要取消"，只需要 FINANCE/Spending 即可。

ROUTER 会判断问题类型：
- **直接处理**：闲聊、显而易见的事
- **Express 分析**：需要 1-3 个领域意见，但不涉及正式决策（不经过 PLANNER / REVIEWER / AUDITOR）
- **全员议政**：大决策，走完整 Draft-Review-Execute 流程

Express 模式结束时，ROUTER 会问："刚才是快速分析。要不要走完整议政？"

## 六领域评分与 Summary Report

每个领域的评分不是简单平均。ROUTER 的 Summary Report 会呈现：

```
总评：X/10 —— 一句话结论

必须处理：[来自各领域 🔴 的合并清单]
需要注意：[来自各领域 🟡 的合并清单]
尚可：[来自各领域 🟢 的合并清单]

领域评分表：
| 领域 | 维度 | 评分 | 一句话 |
|------|------|------|--------|
| 👥 PEOPLE | Talent/... | 7/10 | ... |
| 💰 FINANCE | Income/... | 4/10 | ... |
| ...

Action Items:
1. [具体动作] — 截止时间 — 负责领域
```

**总评不是算术平均**。比如 FINANCE 给 3 分（致命财务风险）但其他领域都给 8 分——总评必须反映 FINANCE 的否决性，而不是被稀释成 7 分的平均。

## 六领域不是"分数面板"，是"结构化专业视角"

很多人问："为什么需要六个领域？让 AI 一次性分析不就好了？"

核心原因：**单一视角的盲区**。一个通用 AI 被问"我要不要辞职"，答案倾向乐观（鼓励追梦）或谨慎（稳妥为先），取决于 prompt。但六个独立领域分别从**钱、人、风险、基建、成长、执行**的角度切入，任一领域如果看到致命问题都能挡住整个决策。

这比"让 AI 全面考虑"可靠得多——因为"全面"往往等于"什么都提了但都不深"。结构化的六领域强制每个维度都被独立深究。
