# Wiki · 知识档案总览

> Wiki 是系统的"知识档案"——一份可复用的结论集合，关于这个世界，而不是关于你。

## Wiki 是什么

Wiki 位于你的 second-brain 的 `wiki/` 目录下。它和 `SOUL.md`、`user-patterns.md`、`decisions/` 是完全不同的东西：

| 存储 | 记录什么 | 例子 |
|------|---------|------|
| `decisions/` | 你做了什么决策（具体、带时间戳） | "2026-04-01 决定使用信托结构" |
| `user-patterns.md` | 你做什么（行为模式） | "倾向避开财务维度" |
| `SOUL.md` | 你是谁（价值观、个性） | "风险偏好：中偏高" |
| **`wiki/`** | **你知道什么（可复用的结论）** | **"日本 NPO 贷款无贷金业法豁免"** |

**SOUL 管理"人"，Wiki 管理"知"。两者不得混淆。**

一句话区分：
- 你知道的**关于世界**的 → Wiki
- 你知道的**关于自己**的 → SOUL

---

## 定位表：Wiki 不是什么

| 混淆 | 实际属于 |
|------|---------|
| "我喜欢早上做决策" | user-patterns.md（行为） |
| "我坚信家庭优先" | SOUL.md（价值观） |
| "2026-04-01 我决定用信托" | decisions/（具体决策） |
| "我感觉 X" | 一般是 SOUL 或 journal，不是 wiki |
| "做 X 的步骤是 ..." | Wiki（方法论） |
| "在日本注册 NPO 需要 ..." | Wiki（世界的事实） |
| "我的公司 A 的报表情况" | 隐私，不写入 wiki |

Wiki 的内容必须满足：
- **关于世界，不是关于你**
- **事实或方法论**（不是感受或意见）
- **跨项目可复用**（不只是当前这个项目的细节）
- **无个人隐私**（通过隐私过滤器）

## Title = Conclusion 原则

这是 wiki 最重要的设计决定。

每个 wiki 条目的**标题就是结论本身**，不是话题。打开文件，一眼就看到答案，不用读正文。

正面例子：
- ✅ "日本 NPO 贷款无贷金业法豁免"
- ✅ "MVP 验证的是需求，不是产品质量"
- ✅ "16:8 间歇性断食对我有效"

反面例子：
- ❌ "NPO 贷金业法研究"（这是话题，不是结论）
- ❌ "MVP 方法论笔记"（这是笔记，不是结论）
- ❌ "间歇性断食研究"（什么研究？结论是什么？）

### 为什么这么严格？

因为 wiki 的价值在于**快速读取**。

想象你几个月后遇到类似问题，ROUTER 在 wiki/INDEX.md 里看到一排标题：

```
## Finance
- NPO 贷款无贷金业法豁免 (0.95)
- NPO 税收减免的 3 个条件 (0.82)
```

ROUTER 一眼就知道答案是什么，立刻告诉你"关于这个话题，我们已经知道 X"。

如果标题是"NPO 贷金业法研究"，ROUTER 得打开文件才知道结论——这就浪费了几秒（和 context token）。

**标题 = 结论 = 快速读取**。

---

## 一个结论一个文件

每个 wiki 条目只记录**一个**结论。不要做"话题汇总"文件。

- ✅ `wiki/finance/lending-law-npo.md`（一个结论：NPO 贷款无豁免）
- ❌ `wiki/finance/npo-notes.md`（里面记了 5 个 NPO 相关结论）

### 为什么不能汇总？

1. **标题失效**：汇总文件没法用"结论"做标题
2. **版本演化困难**：5 个结论各自有自己的 evidence/challenges，混在一起 confidence 没法算
3. **引用混乱**：别的 agent 要引用"NPO 贷款无豁免"这个具体结论，用汇总文件引用不了

每个结论有自己的生命周期——让它们各住各的文件。

## 目录结构

```
wiki/
├── INDEX.md              ← 编译生成的索引（别手动编辑）
├── finance/
│   ├── lending-law-npo.md
│   └── npo-tax-deduction.md
├── startup/
│   ├── mvp-validation.md
│   └── biz-plan-versions.md
├── health/
│   └── intermittent-fasting.md
└── _archive/             ← 废弃条目
```

领域分类由你决定，常见的有：`finance` / `startup` / `health` / `legal` / `tech` / 或具体项目名。

INDEX.md 是**编译产物**——RETROSPECTIVE 在每次 Start Session 自动从 wiki/ 目录下的实际文件生成。**不要手动编辑 INDEX.md**，你的改动会被下次编译覆盖。

## 原则

### 1. 从零成长

wiki/ 初始是空的。没有初始化向导。当 archiver 或 DREAM 在某次 session 中发现**满足 6 条严格标准**的结论时，才会写入第一个条目。

**如果 wiki/ 不存在**：所有角色都跳过 wiki 引用，系统照常运行。

### 2. 证据驱动

每个条目都链接到支持它的决策或经验。不存在"没有证据支持的结论"。

confidence 的计算公式和 SOUL 一样：

```
confidence = evidence_count / (evidence_count + challenges × 2)
```

初始 confidence：
- 3+ 独立证据 → 0.5
- 恰好 2 个证据 → 0.3
- 1 个证据 → 直接丢弃

### 3. 自动写入、用户事后修剪

用户**不预先批准** wiki 条目。

- 满足 6 条标准 → archiver 直接写入
- 不满足 → archiver 丢弃，不写低质量条目
- 事后：删除文件 = 废弃；说"undo recent wiki" = 回滚最近的写入

这个机制和 SOUL 一样——拿掉"事前审批"，通过"事后修剪"来引导。

## Wiki 和 SOUL 的关键差异

虽然两者都有 confidence 机制、都自动写入、都是"从零成长"，但用途完全不同：

| 维度 | Wiki | SOUL |
|------|------|------|
| **关于** | 世界 | 你 |
| **内容类型** | 事实 / 方法 | 价值观 / 个性 |
| **隐私过滤** | 严格（剥离所有个人信息） | 不适用（SOUL 就是关于你） |
| **What SHOULD BE** | 不存在 | 核心字段 |
| **结构** | 一结论一文件 | 一维度一段 |
| **引用者** | ROUTER 意图澄清、REVIEWER 一致性检查 | 全系统 |

最关键的区别：**wiki 没有"应然"**。

wiki 只记录"是什么"——世界的规律、事实、方法。wiki 不会说"你应该相信 X"或"你应该做 Y"——那些是 SOUL 的事。

## Wiki 的四个来源

| 来源 | 怎么产生 | 什么时候 |
|------|---------|---------|
| **DREAM** | N3 阶段从 3 天活动中发现可复用结论 | 每次 adjourn |
| **Session** | 某个六部的报告产出可复用结论 | Journal 中带 `wiki_candidate: true` |
| **archiver Phase 2** | session 结束时扫所有材料 | 每次 session adjourn |
| **你自己** | 直接输入"记住这个事实：X" | 任何时候 |

前三个都会走 6 条严格标准。你自己输入的不走标准（你说什么就是什么），但仍然走隐私过滤器提醒。

---

## 如何开始

**不用做任何事**。

- 第一次 session 时 wiki/ 不存在 → 系统照常运行
- 做几次决策，archiver 开始扫描材料
- 当某次 session 产出满足 6 条标准的结论，系统自动写入第一个条目
- 下次 Start Session 的简报会告诉你"本 session 新增 wiki 条目 X"
- 你打开看看，不满意就删，满意就让它长

**Wiki 的增长非常缓慢**。不是每次 session 都会新增——很多 session 一个条目都不会生成（因为大部分对话的内容不是"世界的可复用结论"）。

这是故意的。**Wiki 不追求数量，追求质量**。10 个 confidence 0.8 的条目，比 100 个 confidence 0.3 的条目有用得多。
