# Concept Specification

Concept files are the nodes in Cortex's synaptic network. Each file represents one reusable idea, entity, or pattern that can be activated across sessions. Synapse edges between concepts live inside the YAML frontmatter of the concept file itself, not in a separate graph database. This keeps the entire network inspectable with a text editor and portable through plain git.

## Positioning

| Storage | What it records | Example |
|---------|----------------|---------|
| `decisions/` | What you decided (specific, timestamped) | "2026-04-01: chose trust structure" |
| `user-patterns.md` | What you do (behavioral patterns) | "Tends to avoid financial dimensions" |
| `SOUL.md` | Who you are (values, personality) | "Risk appetite: medium-high" |
| `wiki/` | What you know (reusable world conclusions) | "NPO lending has no 貸金業法 exemption" |
| `_meta/concepts/` | **What connects to what (synaptic graph)** | "company-a-holding" node + edges to other concepts |

SOUL manages the person. Wiki manages knowledge. Concepts manage the graph that lets the system know "if A is activated, what else should light up". They must not be mixed.

---

## Principles

1. **Grows from zero** — `_meta/concepts/` starts empty. No initialization required.
2. **One concept = one file (strict)** — No multi-topic compilations. Splitting a single concept across files is forbidden (anti-pattern, see §10).
3. **Edges live with nodes** — Outgoing synapse edges are stored in the YAML frontmatter of the source concept. Reverse index is compiled to `SYNAPSES-INDEX.md` (never hand-authored).
4. **Auto-written under strict criteria** — Concepts are auto-created by `archiver` Phase 2 when evidence accumulates. Users nudge by deletion or by saying "undo recent concept".
5. **Permanence determines decay** — Each concept carries a permanence tier that dictates its decay curve (user decision #9, brainstorm §7).
6. **Reversible lifecycle** — `tentative → confirmed → canonical` is a single-direction promotion path, but demotion is always possible (user decision #10).

## File Location

```
_meta/concepts/{domain}/{concept_id}.md
```

| Path segment | Meaning |
|--------------|---------|
| `_meta/concepts/` | Root of the concept network |
| `{domain}/` | Thematic partition (see below) |
| `{concept_id}.md` | One concept per file, named by its ID |

### Domain partitions

Domains are the top-level thematic slots for concepts. Minimum set (aligned with `wiki-spec.md` and `method-library-spec.md`):

- `finance/` — financial entities, instruments, concepts
- `startup/` — venture, product, growth concepts
- `personal/` — personal-life domain concepts (non-identity — identity belongs in SOUL)
- `technical/` — engineering, architecture, implementation patterns
- `method/` — methodologies, workflows, procedures
- `relationship/` — organisations and non-individual relational entities
- `health/` — health, fitness, medical concepts
- `legal/` — legal frameworks, compliance, regulatory concepts
- `writing/` — writing craft, document-refinement methodologies, editorial patterns

Users may add new domain directories as their second-brain grows. There is no schema-enforced whitelist — the archiver creates the directory on first concept assignment.

### Reserved paths

- `_meta/concepts/INDEX.md` — compiled one-liner summary of all concepts (regenerated by `retrospective` Mode 0)
- `_meta/concepts/SYNAPSES-INDEX.md` — compiled reverse edge index (regenerated by `archiver` Phase 2)
- `_meta/concepts/_tentative/` — staging area for concepts below confidence threshold
- `_meta/concepts/_archive/` — retired concepts (kept for audit, ignored at activation)

## YAML Frontmatter Schema

Every concept file begins with YAML frontmatter. All fields are listed; optional fields are marked.

```yaml
---
concept_id: string                # required, unique across the network
canonical_name: string            # display name (human-readable)
aliases: [string]                 # other surface forms this concept may appear as
domain: string                    # finance | startup | personal | technical | method | relationship | ...
status: enum                      # tentative | confirmed | canonical
permanence: enum                  # identity | skill | fact | transient
activation_count: integer         # how many times referenced across sessions
last_activated: ISO 8601          # timestamp of last activation
created: ISO 8601                 # creation timestamp
outgoing_edges:
  - to: concept_id
    weight: integer               # 1-100, Hebbian-strengthened
    via: [tag]                    # what triggered this connection (free-form tags)
    last_reinforced: ISO 8601
provenance:
  source_sessions: [session_id]   # sessions where evidence appeared
  extracted_by: enum              # archiver | manual | dream
decay_policy: enum                # matches permanence tier (see §5)
---

# Canonical Name

Body content (markdown, optional but encouraged)...

## Evidence / Examples
- [link to source session or decision]

## Related Concepts
- [[other-concept-id]]
```

### Field constraints

| Field | Constraint |
|-------|-----------|
| `concept_id` | Lowercase, hyphens only, must start with letter, max 64 chars (see §7) |
| `canonical_name` | Any human-readable string |
| `aliases` | Zero or more. Used by the concept-resolver when mapping raw text to IDs |
| `domain` | Must match the concept's parent directory |
| `status` | `tentative` → `confirmed` → `canonical`. No downgrades through this field — demotion goes through the decay log |
| `permanence` | `identity` | `skill` | `fact` | `transient` |
| `activation_count` | Non-negative integer, monotonically increasing during active life |
| `last_activated` | Used by decay pass to determine dormancy |
| `outgoing_edges.weight` | Bounded to 100 (cap prevents runaway reinforcement) |
| `decay_policy` | Must agree with `permanence` — see §5 |

## Lifecycle

```
1. 🌱 Creation — archiver Phase 2 extracts concept from session with ≥2 evidence points, creates tentative file
2. 📈 Reinforcement — each session activation increments activation_count, updates last_activated, reinforces edges
3. ✅ Promotion — tentative → confirmed (≥3 independent sessions) → canonical (user-pin OR ≥10 independent sessions)
4. 📉 Decay — archiver runs the decay pass at every Adjourn (user decision #10)
5. 💤 Retirement — last_activated > 90 days + all outgoing edges weight < 1.0 + permanence ≠ identity → move to _meta/concepts/_archive/ (activation_count preserved, never decremented)
6. ↩️ Undo — user says "undo recent concept" or deletes the file manually
```

### Creation

`archiver` Phase 2 (runs at session Adjourn, after wiki / SOUL auto-write):

1. Scan session frames for recurring noun phrases, entity mentions, method names
2. For each candidate, apply **ALL 6 creation criteria**:
   1. **Frequency ≥ 2 mentions** in the session (single-mention things are too transient)
   2. **≥ 2 independent evidence points** (distinct frames, distinct reports, or distinct decisions — not the same sentence quoted twice)
   3. **Identity beyond this session** — the candidate is likely reusable in future sessions (LLM judgment)
   4. **Not a person, value, trait, or procedure** — people belong to SOUL/user-patterns; values/traits belong to SOUL; procedures belong to the method library
   5. **Privacy filter clears** — strips names (unless public figures), specific amounts/accounts/IDs, family/friend references, traceable date+location combinations. If stripping leaves the candidate meaningless → discard (it was a personal note, not a reusable concept)
   6. **Domain routing succeeds** — LLM picks one of the domains listed in §File Location Domain partitions (or creates a new domain directory). A candidate that cannot be routed to any domain is not yet a concept
3. All 6 pass → create file in `_meta/concepts/_tentative/{domain}/{concept_id}.md`
4. Any criterion fails → discard, log in `decay-log.md` for later audit

> Implementations: `pro/agents/archiver.md` §Phase 2 Mid-Step (Concept Extraction + Hebbian Update) and `tools/migrate.py` use the same 6-criteria enumeration. When this spec changes, those must be updated in lock-step.

### Promotion

| From → To | Trigger |
|-----------|---------|
| `tentative → confirmed` | Concept activated across **≥3 independent sessions**, file moves from `_tentative/` to `{domain}/` |
| `confirmed → canonical` | Either user pins manually (`status: canonical` in frontmatter), or the concept has been activated across **≥10 independent sessions** |

Promotion is recorded in `_meta/cortex/decay-log.md` with timestamp and triggering session.

### Reinforcement

Each activation during a session:
- `activation_count += 1`
- `last_activated = {current ISO timestamp}`
- For each edge (A → B) where both A and B were co-activated in the same frame: `weight += 1` (bounded at 100), `last_reinforced = today`

### Decay

Archiver Phase 2 runs a decay pass at every Adjourn (user decision #10). Decay is applied per permanence tier:

| Permanence | Decay behaviour | Rationale |
|-----------|-----------------|-----------|
| `identity` | No decay | SOUL-adjacent values never fade |
| `skill` | Logarithmic decay to a floor | Skills are retained once learned, rarely forgotten |
| `fact` | Exponential decay | Context-bound facts lose relevance with time |
| `transient` | Cliff decay to zero | Event-bound concepts expire hard |

Edge weights decay alongside concept dormancy. When `weight ≤ 0`, the edge is removed on the next Adjourn pass.

### Retirement

Retirement is driven by **dormancy + edge-weight collapse**, not by decrementing `activation_count`. The `activation_count` field remains **monotonically non-decreasing** for the concept's entire active life (see §Reinforcement); retirement is a lifecycle transition triggered by different signals:

1. **Dormancy threshold** — `last_activated` is older than **90 days** (no session has re-activated the concept in that window), AND
2. **Edge-weight collapse** — every edge in `outgoing_edges` has `weight < 1.0` after the current Adjourn's decay pass (i.e., the concept has become disconnected from the live graph), AND
3. **Permanence tier is not `identity`** — identity-tier concepts never retire (see §5 Decay table)

When all three conditions hold, the concept is moved to `_meta/concepts/_archive/{domain}/{concept_id}.md` on the next Adjourn pass. Archived concepts are:
- Still in git history (no data loss)
- Ignored by the hippocampus at activation time
- Visible to the AUDITOR during patrols
- `activation_count` preserved at its final monotonic value (for historical audit)

A future re-activation (same subject surfaces again) restores the concept from `_archive/` back to `{domain}/` and continues incrementing `activation_count` from where it left off.

### Undo

Two undo paths:

1. **Manual deletion** — user removes the file → the next `archiver` run rebuilds `SYNAPSES-INDEX.md` and prunes dangling edges
2. **Verbal undo** — user says "undo recent concept" in a session → the archiver's next invocation rolls back the most recent auto-writes (same pattern as wiki undo, see `references/wiki-spec.md`)

## Hebbian Update Algorithm

Runs inside `archiver` Phase 2, immediately after wiki/SOUL auto-write and before SYNAPSES-INDEX regeneration.

```
1. Extract the set of concepts activated in the session from the full Cortex payloads pasted to the user and written to `_meta/journal/{date}-cortex.md`
2. For each ordered pair (A, B) that co-occurred in at least one frame:
     - Find or create edge A → B in A's outgoing_edges
     - Find or create edge B → A in B's outgoing_edges
     - weight += 1 (capped at 100)
     - last_reinforced = today
3. For each new concept (not in any existing file):
     - Pass the 6-criteria check (cross-session candidate + ≥2 evidence points)
     - On pass → create tentative entry with initial frontmatter
     - On fail → log in decay-log.md, do not create
4. For each activated concept (existing file):
     - activation_count += 1
     - last_activated = now
5. Decay pass (see §5 Decay)
     - Apply per-permanence decay curve
     - Remove edges with weight ≤ 0
6. Recompile SYNAPSES-INDEX.md from all concept files
```

Hebbian updates are idempotent within a single session — re-running the pass produces the same result (co-occurrence count is bounded by the session, not by agent invocations).

## Spreading Activation Rules

Used by the `hippocampus` subagent at the Pre-Router Cognitive Layer (Step 0.5, see `devdocs/architecture/cortex-integration.md` §4). Spreading activation produces the list of concepts the system considers "warm" for the current message.

```
Wave 1 — Direct match
  - Concepts whose canonical_name or aliases match the user message
  - No ranking adjustment — these are the seeds

Wave 2 — Strong neighbours
  - For each seed, follow outgoing edges with weight ≥ 3
  - Collect destinations, preserve edge weight for ranking

Wave 3 — Weak neighbours
  - For each seed, follow outgoing edges with weight ≥ 1 and < 3
  - Used only for sub-threshold pre-activation (brainstorm §8.5)
```

After three waves, results are ranked by `relevance × edge_weight × concept.activation_count` and bounded to **top 5-7 concepts total** (hard cap — prevents cognitive flood).

Sub-threshold pre-activation: concepts from Wave 3 that did not make the top 5-7 are still cached with `salience = 0.5 × edge_weight / 10` for the rest of the session. They do not fire on their own but can cross threshold if a subsequent frame reinforces them.

## Concept ID Generation

A canonical concept ID is:
- Lowercase ASCII plus hyphens only (e.g. `iterative-document-refinement`)
- Must start with a letter
- Maximum 64 characters
- Globally unique within the concept network

### Deriving the ID

From a raw canonical name, the archiver lowercases, replaces non-alphanumeric runs with single hyphens, trims leading/trailing hyphens, and truncates to 64 characters. Example:

| Canonical name | Concept ID |
|----------------|-----------|
| "Iterative Document Refinement" | `iterative-document-refinement` |
| "Company-A (Holding)" | `company-a-holding` |
| "三省六部" | (untransliterable CJK → falls back to transliteration or an explicit alias, see below) |

### Conflict resolution

If the derived ID already exists for a **different** canonical concept, the archiver appends the domain as a suffix:

```
iterative-document-refinement            → existing, method domain
iterative-document-refinement--writing   → new, writing domain
```

(Double hyphen separates the domain suffix to avoid colliding with words that contain a single hyphen.)

### CJK and non-Latin inputs

When the raw canonical name has no Latin representation, the archiver:
1. Uses a short Romaji/Pinyin transliteration if obvious
2. Falls back to a semantic English label (`cabinet-three-departments` for 三省六部) and stores the original as an alias

The ID is a technical identifier, not a display name — `canonical_name` and `aliases` carry the user-visible forms.

## Privacy

Concept files store **cross-session reusable knowledge**, not personal information.

| Belongs in concepts | Belongs elsewhere |
|---------------------|-------------------|
| `mvp-validation-methodology` | Individual's name → SOUL / user-patterns (never concepts) |
| `trust-structure-in-japan` | Specific bank account numbers → never stored anywhere |
| `pre-mortem-technique` | Family members' names → SOUL (with privacy filter) |

The **LLM privacy filter** (see `references/wiki-spec.md`) runs before any concept is created:
- Strip names (unless public figures directly relevant to the conclusion)
- Strip specific amounts, account numbers, ID numbers
- Strip specific company names (unless it's a public-domain case study)
- Strip family/friend references
- Strip traceable date+location combinations
- If stripping leaves the concept meaningless → discard (it wasn't reusable, it was a personal note)

Personal information flows into SOUL.md, not into concepts. Concepts about individual people (coworkers, family, friends) are explicitly forbidden — that territory is SOUL (user decision #8).

## Migration from v1.6.2a

Pre-v1.7 there is no concepts directory. Migration to v1.7 runs once, orchestrated by `tools/migrate.py` (user decision #7):

1. Scan **last 3 months** of `_meta/journal/` entries (bounded window — older material goes untouched)
2. Extract candidate concepts using the same algorithm as the archiver Hebbian pass
3. For each candidate that passes the 6-criteria check and privacy filter:
   - Create the file under `_meta/concepts/_tentative/{domain}/{concept_id}.md`
   - Initial `activation_count` = count of independent sessions where the concept appeared (capped at 10)
4. Compute initial synapse weights from co-occurrence in the same journal entries (capped at 10)
5. Move concepts with `activation_count ≥ 3` from `_tentative/` to `{domain}/` with `status: confirmed`
6. Concepts with `activation_count ≥ 10` get `status: canonical`
7. Generate the first `SYNAPSES-INDEX.md` and `INDEX.md`
8. Log the migration result to `_meta/cortex/bootstrap-status.md`

Migration is idempotent — re-running it against an already-migrated tree produces no additional files. The same script is also the Step 0.5 auto-bootstrap path when Cortex is enabled and required indexes are missing or empty.

## Anti-patterns

These are forbidden by spec and must be rejected at write-time by the archiver:

1. **Manually editing `outgoing_edges`** — the archiver owns edge state. User edits are overwritten on the next Adjourn. To break a connection, delete one of the concept files or say "undo recent concept".
2. **Creating concepts about individual people** — that's SOUL territory (user decision #8). Personal relationships belong in SOUL.md; the `relationship/` domain is reserved for organisations and non-individual entities.
3. **Splitting a single concept across multiple files** — a concept is one file. If a concept is too big to describe in one file, it's actually multiple concepts that need separate entries with explicit edges between them.
4. **Hand-authoring `SYNAPSES-INDEX.md`** — it's a compiled artefact, regenerated every Adjourn. Hand edits will be overwritten.
5. **Reusing an ID across domains without the double-hyphen suffix** — the archiver enforces uniqueness; collisions resolve via the rule in §7.
6. **Storing personal data in concept body or evidence section** — the privacy filter must run; un-filtered text violates the spec.

## How Each Role Uses Concepts

All roles check if `_meta/concepts/INDEX.md` exists before referencing it. During Step 0.5, a missing or empty concept index is an auto-bootstrap precondition: the orchestrator runs `tools/migrate.py` before spawning Cortex subagents. If bootstrap fails, concept input is null and GWT records the condition through `degradation_summary`. Outside Step 0.5, roles operate normally without concept input.

| Role | What they read | How they use it |
|------|---------------|-----------------|
| **hippocampus** | `_meta/concepts/INDEX.md` + files on the spreading-activation path | Generates the "warm concepts" signal for GWT arbitration |
| **gwt-arbitrator** | Concept-link signals produced by hippocampus | Weights concept signals into the salience scalar |
| **ROUTER** | Annotated input that includes warm concepts | Sharper intent classification — treats warm concepts as context, not facts |
| **Six Domains (Pro)** | Concept entries passed in dispatch context | Start analysis from established ideas rather than re-deriving them |
| **REVIEWER** | `_meta/concepts/INDEX.md` | Consistency check — flags when new conclusions conflict with canonical concepts |
| **AUDITOR** | `_meta/concepts/` directory (during patrols) | Concept health — stale nodes, orphan edges, privacy violations |
| **DREAM** | `_meta/concepts/INDEX.md` + `SYNAPSES-INDEX.md` | N3 uses the graph for cross-domain associations; REM creative connections |
| **archiver** | Session frames + existing concept files | Runs the Hebbian update algorithm + decay pass at Phase 2 (writes `outgoing_edges`, `SYNAPSES-INDEX.md`, and updates `activation_count`/`last_activated`) |
| **retrospective** | All concept files (during Mode 0) | Regenerates `INDEX.md` (read-only; decay is archiver's write responsibility, see §Decay). Flags dormant concepts in the briefing. |

## Constraints

- **One concept per file** — splitting is forbidden
- **Archiver owns writes to `outgoing_edges`** — user edits will be overwritten
- **Personal data excluded by privacy filter** — individual people belong in SOUL
- **Edge weights capped at 100** — runaway reinforcement is bounded
- **Spreading activation bounded at top 5-7** — prevents cognitive flood
- **Concept ID ≤ 64 chars, lowercase + hyphens, starts with letter**
- **Migration reads only last 3 months of `_meta/journal/`** — older material untouched
- **SYNAPSES-INDEX.md and INDEX.md are compiled** — never hand-author them
- **Local-only — no Notion sync** — concepts stay in `_meta/` (user decision #12)
