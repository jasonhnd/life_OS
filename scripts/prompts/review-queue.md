# User-invoked prompt · review-queue walker (v1.8.0 R-1.8.0-013)

> Borrowed from llm_wiki async review pattern. Single entry point for
> "things that need user attention" across all maintenance tasks.

## Trigger keywords

- `处理 queue` / `看 queue` / `review queue` / `walk queue`
- `今天有什么要处理的` / `what needs my attention`
- session-start-inbox hook reports `📋 Review queue: N P0 / M P1 / K P2 open` and user says "处理"

## Context

`_meta/review-queue.md` is the single source of truth for action items
generated by maintenance prompts (auditor-patrol, advisor-monthly,
strategic-consistency, archiver-recovery, eval-history-monthly) and ROUTER
deadline detection.

Each item has YAML schema per `references/review-queue-spec.md`:

```yaml
- id: r{YYYY-MM-DD}-{NNN}
  created: <ISO8601 with TZ>
  source: <auditor-patrol|advisor-monthly|strategic-consistency|archiver-recovery|eval-history-monthly|router|user>
  type: <stale-wiki|drift|conflict|violation|action-item|decision-due|outcome-unmeasured|other>
  priority: P0 | P1 | P2
  summary: <one line, max 100 chars>
  detail_path: <full report path or null>
  related:
    - "[[wikilink-1]]"
    - "[[wikilink-2]]"
  suggested_action: <what user can do>
  status: open | reviewed | resolved | dismissed
  closed_at: null | <ISO8601>
  closed_by: null | user-resolved | auto-archived | superseded
```

## Required actions

### 1. Read queue and summarize

Read `_meta/review-queue.md`. Parse all items under `## Open items`.

Show user a 1-screen summary grouped by priority:

```
📋 Review queue · 3 P0 / 5 P1 / 2 P2 open · oldest 12 days

P0 (urgent — handle today):
1. [r2026-04-29-001] auditor-patrol · stale-wiki · "wiki entry X 90+ 天没更新"
2. [r2026-04-25-003] advisor-monthly · action-item · "regret accumulation > 3 in finance domain"
3. [r2026-04-17-002] strategic-consistency · conflict · "project A vs project B time conflict"

P1 (this week):
4-8. [...]

P2 (whenever):
9-10. [...]

Walk through? (y / 跳到 P0 only / 退出)
```

### 2. Walk through items one-by-one

For each item user agrees to walk:

**Step A**: Read `detail_path` if not null. Show user:
- Item summary
- Source report excerpt (first 30 lines or relevant section)
- `suggested_action`
- Related wikilinks (resolve to actual file paths if possible)

**Step B**: Ask user choice:

```
What do you want to do?
  (A) act now — I'll help you take the suggested action
  (R) reviewed, no act — mark reviewed, decide later
  (D) dismiss — explicitly say "don't track this"
  (S) skip — leave open, move to next
  (Q) quit walker
```

**Step C**: Execute choice:

- **A (act now)**: Walk user through suggested_action. After completion, set
  status to `resolved`, `closed_at` to now ISO8601, `closed_by` to `user-resolved`.
  Move item from `## Open items` to `## Recently resolved`.

- **R (reviewed)**: Set status to `reviewed`. Item stays in `## Open items`
  with status updated. (No close fields.)

- **D (dismiss)**: Set status to `dismissed`, `closed_at` to now,
  `closed_by` to `user-resolved`. Move to `## Recently resolved`.

- **S (skip)**: No change, move to next item.

- **Q (quit)**: Stop walker. Report progress: "Walked N of M items.
  Resolved X / reviewed Y / dismissed Z / skipped K."

### 3. Apply Edits

Use **Edit tool** (NOT Write — would overwrite entire queue) to:

- For status update with no section change (R): replace the single `status: open` line with `status: reviewed`.

- For status update with section move (A or D): two-step Edit:
  1. Remove the YAML block from `## Open items` section
  2. Add updated YAML block (with `status: resolved/dismissed`, `closed_at`, `closed_by` filled) to top of `## Recently resolved` section

### 4. Archive trim (run at end of walker if applicable)

If `## Recently resolved` exceeds 100 entries:
- Move oldest entries past 100 to `_meta/review-queue/archive/{YYYY-MM}.md` (append-only monthly file by `closed_at` month)
- Trim from main queue file
- Report: "Archived N old resolved items to _meta/review-queue/archive/{month}.md"

### 5. Final report

```
✅ Walker complete
- Items walked: N of M
- Resolved (acted on): X
- Reviewed (decide later): Y
- Dismissed: Z
- Skipped: K
- Remaining open: M - X - Z
```

If user took action that was substantial (e.g., updated SOUL, deleted wiki,
killed project), suggest archiving via "退朝" / "adjourn" so the actions
get captured in session journal.

## "Add item" sub-flow (manual user append)

If user says "add to queue" / "track this":

1. Ask: priority (P0/P1/P2)? source (default: user)? summary? suggested_action? related (wikilinks)?
2. Construct YAML per schema, id = next `r{TODAY}-{NNN}` sequence
3. Edit `_meta/review-queue.md` to append at end of `## Open items`
4. Confirm: "Added [id] to queue."

## HARD RULES

- **Use Edit tool, NEVER Write tool** for queue updates (preserves other items)
- **Status transitions are persistent** — quit anytime, partial walk-through saves
- **Don't auto-resolve items** — every status change requires explicit user choice (A/R/D)
- **Privacy filter applies** — if walker surfaces a `_meta/people/<id>.md` related
  item with `privacy_tier: high`, redact `aliases` field in display (per
  `references/people-spec.md` privacy rule)
- **Audit trail**: append `_meta/runtime/{sid}/review-queue-walk.json` recording
  N walked / X resolved / Y reviewed / Z dismissed / archive trimmed (if any)

## Anti-patterns

- DON'T re-prioritize items mid-walk (P0 stays P0; only status changes)
- DON'T silently archive without telling user the count
- DON'T act on item without showing detail_path content first (user must see why)
- DON'T mark resolved if user only said "looked at it" — that's R, not A
